## 1. 软件架构设计（核心重点）

> 💡 **命题趋势**：案例分析必答题每次必考架构风格，选择题每次3-5题。近年趋势是传统架构+新技术架构结合。

### 🔴 五大架构风格（每次必考）

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：官方教程第7章、51CTO高频知识点、案例分析必答题固定考点

| 风格分类 | 子风格 | 核心特征 | 适用场景 |
|---------|--------|---------|---------|
| **数据流风格** | 批处理序列 | 每步必须前一步完成，数据以整体传递 | 数据处理流水线 |
| | 管道/过滤器 | 构件=过滤器，每个有输入输出，增量处理 | 编译器、Unix管道 |
| **调用/返回风格** | 主程序/子程序 | 层次分解，逐步求精 | 传统结构化程序 |
| | 面向对象 | 数据+操作封装为对象 | OOP系统 |
| | 层次结构 | 每层为上层服务，作下层客户 | 网络协议、OS |
| **独立构件风格** | 进程通信 | 构件=进程，通过消息传递交互 | 分布式系统 |
| | 事件驱动(隐式调用) | 触发/广播事件，不直接调用 | GUI、消息系统 |
| **虚拟机风格** | 解释器 | 四部分：程序+解释引擎+程序内部状态+被解释程序当前状态 | 脚本语言、规则引擎 |
| | 基于规则的系统 | 规则集+推理引擎+工作内存 | 专家系统、AI推理 |
| **仓库/数据共享风格** | 数据库系统 | 构件控制共享数据=传统数据库 | 信息管理系统 |
| | 黑板系统 | 知识源(KS)响应黑板变化，控制器决定哪个KS执行 | 语音识别、信号处理 |

**🔑 必背要点**：
- 管道/过滤器：**支持复用、可维护**；每个过滤器独立完成，不知道其前后过滤器；不适合交互式应用
- 事件驱动：**松耦合**，但难以保证执行顺序
- 分层：每层只与相邻层交互，**松耦合**但性能有损
- 黑板 vs 数据库：关键区别是**触发方式**——黑板由数据状态触发，数据库由请求触发

### 🔴 C/S 与 B/S 架构

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅
> **来源**：官方教程、选择题+案例分析

| 对比维度 | C/S | B/S |
|---------|-----|-----|
| 客户端 | 需安装专用客户端 | 浏览器即可 |
| 维护成本 | 高（需更新客户端） | 低（服务端更新） |
| 安全性 | 较高 | 较低 |
| 性能 | 客户端可分担计算 | 服务器压力大 |
| 三层C/S | 表示层-功能层-数据层 | 浏览器-Web服务器-数据库 |

### 🔴 面向服务架构（SOA）

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：官方教程第20章、案例分析+论文

- **三大角色**：服务提供者、服务请求者、服务注册中心
- **三大操作**：发布(UDDI)、发现(UDDI)、绑定(SOAP)
- **WSDL**：描述服务接口
- **ESB（企业服务总线）核心功能**：
  - 消息路由与寻址（位置透明性）
  - 服务注册与命名管理
  - 多种消息传递范型（请求/响应、发布/订阅）
  - 多种传输协议支持
  - 数据格式转换
  - 日志和监控

### 🔴 微服务架构

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：案例分析+论文高频，选择题也常考

- **核心思想**：将单体应用拆分为一组小型服务，每个服务独立部署、独立进程
- **与SOA对比**：

| 对比维度 | SOA | 微服务 |
|---------|-----|-------|
| 服务粒度 | 粗粒度 | 细粒度 |
| 通信 | ESB（重量级） | RESTful/轻量级消息 |
| 数据管理 | 共享数据库 | 每个服务独立数据库 |
| 治理 | 集中治理 | 去中心化治理 |

- **关键技术**：服务注册发现、API网关、熔断器(Circuit Breaker)、负载均衡、配置中心
- **服务网格(Service Mesh)**：2024-2025新增考点，Sidecar模式

### 🔴 MVC 模式

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅
> **来源**：官方教程第16章

- **Model（模型）**：业务数据和业务逻辑，响应状态查询，通知视图更新
- **View（视图）**：显示模型状态，接受用户交互
- **Controller（控制器）**：接受用户输入，调用模型，选择视图
- **变体**：MVP（Presenter替代Controller，2024上考过）、MVVM

### 🟡 C2 风格

> **出题记录**：2023下✅ 2024下✅
> **来源**：官方教程

- 通过连接件绑定的并行构件网络
- 构件和连接件都有**顶部和底部**
- 构件的顶部→连接件底部，构件底部→连接件顶部
- **构件之间不能直接连接**（必须通过连接件）
- 两个连接件连接时：一个底部→另一个顶部

### 🟡 DSSA（特定领域软件架构）

## 2. 系统质量属性与架构评估

> 💡 **命题趋势**：案例分析必答题的固定搭配——架构风格+质量属性+效用树/敏感点/权衡点

### 🔴 六大质量属性（每次必考）

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：官方教程第8章、51CTO高频知识点、案例分析必答题

| 质量属性 | 定义 | 关键架构策略 |
|---------|------|------------|
| **性能(Performance)** | 响应时间/吞吐量 | 增加计算资源、减少计算开销、引入并发、资源调度 |
| **可用性(Availability)** | 系统正常运行时间比例 | 心跳检测、Ping/Echo、主动冗余、被动冗余、选举 |
| **安全性(Security)** | 抵抗未授权访问能力 | 入侵检测、用户认证、用户授权、追踪审计 |
| **可修改性(Modifiability)** | 修改的难易程度 | 接口-实现分离、信息隐藏、抽象、限制通信路径 |
| **可靠性(Reliability)** | 无故障运行能力 | 容错、冗余、检错 |
| **可测试性(Testability)** | 测试发现缺陷的难易 | 记录/回放、接口与实现分离、内置监控 |

### 🔴 质量属性场景六要素

> **出题记录**：2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：案例分析必答题，近年扩展到完整六要素

| 要素 | 说明 | 示例 |
|-----|------|------|
| **刺激源(Source)** | 产生刺激的实体 | 用户、外部系统、攻击者 |
| **刺激(Stimulus)** | 到达系统的事件 | 请求、故障、攻击 |
| **环境(Environment)** | 刺激发生时系统状态 | 正常运行、过载、启动中 |
| **制品(Artifact)** | 被刺激影响的部分 | 系统、子系统、构件 |
| **响应(Response)** | 系统的反应 | 处理请求、记录日志、通知 |
| **响应度量(Measure)** | 可度量的响应指标 | 延迟<2s、可用性99.99% |

### 🔴 敏感点、权衡点、风险点

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：案例分析必答题送分点

| 概念 | 定义 | 举例 |
|-----|------|-----|
| **敏感点(Sensitivity Point)** | 一个或多个构件的特性，影响**某个**质量属性 | 加密级别影响安全性和性能 |
| **权衡点(Tradeoff Point)** | 影响**多个**质量属性的敏感点 | 加密级别同时影响安全性↑和性能↓ |
| **风险点(Risk)** | 潜在问题的架构决策 | 某组件故障可能导致全系统不可用 |
| **非风险点(Non-risk)** | 经分析可接受的架构决策 | 冗余设计已覆盖该故障场景 |

**🔑 辨析技巧**：
- 只影响一个质量属性 → 敏感点
- 影响多个质量属性 → 权衡点
- 权衡点一定是敏感点，敏感点不一定是权衡点

### 🔴 ATAM（架构权衡分析方法）

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025下✅
> **来源**：案例分析+选择题

- **目的**：评估架构决策的质量属性之间的权衡
- **产出**：质量属性效用树、风险/非风险、敏感点/权衡点
- **效用树(Utility Tree)**：
  - 根节点=整体效用
  - 第二层=质量属性（性能、可用性、安全性...）
  - 第三层=质量属性细化
  - 叶子节点=具体场景，标注(重要性, 难度)如(H,M)
- **两轮活动、九个步骤**：
  - 第一轮（以架构为中心）：演示商业驱动→演示架构→识别架构方法→生成质量属性效用树→分析架构方法
  - 第二轮（以涉众为中心）：头脑风暴场景→分析架构方法→呈现结果

### 🟡 SAAM（场景化架构分析方法）

> **出题记录**：2023下✅ 2024下✅
> **来源**：选择题

- 最初用于分析架构**可修改性**，后扩展到其他质量属性
- 步骤：开发场景→描述架构→分类场景→评估场景→总结

### 🟢 CBAM / ARID

> **出题记录**：2024上✅
> **来源**：选择题

- **CBAM**：成本效益分析法，在ATAM基础上加入成本因素
- **ARID**：活动评审中间设计，结合ATAM和ADR

## 3. 设计模式

> 💡 **命题趋势**：选择题每次2-3题考设计模式辨析，案例分析偶尔考。需掌握23种GoF模式的分类和意图。

### 🔴 设计模式分类与高频模式

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：官方教程、选择题固定考点

**创建型模式（5个）—— 对象创建**

| 模式 | 意图 | 频率 |
|-----|------|------|
| **工厂方法(Factory Method)** | 定义创建对象的接口，让子类决定实例化哪个类 | 🔴 |
| **抽象工厂(Abstract Factory)** | 创建一系列相关对象，不指定具体类 | 🔴 |
| **单例(Singleton)** | 保证一个类只有一个实例 | 🔴 |
| 建造者(Builder) | 将复杂对象的构建与表示分离 | 🟡 |
| 原型(Prototype) | 通过复制原型创建新对象 | 🟢 |

**结构型模式（7个）—— 类/对象组合**

| 模式 | 意图 | 频率 |
|-----|------|------|
| **适配器(Adapter)** | 将一个类的接口转换为客户期望的另一个接口 | 🔴 |
| **代理(Proxy)** | 为其他对象提供一种代理以控制访问 | 🔴 |
| **外观(Facade)** | 为子系统提供统一的高层接口 | 🟡 |
| **装饰(Decorator)** | 动态地给对象添加额外职责 | 🟡 |
| 桥接(Bridge) | 将抽象与实现分离，使二者独立变化 | 🟡 |
| 组合(Composite) | 将对象组合成树形结构表示"部分-整体"层次 | 🟢 |
| 享元(Flyweight) | 运用共享技术有效支持大量细粒度对象 | 🟢 |

**行为型模式（11个）—— 对象间交互**

| 模式 | 意图 | 频率 |
|-----|------|------|
| **观察者(Observer)** | 定义一对多依赖，状态改变时通知所有依赖者 | 🔴 |
| **策略(Strategy)** | 定义一系列算法，将每一个封装起来，使它们可以互相替换 | 🔴 |
| **模板方法(Template Method)** | 定义算法骨架，将某些步骤延迟到子类 | 🟡 |
| **状态(State)** | 允许对象在内部状态改变时改变它的行为 | 🟡 |
| 命令(Command) | 将请求封装成对象 | 🟡 |
| 中介者(Mediator) | 用一个中介对象封装一系列对象交互 | 🟢 |
| 迭代器(Iterator) | 提供一种方法顺序访问聚合对象中各元素 | 🟢 |
| 职责链(Chain of Responsibility) | 将请求沿着处理者链传递 | 🟢 |
| 备忘录(Memento) | 捕获对象内部状态以便恢复 | 🟢 |
| 访问者(Visitor) | 在不改变类的前提下定义新操作 | 🟢 |
| 解释器(Interpreter) | 给定一个语言，定义文法表示和解释器 | 🟢 |

### 🔴 设计模式七大原则

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅
> **来源**：选择题、51CTO高频知识点

| 原则 | 核心思想 |
|-----|---------|
| **开闭原则(OCP)** | 对扩展开放，对修改关闭 |
| **里氏替换(LSP)** | 子类必须能替换父类 |
| **依赖倒置(DIP)** | 依赖抽象，不依赖具体 |
| **接口隔离(ISP)** | 使用多个专门接口而非单一总接口 |
| **单一职责(SRP)** | 一个类只有一个引起变化的原因 |
| **迪米特法则(LoD)** | 最少知识原则，只与直接朋友通信 |
| **组合/聚合复用** | 优先使用组合而非继承 |

## 4. UML 建模

> 💡 **命题趋势**：案例分析选答题常考UML图，选择题每次1-3题。需要会识别各种图并理解关系类型。

### 🔴 UML 图的分类

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅
> **来源**：选择题+案例分析

**结构图（静态）**：类图、对象图、包图、构件图、部署图、组合结构图

**行为图（动态）**：用例图、活动图、状态机图、交互图（顺序图、通信图、交互概览图、时序图）

### 🔴 类之间的关系

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅

| 关系 | 符号 | 强度 | 说明 |
|-----|------|------|------|
| **依赖** | - - - -> | 最弱 | 使用关系，临时的 |
| **关联** | ——— | 中 | 结构关系，长期的 |
| **聚合** | ◇——— | 较强 | 整体-部分，可分离(has-a) |
| **组合** | ◆——— | 强 | 整体-部分，不可分离，同生共死 |
| **泛化** | ——▷ | 强 | 继承关系(is-a) |
| **实现** | - - -▷ | 强 | 接口实现 |

## 7. 计算机组成与体系结构

### 🔴 流水线与加速比

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅
> **来源**：选择题计算题

- **流水线周期** = 最慢段的执行时间
- **总时间** = 第一条指令完整时间 + (n-1) × 流水线周期
- **加速比** = 不用流水线的时间 / 用流水线的时间
- **等时各段公式**：总时间 = (k + n - 1) × Δt，加速比 = nk/(k+n-1)
- **Amdahl 定律**：加速比 = 1 / [(1-P) + P/S]，其中P=可并行比例，S=并行加速倍数

### 🟡 存储体系

### 🟡 编码与校验

> **出题记录**：2024上✅ 2025上✅

- **曼彻斯特编码**：每位中间有跳变，自带时钟同步

## 18. 企业架构框架

### 🔴 TOGAF（The Open Group Architecture Framework）

> **出题记录**：2024上✅ 2024下✅ 2025上✅
> **来源**：官方教程第20章、选择题

**TOGAF 核心 = ADM（架构开发方法）**

ADM 由 **9个阶段** 组成（迭代循环）：

| 阶段 | 名称 | 核心任务 |
|------|------|---------|
| 预备阶段 | Preliminary | 确定架构能力、原则、工具 |
| A | 架构愿景 | 确定范围、利益相关者、约束 |
| B | 业务架构 | 描述基线和目标业务架构 |
| C | 信息系统架构 | 数据架构 + 应用架构 |
| D | 技术架构 | 描述支撑信息系统的技术基础设施 |
| E | 机会与解决方案 | 识别交付手段（项目/程序） |
| F | 迁移规划 | 制定详细实施计划和迁移路线图 |
| G | 实施治理 | 对实施进行架构合规监督 |
| H | 架构变更管理 | 管理架构生命周期中的变更 |

**TOGAF 四大架构域**：
1. **业务架构**（Business）
2. **数据架构**（Data）
3. **应用架构**（Application）
4. **技术架构**（Technology）

**🔑 必背要点**：
- TOGAF 是一个**方法论**（告诉你"怎么做"）
- ADM 是**迭代**的，不是线性的
- 核心交付物：架构愿景、架构定义文档、架构路线图、迁移规划

> **来源**：CSDN blog.csdn.net/qq_43692950/article/details/126569772、InfoQ togaf-enterprise-architecture

### 🟡 Zachman 框架

> **出题记录**：2024上✅ 2025下✅
> **来源**：选择题，通常与TOGAF对比出题

**核心概念**：6×6 矩阵（分类本体）

**六行（视角/利益相关者）**：
1. 范围/规划者（Planner）— 执行远景
2. 企业模型/所有者（Owner）— 业务概念
3. 系统模型/设计者（Designer）— 逻辑设计
4. 技术模型/建造者（Builder）— 物理设计
5. 详细描述/实施者（Implementer）— 构件配置
6. 实际系统/工人（Worker）— 运行实例

**六列（描述维度 = 5W1H）**：
- What（数据）、How（功能）、Where（网络）、Who（人员）、When（时间）、Why（动机）

**🔑 必背要点**：
- Zachman 是**分类框架**（告诉你"是什么"），**不提供方法论**
- 不能增加行或列 — 36个单元格是完备的

**TOGAF vs Zachman 对比**：

| 对比维度 | TOGAF | Zachman |
|---------|-------|---------|
| 本质 | 方法论（How to do） | 分类本体（What is） |
| 核心 | ADM 9阶段 | 6×6 矩阵 |
| 是否提供过程指导 | ✅ 有详细步骤 | ❌ 只分类不指导 |
| 架构域 | 4个（BDAT） | 6列（5W1H） |
| 适用 | 架构开发全过程 | 架构制品的组织归类 |

## 19. 分布式一致性

> 💡 **命题趋势**：2024下论文题直接考"分布式事务"，案例分析中CAP/BASE是常见知识点。2026上极大概率在案例分析或论文中出现。

### 🔴 CAP 定理

> **出题记录**：2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：案例分析 + 选择题

**三个属性**（只能同时满足两个）：
- **C（Consistency，一致性）**：所有节点在同一时间看到相同数据
- **A（Availability，可用性）**：每个请求都能收到非错误响应
- **P（Partition tolerance，分区容错性）**：网络分区时系统仍能运行

**CAP 选择**：
| 组合 | 说明 | 代表系统 |
|------|------|---------|
| **CP** | 保证一致性+分区容错，牺牲可用性 | ZooKeeper、HBase、Redis Cluster |
| **AP** | 保证可用性+分区容错，牺牲强一致 | Cassandra、DynamoDB、Eureka |
| **CA** | 保证一致性+可用性（理论上网络不分区才可能） | 单机关系数据库 |

**🔑 必背要点**：
- 分布式系统中 P 是**必须保证**的 → 实际只在 CP 和 AP 之间选
- CAP 是 Eric Brewer 在2000年提出的猜想，2002年被证明为定理

> **来源**：JavaGuide javaguide.cn/distributed-system/protocol/cap-and-base-theorem.html、HollisChuang hollischuang.com/archives/666

---

### 🔴 BASE 理论

> **出题记录**：2024上✅ 2024下✅ 2025下✅
> **来源**：案例分析 + 选择题

BASE = ACID 的替代方案，适用于大规模互联网分布式系统

| 要素 | 英文 | 含义 |
|------|------|------|
| **BA** | Basically Available | 基本可用：出现故障时允许响应时间增加或降级服务 |
| **S** | Soft State | 软状态：允许数据存在中间状态（不同节点数据暂时不一致） |
| **E** | Eventually Consistent | 最终一致性：经过一段时间后数据达到一致 |

**🔑 必背要点**：
- BASE 由 eBay 架构师 Dan Pritchett 于2008年提出
- BASE 是对 CAP 中一致性和可用性权衡的结果
- ACID 强一致 vs BASE 最终一致

> **来源**：JavaGuide、博客园 duanxz/p/5229352.html

---

### 🔴 分布式一致性协议

> **出题记录**：2024下✅（论文）2025上✅ 2025下✅
> **来源**：论文 + 案例分析

**2PC（两阶段提交）**：
- **阶段1（Prepare/投票）**：协调者询问参与者是否可以提交
- **阶段2（Commit/提交）**：协调者根据投票结果通知提交或中止
- **缺点**：同步阻塞、单点故障、数据不一致风险

**3PC（三阶段提交）**：
- 在2PC基础上增加 **CanCommit** 阶段 + **超时机制**
- 三阶段：CanCommit → PreCommit → DoCommit
- **优点**：减少阻塞（参与者超时默认提交）
- **缺点**：网络分区时仍可能不一致

**Paxos 算法**：
- Lamport 提出的**共识算法**
- 三种角色：Proposer（提议者）、Acceptor（接受者）、Learner（学习者）
- 保证在多数节点存活时能达成一致
- 难以实现，工程实践中通常用简化版

**Raft 算法**：
- Paxos 的工程化简化版
- 三种角色：Leader、Follower、Candidate
- 三个子问题：Leader选举、日志复制、安全性
- **代表实现**：etcd、Consul

**TCC（Try-Confirm-Cancel）**：
- 应用层的分布式事务
- Try：预留资源
- Confirm：确认提交
- Cancel：取消释放
- **适用**：对一致性要求高的业务场景（如转账）

**Saga 模式**：
- 长事务分解为多个本地事务 + 补偿事务
- 正向执行失败则逆向执行补偿
- **适用**：微服务间的长业务流程

> **来源**：官方教程分布式章节、JavaGuide、2024下论文真题"分布式事务"

### 🔴 七种耦合（从弱到强）

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：选择题必考，官方教程第3章

| 排序 | 耦合类型 | 定义 | 示例 |
|------|---------|------|------|
| 1（最弱） | **非直接耦合** | 两个模块之间没有直接关系 | 模块各自独立 |
| 2 | **数据耦合** | 通过参数传递简单数据 | 传递整数、字符串 |
| 3 | **标记耦合(印记耦合)** | 通过参数传递数据结构（记录/结构体） | 传递一个结构体 |
| 4 | **控制耦合** | 传递控制信息（标志、开关） | 传递 flag 决定对方行为 |
| 5 | **外部耦合** | 共享外部数据格式、通信协议或设备 | 共享文件格式 |
| 6 | **公共耦合(公共环境耦合)** | 共享全局数据区/公共变量 | 共享全局变量 |
| 7（最强） | **内容耦合** | 一个模块直接访问/修改另一个模块的内部数据或代码 | 直接 GOTO 到另一模块内部 |

**🔑 记忆口诀**：非数标控外公内（从弱到强）

### 🔴 七种内聚（从弱到强）

> **出题记录**：2023下✅ 2024上✅ 2024下✅ 2025上✅ 2025下✅
> **来源**：选择题必考

| 排序 | 内聚类型 | 定义 | 示例 |
|------|---------|------|------|
| 1（最弱） | **偶然内聚** | 模块内各元素无实质联系，只是碰巧放在一起 | 工具箱式模块 |
| 2 | **逻辑内聚** | 逻辑上相关的功能放在一起，通过参数选择执行哪个 | 一个模块处理所有I/O |
| 3 | **时间内聚** | 需要在同一时间段执行的功能放在一起 | 初始化模块 |
| 4 | **过程内聚** | 按特定次序执行的功能放在一起 | 按流程图中的顺序 |
| 5 | **通信内聚** | 使用相同输入数据或产生相同输出数据的功能 | 在同一数据上的不同操作 |

| 7（最强） | **功能内聚** | 模块内所有元素共同完成一个且仅一个功能 | sin()函数 |

**🔑 记忆口诀**：偶逻时过通顺功（从弱到强）

## 22. 面向方面编程（AOP）

> 💡 **命题趋势**：选择题1题，考AOP核心概念辨析。

### 🟡 AOP 核心概念

> **出题记录**：2024上✅ 2024下✅
> **来源**：选择题

| 概念 | 英文 | 定义 |
|------|------|------|
| **切面(Aspect)** | Aspect | 横切关注点的模块化（如日志、事务、安全） |
| **连接点(Join Point)** | Join Point | 程序执行中的某个点（如方法调用、异常抛出） |
| **切入点(Pointcut)** | Pointcut | 匹配连接点的规则/表达式 |
| **通知/增强(Advice)** | Advice | 在连接点执行的动作（Before/After/Around） |
| **织入(Weaving)** | Weaving | 将切面代码插入目标对象的过程 |

**织入时机**：
- 编译时织入（AspectJ）
- 类加载时织入
- 运行时织入（Spring AOP，基于动态代理）

**🔑 必背要点**：
- AOP 解决的问题：**横切关注点**（cross-cutting concerns）的分离
- 典型应用：日志记录、事务管理、安全控制、性能监控
- AOP 不改变原有代码（非侵入式）

> **来源**：官方教程第16章、Spring 官方文档

## 23. 嵌入式实时调度算法

> 💡 **命题趋势**：案例分析选答题每次有一道嵌入式题，考RMS/EDF判断是否可调度。

### 🟡 RMS（单调速率调度）

> **出题记录**：2023下✅ 2024下✅ 2025上✅
> **来源**：案例分析嵌入式题

- **调度策略**：优先级固定，**周期越短优先级越高**
- **可调度充分条件**（非必要条件）：

$$\sum_{i=1}^{n} \frac{C_i}{T_i} \leq n(2^{1/n} - 1)$$

其中 Cᵢ = 执行时间，Tᵢ = 周期

| n | 上界 |
|---|------|
| 1 | 1.000 |
| 2 | 0.828 |
| 3 | 0.780 |
| n→∞ | ln2 ≈ 0.693 |

**例题验证**：任务A(C=1,T=4)，任务B(C=2,T=6)，任务C(C=3,T=12)
- CPU利用率 = 1/4 + 2/6 + 3/12 = 0.25 + 0.333 + 0.25 = **0.833**
- n=3时上界 = 0.780
- 0.833 > 0.780 → **不满足充分条件**（但不代表一定不可调度，需精确分析）

### 🟡 EDF（最早截止时间优先）

> **出题记录**：2024上✅ 2025下✅
> **来源**：案例分析

- **调度策略**：动态优先级，**截止时间越近优先级越高**
- **可调度充要条件**：

$$\sum_{i=1}^{n} \frac{C_i}{T_i} \leq 1$$

- **EDF vs RMS**：

| 对比 | RMS | EDF |
|------|-----|-----|
| 优先级 | 静态（固定） | 动态（运行时变化） |
| 最优性 | 静态优先级最优 | 动态优先级最优 |
| CPU利用率上界 | n(2^(1/n)-1) ≈ 69.3% | 100% |
| 实现复杂度 | 简单 | 较复杂 |
| 过载行为 | 可预测 | 不可预测（多米诺效应） |

> **来源**：官方教程嵌入式章节、历年案例分析真题

## 27. 数据流图（DFD）

> 💡 **命题趋势**：案例分析偶尔考DFD分层+数据流补充。

### 🟡 DFD 核心概念

> **出题记录**：2023下✅ 2024下✅ 2025上✅
> **来源**：案例分析

**四种基本元素**：
1. **外部实体**（正方形/矩形）— 系统之外的数据来源/去向
2. **加工/处理**（圆形/椭圆）— 数据变换
3. **数据存储**（开口矩形/双横线）— 数据保存
4. **数据流**（箭头线）— 数据流向

**分层规则**：
- 顶层图（Context Diagram）：只有一个加工，表示整个系统
- 0层图：将顶层加工分解
- 逐层细化，子图输入输出必须与父图对应加工的输入输出**平衡**

**🔑 答题要点**：
- 父图与子图的**数据流平衡**（数量和名称一致）
- 加工至少有一个输入流和一个输出流
- 数据存储至少有一个读和一个写
- 外部实体之间不能有直接数据流

> **来源**：官方教程第3章
