# 案例分析答题技巧

> 适用于系统架构设计师、信息系统项目管理师、系统分析师等高级资格考试的案例分析（下午 I）科目。

---

## 一、考试概况

| 项目 | 说明 |
|------|------|
| 考试形式 | 机考，在线答题 |
| 考试时间 | 90 分钟 |
| 题目数量 | 共 5 题，选做 3 题（第 1 题必答，第 2-5 题选做 2 题） |
| 满分 | 75 分（每题 25 分） |
| 合格线 | 45 分 |

---

## 二、时间分配策略

### 推荐时间分配

| 阶段 | 时间 | 说明 |
|------|------|------|
| 通读全部题目 | 5-8 分钟 | 快速浏览 5 题，确定选做题目 |
| 第 1 题（必答） | 25-28 分钟 | 必答题通常是基础题，务必拿分 |
| 选做题 1 | 25-28 分钟 | 选自己最擅长的 |
| 选做题 2 | 25-28 分钟 | 选次擅长的 |
| 检查 | 5 分钟 | 检查有无遗漏 |

### 时间管理要点

1. **不要在一题上纠缠超过 30 分钟**
2. **机考环境可以来回切换题目**，先易后难
3. **每题控制在 25-28 分钟**，留出余量
4. **宁可写得简略也不要空白**，空白 = 0 分

---

## 三、选题策略

### 选题原则

1. **先看问题，再看材料** — 带着问题读材料更高效
2. **选自己熟悉的知识域** — 不要因为材料短就选，关键看问题
3. **避开完全陌生的题** — 即使其他题很长，熟悉的题拿分更稳

### 各类题型难度参考

| 题型 | 出现频率 | 难度 | 建议 |
|------|----------|------|------|
| 架构设计/架构评估 | 必考 | ⭐⭐⭐ | 必答题常考，重点准备 |
| 设计模式 | 高频 | ⭐⭐ | 套路明确，推荐选做 |
| 数据库设计 | 高频 | ⭐⭐ | ER 图、规范化，练熟即可 |
| Web 架构/ESB/微服务 | 中频 | ⭐⭐⭐ | 概念清晰即可答 |
| 嵌入式/实时系统 | 低频 | ⭐⭐⭐⭐ | 非专业慎选 |
| 信息安全 | 中频 | ⭐⭐⭐ | 需记忆较多概念 |

## 四、各题型答题框架

### 1. 架构设计题

**常见问题模式**：给出场景，分析架构风格/模式的适用性

**答题框架**：
```
1. 明确指出推荐的架构风格/模式名称
2. 简述该架构的核心特点（2-3 点）
3. 结合题目场景说明为什么适合（匹配需求）
4. 指出可能的风险和应对措施
5. 画架构图（如有要求）
```

**示例答题**：
> 推荐采用微服务架构。
> 理由：(1) 系统需要独立部署各业务模块 → 微服务支持独立部署和扩展；
> (2) 团队规模大，需并行开发 → 微服务天然支持团队自治；
> (3) 系统对可用性要求高 → 服务隔离，单个服务故障不影响整体。

### 2. 设计模式题

**常见问题模式**：根据类图/代码填空，分析使用了什么设计模式

**答题框架**：
```
1. 识别设计模式名称
2. 说明该模式的意图和结构
3. 结合题目代码/类图指出各角色对应关系
4. 补全代码/类图中的空白部分
5. 说明使用该模式的优点
```

**关键**：熟记 23 种 GoF 设计模式的类图和角色名称

### 3. 数据库设计题

**常见问题模式**：根据需求画 ER 图、做规范化分析

**答题框架**：
```
1. 识别实体和属性
2. 确定实体间关系（1:1, 1:N, M:N）
3. 画 ER 图（标注基数）
4. 进行规范化分析（指出函数依赖，判断范式等级）
5. 如需分解，给出分解后的关系模式
```

### 4. 质量属性/架构评估题

**答题框架**：
```
1. 识别题目涉及的质量属性（性能、可用性、安全性等）
2. 分析质量属性场景六要素：
   - 刺激源 → 刺激 → 环境 → 制品 → 响应 → 响应度量
3. 给出对应的架构策略
4. 分析策略间的权衡关系
```

### 5. 项目管理案例题（项管师）

**答题框架**：
```
1. 指出案例中存在的问题（逐条列举）
2. 分析产生问题的原因
3. 给出改进建议/正确做法
4. 引用相关理论或过程组支撑
```

## 五、得分点分析

### 得分技巧

1. **分条作答**：用 (1)(2)(3) 或 ①②③ 编号，阅卷老师按点给分
2. **使用专业术语**：如「质量属性场景」「架构策略」而非口语化描述
3. **结合材料**：答案中引用题目给出的具体信息，体现「审题认真」
4. **图文并茂**：有图的题一定要画图，图分占比大
5. **宁多勿少**：不确定的点也写上，多写不扣分
6. **格式清晰**：段落分明，关键词突出

### 踩分点示例

以架构评估题为例，典型得分点：
- 正确识别质量属性（2 分）
- 正确给出质量属性场景（3 分）
- 架构策略名称正确（2 分）
- 策略解释正确（3 分）
- 权衡分析合理（3 分）

## 六、常见失分原因

### 1. 审题不仔细

- ❌ 没看清「从以下 4 题中选做 2 题」
- ❌ 漏看题目中的限制条件
- ❌ 混淆「简述」和「详细分析」的答题深度

### 2. 答题格式差

- ❌ 一大段文字没有分点
- ❌ 该画图不画图
- ❌ 字迹潦草（机考环境下为排版混乱）

### 3. 知识点模糊

- ❌ 设计模式只记名字不记结构
- ❌ 架构风格特点混淆（如管道-过滤器和事件驱动）
- ❌ 质量属性策略张冠李戴

### 4. 时间分配不当

- ❌ 一题花了 40+ 分钟，后面来不及
- ❌ 选题犹豫不决浪费时间
- ❌ 在不会的题上死磕

### 5. 答非所问

- ❌ 问「有哪些问题」却回答「怎么改进」
- ❌ 问「原因分析」却列举现象

## 七、公式速记清单

### 挣值管理公式

| 指标 | 公式 | 含义 |
|------|------|------|
| SV | EV - PV | 进度偏差（正=提前） |
| CV | EV - AC | 成本偏差（正=节约） |
| SPI | EV / PV | 进度绩效指数（>1=提前） |
| CPI | EV / AC | 成本绩效指数（>1=节约） |
| EAC | BAC / CPI | 完工估算（按当前效率） |
| ETC | EAC - AC | 完工尚需估算 |
| VAC | BAC - EAC | 完工偏差 |

> 记忆口诀：**偏差看 EV 减，指数看 EV 除；前减后除都看 EV**

### PERT 三点估算

| 公式 | 说明 |
|------|------|
| 期望值 = (O + 4M + P) / 6 | 加权平均 |
| 标准差 σ = (P - O) / 6 | 离散程度 |
| 方差 = σ² = [(P - O) / 6]² | 用于关键路径方差计算 |

### 沟通渠道

| 公式 | 说明 |
|------|------|
| 渠道数 = N(N-1) / 2 | N 为团队人数 |

### 决策树 / 期望货币值

| 公式 | 说明 |
|------|------|
| EMV = Σ(概率 × 影响值) | 期望货币值 |

### 可用性 / 可靠性

| 指标 | 公式 |
|------|------|
| 可用性 | MTBF / (MTBF + MTTR) |
| MTBF | 平均故障间隔时间 |
| MTTR | 平均修复时间 |

### 功能点估算

| 指标 | 说明 |
|------|------|
| UFP | 未调整功能点 = Σ(信息域特征 × 权重) |
| FP | 功能点 = UFP × [0.65 + 0.01 × Σfi] |

## 八、机考环境注意事项

1. **提前熟悉答题界面**：考前可在官方模拟系统练习
2. **善用复制粘贴**：重复出现的术语可以复制
3. **画图工具**：了解机考系统是否提供画图功能，如无则用文字描述
4. **保存习惯**：定期检查答案是否自动保存
5. **输入法切换**：提前设置好中英文切换快捷键
6. **时间显示**：关注屏幕上的倒计时
