## 试题一：嵌入式实时系统设计

### 题目描述

嵌入式系统是当前航空、航天、船舶及工业、医疗等领域的核心技术，嵌入式系统可包括实时系统与非实时系统两种。某宇航公司长期从事航空航天飞行器电子设备的研制工作，随着业务的扩大，需要大量大学毕业生补充到科研生产部门。按照公司规定，大学毕业生必须进行相关基础知识培训，为此，公司经理安排王工对他们进行了长达一个月的培训。

### 问题1

王工在培训中指出：嵌入式系统主要负责对设备的各种传感器进行管理与控制。而航空航天飞行器的电子设备由于对时间具有很强的敏感性，通常由嵌入式实时系统进行管控，请用300字以内文字说明什么是实时系统，实时系统有哪些主要特性。

**参考答案**:

**实时系统定义**：实时系统是指能够在确定的时间内对外部事件做出响应并完成处理的计算机系统。实时系统不仅要求计算结果的正确性，还要求在规定的时间期限（deadline）内完成响应，时间约束是其核心特征。

**实时系统的主要特性**：

1. **时间约束性（Timeliness）**：系统必须在严格的时间限制内完成任务，超时可能导致系统失效
2. **确定性（Determinism）**：系统的响应时间必须是可预测的，具有确定的上限
3. **可靠性（Reliability）**：系统必须持续稳定运行，尤其在关键任务场景下不允许故障
4. **响应性（Responsiveness）**：系统能快速感知外部事件并做出响应
5. **交互性**：实时系统通常与物理环境紧密交互，通过传感器获取环境数据，通过执行器作用于环境
6. **并发性**：需要同时处理多个实时任务，要求有效的任务调度机制

### 问题2

实时系统根据应用场景、时间特征以及工作方式的不同，存在多种实时特性，大致有三种分类方法，即时间类别、时间需求和工作方式结构。填写图中(1)～(8)处空白。

**备选答案**：时限的危害程度；时间角色；弱；时间响应；固定；时限/反应时间；时间明确；输入/输出激励；时间触发；强；周期/零星/非周期；事件触发。

**参考答案**:

| 分类方式 | 子类别 | 说明 |
|---------|--------|------|
| **时间类别** | (1) 时间角色 | 系统中时间扮演的角色 |
| | (2) 时间明确 | 时间约束是否明确定义 |
| | (3) 时间响应 | 系统的时间响应特征 |
| **时间需求** | (4) 时限/反应时间 | 系统响应的时间期限 |
| | (5) 时限的危害程度 | 分为**强**实时和**弱**实时 |
| | - 强实时 | 超过时限可能导致灾难性后果 |
| | - 弱实时 | 超过时限导致性能降低但不致灾 |
| **工作方式结构** | (6) 时间触发 | 按固定时间周期触发 |
| | (7) 事件触发 | 由外部事件触发响应 |
| | (8) 周期/零星/非周期 | 任务的时间模式分类 |

### 问题3

可靠性是实时系统的关键特性之一，区分软件的错误(Error)、缺陷（Defect）、故障(Fault)和失效(Failure)概念是软件可靠性设计工作的基础。请简要说明错误、缺陷、故障和失效的定义，并在图中标出各阶段。

**参考答案**:

| 概念 | 定义 | 出现阶段 | 表现形式 |
|------|------|---------|---------|
| **错误（Error）** | 开发人员在开发过程中产生的人为失误（如编码错误、设计错误） | 开发阶段 | 不正确的逻辑、算法错误、遗漏等 |
| **缺陷（Defect）** | 错误在软件产品中的体现，存在于代码或文档中的问题 | 开发→测试阶段 | 代码中的bug、文档不一致、接口错误 |
| **故障（Fault）** | 软件运行时，缺陷被激活导致的异常状态 | 运行阶段 | 程序异常行为、计算结果偏差、逻辑路径异常 |
| **失效（Failure）** | 系统偏离其预期功能，不能完成规定的功能 | 运行阶段（对外表现） | 系统崩溃、输出错误结果、功能不可用 |

**因果链**：错误 → 缺陷 → 故障 → 失效

## 试题二：FACE宇航系统开放式软件架构

### 题目描述

某公司一直从事宇航系统研制任务，随着宇航产品综合化、网络化技术发展的需要，公司的业务量急剧增加，研制新的软件架构已迫在眉睫。公司架构师王工广泛调研了多种现代架构的基础，建议采用基于FACE (Future Airborne Capability Environment)的宇航系统开放式软件架构，以实现宇航系统的跨平台复用，实现宇航软件高质量、低成本的开发。

### 问题1（9分）

王工指出，软件开发中需求分析是根本，架构设计是核心，不考虑软件需求便进行软件架构设计很可能导致架构设计的失败，因此，如何把软件需求映射到软件架构至关重要。请从描述语言、非功能性需求描述、需求和架构的一致性等三个方面，用300字以内的文字说明软件需求到架构的映射存在哪些难点。

**参考答案**:

1. **描述语言的差异**：需求通常使用自然语言或半形式化方法（如用例图）描述，而架构设计使用组件图、部署图等形式化手段描述。两者之间缺乏统一的描述语言和标准化的转换规则，导致需求到架构的映射存在语义鸿沟，容易产生理解偏差和信息丢失。

2. **非功能性需求描述的困难**：功能需求可以较直接地映射到架构组件和接口，但非功能性需求（如性能、可靠性、安全性、可维护性等）往往难以精确量化和形式化描述。非功能需求通常是跨组件、全局性的，难以直接分配到具体的架构元素上，需要通过架构策略和机制来满足。

3. **需求和架构的一致性验证困难**：随着需求的迭代变更，确保架构与需求始终保持一致性非常困难。缺乏有效的追踪机制来验证每个需求是否在架构中得到了恰当的体现，尤其当需求发生变更时，评估其对架构的影响范围和程度存在很大挑战。

### 问题2（10分）

FACE架构包括操作系统、I/O服务、平台服务、传输服务和可移植组件等5个段。请简要说明FACE 5个段的含义。

**参考答案**:

| 段名称 | 英文 | 功能说明 |
|--------|------|---------|
| **操作系统段（OS Segment）** | Operating System | 提供底层操作系统服务，包括任务调度、内存管理、中断处理等基础系统功能，是整个架构的基础运行平台 |
| **I/O服务段（I/O Services Segment）** | I/O Services | 提供输入/输出服务的抽象层，屏蔽底层硬件I/O的差异，统一管理设备驱动和数据采集，实现硬件无关性 |
| **平台服务段（Platform Services Segment）** | Platform Services | 提供跨平台的通用服务功能，如日志管理、配置管理、健康监控、时间服务等，为上层应用提供标准化的平台级服务 |
| **传输服务段（Transport Services Segment）** | Transport Services | 提供数据传输和通信服务，支持不同组件之间的数据交换和消息传递，实现标准化的通信接口 |
| **可移植组件段（Portable Components Segment）** | Portable Components | 包含可在不同平台间移植的应用软件组件，遵循FACE标准接口规范，实现应用层的跨平台可移植性 |

### 问题3（6分）

FACE架构的核心能力是可支持应用程序的跨平台执行和可移植性，要达到可移植能力，必须解决应用程序的紧耦合和封装的障碍。请简要说明在可移植性上，应用程序的紧耦合和封装问题的主要表现及解决方案。

**参考答案**:

**紧耦合问题表现**：应用程序直接依赖特定平台的API、特定硬件驱动或特定操作系统的系统调用，导致应用与底层平台紧密绑定，更换平台时需要大量修改代码。

**封装问题表现**：应用程序未能对外部依赖（如硬件接口、通信协议、数据格式）进行有效封装，内部实现细节暴露给外部，导致组件间相互依赖，难以独立替换和移植。

**解决方案**：
1. 采用FACE标准定义的分层接口（OS接口、I/O接口、传输接口），将应用与底层平台解耦
2. 通过抽象层和适配器模式封装平台相关的实现细节
3. 遵循面向接口编程原则，应用程序仅依赖标准接口而非具体实现
4. 使用依赖注入等设计模式降低组件间的耦合度

### 题目描述

Scrum是一个增量的、迭代的敏捷软件开发过程。某软件公司计划开发一个基于Web的Scrum项目管理系统，用于支持项目团队采用Scrum敏捷开发方法进行软件开发，辅助主管智能决策。此项目管理系统提供的主要服务包括项目团队的管理、敏捷开发过程管理和工件的管理。

## 试题四：Redis数据类型与缓存策略

### 题目描述

某互联网文化发展公司因业务发展，需要建立网上社区平台，为用户提供一个对网络文化产品（如互联网小说、电影、漫画等）进行评论、交流的平台。该平台的部分功能如下：

(a) 用户帖子的评论计数器；
(b) 支持粉丝列表功能；
(c) 支持标签管理；
(d) 支持共同好友功能等；
(e) 提供排名功能，如当天最热前10名帖子排名、热搜榜前5排名等；
(f) 用户信息的结构化存储；
(g) 提供好友信息的发布/订阅功能。

该系统在性能上需要考虑高性能、高并发，以支持大量用户的同时访问。开发团队经过综合考虑，在数据管理上决定采用Redis+数据库（缓存+数据库）的解决方案。

### 问题1（10分）

Redis支持丰富的数据类型，并能够提供一些常见功能需求的解决方案。请选择题干描述的(a)~(g)功能选项，填入表中(1)~(5)的空白处。

**参考答案**:

| Redis数据类型 | 适用功能 | 说明 |
|-------------|---------|------|
| **String（字符串）** | (a) 评论计数器 | 使用INCR/DECR原子操作实现计数器功能 |
| **Hash（哈希）** | (f) 用户信息结构化存储 | 用Hash存储用户信息的多个字段（姓名、头像、简介等） |
| **List（列表）** | (b) 粉丝列表 | 有序列表，支持按时间排序的粉丝关注列表 |
| **Set（集合）** | (c)(d) 标签管理、共同好友 | 集合的交集操作可以计算共同好友；标签用集合存储实现去重 |
| **Sorted Set（有序集合）** | (e) 排名功能 | 按分数（score）自动排序，天然支持排行榜功能 |
| **Pub/Sub（发布订阅）** | (g) 好友信息发布/订阅 | Redis原生发布/订阅机制 |

### 问题2（7分）

Redis有两种持久化方式：RDB和AOF。请从磁盘更新频率、数据安全、数据一致性、重启性能和数据文件大小五个方面比较两种方式，并简要说明开发团队选择RDB的原因。

**参考答案**:

| 对比维度 | RDB | AOF |
|---------|-----|-----|
| **磁盘更新频率** | 低（按配置的时间间隔生成快照） | 高（每次写操作或每秒追加） |
| **数据安全** | 较低（可能丢失最后一次快照后的数据） | 较高（最多丢失1秒数据，everysec策略） |
| **数据一致性** | 较差（快照间隔内的数据可能丢失） | 较好（记录所有写操作，可重放恢复） |
| **重启性能** | **快**（直接加载二进制快照文件） | **慢**（需重放所有写命令） |
| **数据文件大小** | **小**（二进制压缩格式） | **大**（文本格式记录所有命令） |

**选择RDB的原因**：该网上社区平台要求7×24小时不间断服务，在系统出现宕机等故障时，能在**最短时间内**通过重启重新建立服务。RDB方式的重启恢复速度快、文件体积小，更适合快速恢复的场景。社区平台的数据允许少量丢失（评论、点赞等非关键金融数据），因此RDB的数据安全性可以接受。

### 问题3（8分）

缓存中存储当前的热点数据，Redis为每个KEY值都设置了过期时间。为了清除非热点数据，Redis选择"定期删除+惰性删除"策略。如果该策略失效，Redis内存使用率会越来越高。请简要描述该策略的失效场景，并给出三种内存淘汰机制。

**参考答案**:

**策略失效场景**：当大量key设置了过期时间，但在定期删除的随机抽样中未被选中，同时这些key也没有被客户端访问（不触发惰性删除），导致过期key无法被及时清除，积压在内存中。随着这种"漏网之鱼"越来越多，Redis内存使用率持续升高。

**三种内存淘汰机制**：

1. **allkeys-lru**：从所有key中淘汰最近最少使用（Least Recently Used）的key。适合缓存场景，保留热点数据。

2. **volatile-lru**：仅从设置了过期时间的key中，淘汰最近最少使用的key。不影响没有设置过期时间的永久key。

3. **allkeys-random**：从所有key中随机淘汰。实现简单，但无法保证热点数据的保留。

> 其他可选方案：volatile-random（从过期key中随机淘汰）、volatile-ttl（淘汰TTL最短的key）、noeviction（不淘汰，写入报错）

## 试题五：无人直升机控制系统 — 状态图与活动图

### 题目描述

某公司拟研制一款高空监视无人直升机，该无人机采用遥控-自主复合型控制实现垂直升降。该直升机飞行控制系统由机上部分和地面部分组成。其主要工作原理是地面综合控制计算机负责发送相应指令，飞控计算机按照预定程序实现相应功能。

纵向控制基本功能：
- (a) 飞控计算机加电后完成系统初始化，进入准备起飞状态
- (b) 等待起飞指令，收到后进入垂直起飞状态
- (c) 垂直起飞中飞行异常转入无线电遥控飞行状态
- (d) 达到预定高度后进入高度保持状态
- (e) 收到目标高度后进入垂直升降状态，异常转遥控飞行
- (f) 到达目标高度后进入高度保持
- (g) 收到任务结束指令后进入降落状态
- (h) 降落中异常转遥控飞行
- (i) 降到着陆高度后进入着陆状态
- (j) 遥控飞行中收到着陆指令进入着陆状态

### 问题1

状态图和活动图是软件系统设计建模中常用的两种手段，请简要说明状态图和活动图的含义及其区别。

**参考答案**:

**状态图（State Diagram）**：描述一个对象在其生命周期内所经历的各种状态以及引起状态转换的事件。状态图强调对象的行为由其状态决定，关注的是**对象状态的变化**和触发状态转换的条件。

**活动图（Activity Diagram）**：描述系统中各种活动的执行流程，包括活动的顺序、并发、分支和汇合。活动图强调的是**工作流程和业务过程**，关注的是活动之间的控制流。

**主要区别**：
| 维度 | 状态图 | 活动图 |
|------|--------|--------|
| **关注点** | 对象的状态变化 | 系统的工作流程 |
| **触发机制** | 事件驱动状态转换 | 活动完成后自动流转 |
| **并发性** | 不强调并发 | 支持并发分支（分叉/汇合） |
| **适用场景** | 描述单个对象的行为 | 描述业务流程或用例流程 |
| **基本元素** | 状态、转换、事件、动作 | 活动、转换、判定、分叉/汇合 |

### 问题2

根据纵向控制功能需求，完善状态图中的(1)～(5)。

**参考答案**:

根据题干描述的飞行控制流程：

- (1) **准备起飞** — 系统初始化完成后的初始状态
- (2) **垂直起飞** — 收到起飞指令后进入
- (3) **高度保持** — 达到预定高度或目标高度后进入
- (4) **垂直升降** — 收到新的目标高度指令后进入
- (5) **无线电遥控飞行** — 飞行异常时的安全保障状态

**状态转换关系**：
```
初始化 → 准备起飞 → [起飞指令] → 垂直起飞 → [达到预定高度] → 高度保持
                                    ↓异常                         ↓[目标高度]
                              无线电遥控飞行 ←────────── 垂直升降
                                    ↓[着陆指令]          ↓异常↑[达到目标高度]
                                    着陆 ←── 飞机降落 ← 高度保持
                                             ↑异常→遥控
```

## 试题一：质量属性场景分析 — AES-256加密与性能权衡

### 题目描述

某公司拟构建一个**安全数据管理平台**，该平台需要处理大量敏感数据的加密存储与传输。系统采用**AES-256加密算法**保障数据安全。在系统设计过程中，项目组需要对系统的质量属性进行深入分析。

### 问题1（14分）

质量属性场景的六要素及其在本系统中的应用。

**参考答案**:

| 要素 | 含义 | 本题系统示例 |
|------|------|-------------|
| **刺激源** | 生成刺激的实体 | 外部用户请求、恶意攻击者 |
| **刺激** | 到达系统的事件 | 用户上传敏感数据、暴力破解尝试 |
| **制品** | 被刺激影响的系统部分 | 加密模块、数据传输通道 |
| **环境** | 刺激发生时系统所处状态 | 正常负载运行、高并发峰值 |
| **响应** | 系统对刺激的反应 | 完成数据加密、拒绝非法请求 |
| **响应度量** | 对响应的量化衡量 | 加密延迟≤2秒、可用性≥99.9% |

### 问题2（11分）

请分析AES-256加密对系统性能和安全性的影响，并说明如何在性能与安全之间取得平衡。

**参考答案**:

**性能影响**：AES-256密钥256位，14轮加密（AES-128为10轮），计算量约比AES-128慢40%；吞吐量下降；端到端延迟增加。

**安全性提升**：2^256密钥空间暴力破解不可行；14轮加密抗差分/线性攻击更强；抗量子计算有效安全强度128位。

**平衡策略**：①硬件加速（AES-NI指令集）；②分层加密（高敏感AES-256，普通AES-128）；③AES-256-GCM模式兼顾加密和认证；④异步加密处理；⑤缓存优化减少重复加解密；⑥CTR/GCM并行化处理。

## 试题二：领域驱动设计（DDD） — 餐厅点单系统

### 题目描述

某公司基于DDD方法构建餐厅点单系统，支持菜单管理、在线点单、订单处理、支付结算等功能。采用限界上下文进行模块划分。POS终端部署在门店本地，经常遇到网络波动导致数据同步失败。

### 问题1（13分）

识别限界上下文及映射关系，说明解耦价值。

**参考答案**:

**限界上下文**：菜单管理、订单、支付、用户、出餐/配送

**映射关系**：菜单→订单（防腐层ACL）、订单→支付（发布-订阅）、订单→出餐（发布-订阅）、用户→订单（共享内核）

**解耦价值**：语义隔离、独立演化、团队自治、技术异构、故障隔离

### 问题2（12分）

针对网络波动问题，给出技术解决方案。

**参考答案**：采用**本地优先+事件驱动异步同步**方案：
1. 本地缓存（SQLite/IndexedDB）+ 乐观UI
2. 操作日志队列（Outbox Pattern）+ 网络恢复后顺序提交
3. 幂等性设计防重复提交
4. 版本号/向量时钟冲突检测
5. 最终一致性 + Saga模式处理分布式事务

## 试题三：Redis缓存击穿 — 互斥锁 vs 逻辑过期

### 题目描述

某电商系统"双十一"期间，热点key缓存过期瞬间大量请求穿透到MySQL。王工提出物理过期+互斥锁方案，李工提出逻辑过期+异步更新方案。

### 问题1（8分）

分别说明两种方案的实现原理。

### 问题2（8分）

从一致性、可用性、性能、复杂度对比优缺点。

### 问题3（9分）

推荐方案及优化建议。

**参考答案**:

| 对比维度 | 互斥锁方案 | 逻辑过期方案 |
|---------|-----------|-------------|
| 一致性 | ✅ 强一致 | ⚠️ 最终一致 |
| 可用性 | ⚠️ 锁等待阻塞 | ✅ 高可用 |
| 性能 | ⚠️ 不稳定 | ✅ 稳定快速 |
| 复杂度 | 较低 | 较高 |

**推荐**：电商场景优先逻辑过期方案（高可用+高并发），交易类数据用互斥锁（强一致）。
