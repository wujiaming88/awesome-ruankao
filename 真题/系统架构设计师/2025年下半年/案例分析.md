## 试题一（必答题）：质量属性场景分析 — AES-256加密与性能权衡

### 题目描述

某公司拟构建一个安全数据管理平台，该平台需要处理大量敏感数据的加密存储与传输。系统采用AES-256加密算法保障数据安全。项目组在进行架构设计时，需要对系统的质量属性进行分析。

题目给出了多个质量属性场景描述，要求分析和填空。

### 问题1（14分）

质量属性场景的六要素及其在本系统中的应用。请根据题目描述补充填空。

**参考答案**:

**质量属性场景六要素**：

| 要素 | 英文 | 含义 | 示例 |
|------|------|------|------|
| **刺激源** | Source of Stimulus | 生成刺激的实体 | 外部用户、内部系统组件、攻击者 |
| **刺激** | Stimulus | 到达系统的事件 | 用户请求、系统故障、安全攻击 |
| **制品** | Artifact | 被刺激影响的系统部分 | 加密模块、数据库、Web服务器 |
| **环境** | Environment | 刺激发生时系统所处的状态 | 正常运行、过载、降级模式 |
| **响应** | Response | 系统对刺激的反应 | 加密数据、记录日志、切换备份 |
| **响应度量** | Response Measure | 对响应的量化衡量 | 延迟≤2秒、可用性≥99.9% |

### 问题2（11分）

请分析采用AES-256加密对系统性能和安全性的影响，并说明如何在性能与安全之间取得平衡。

**参考答案**:

**AES-256对性能的影响**：
1. **计算开销**：AES-256密钥长度为256位，加密轮次为14轮（AES-128为10轮），计算量更大
2. **吞吐量下降**：加解密操作增加数据处理时间，影响系统吞吐量
3. **延迟增加**：每次数据读写都需要经过加解密，增加响应延迟

**AES-256对安全性的提升**：
1. **密钥空间大**：2^256种可能的密钥组合，暴力破解不可行
2. **抗分析攻击**：14轮加密提供更强的扩散和混淆效果
3. **行业标准**：符合国际安全标准要求（如FIPS 197）

**性能与安全的平衡策略**：
1. **硬件加速**：使用支持AES-NI指令集的CPU，将加密操作卸载到硬件层
2. **分层加密**：对敏感数据使用AES-256，对非敏感数据使用AES-128或不加密
3. **加密模式选择**：采用AES-256-GCM模式，同时提供加密和认证，减少额外开销
4. **异步加密**：将加密操作放入异步队列，不阻塞主业务流程
5. **缓存优化**：对频繁访问的已加密数据进行缓存，减少重复加解密

## 试题二（选做）：领域驱动设计（DDD） — 餐厅点单系统

### 题目描述

某公司基于领域驱动设计（DDD）构建一个餐厅点单系统，系统需要支持多个餐厅的菜单管理、订单处理、支付等功能。系统设计采用限界上下文（Bounded Context）进行模块划分。

### 问题1

请根据业务描述，识别该系统的限界上下文，并说明限界上下文的解耦价值。

**参考答案**:

**限界上下文划分**：

| 限界上下文 | 核心职责 | 领域模型 |
|-----------|---------|---------|
| **菜单管理上下文** | 菜品维护、价格管理、分类管理 | 菜品、类别、规格 |
| **订单上下文** | 下单、改单、取消订单 | 订单、订单项、订单状态 |
| **支付上下文** | 支付处理、退款、对账 | 支付单、支付方式、交易记录 |
| **用户上下文** | 用户注册、认证、会员管理 | 用户、会员等级、积分 |
| **配送/出餐上下文** | 出餐管理、配送跟踪 | 出餐单、配送任务 |

**限界上下文的解耦价值**：
1. **语义隔离**：每个上下文拥有独立的领域模型和通用语言，避免概念混淆（如"订单"在订单上下文和支付上下文中含义不同）
2. **独立演化**：各上下文可以独立开发、部署和扩展，不影响其他模块
3. **团队自治**：不同团队可以独立负责各自的上下文，降低协作成本
4. **技术异构**：不同上下文可以选择最适合的技术栈
5. **故障隔离**：一个上下文的故障不会蔓延到其他上下文

### 问题2

针对网络波动导致数据同步失败的问题，请给出解决方案。

**参考答案**:

针对网络波动问题，推荐采用**本地缓存+异步同步**方案：

1. **本地缓存策略**：
   - 在客户端（POS终端）维护本地SQLite数据库
   - 用户下单时先写入本地缓存，立即响应用户
   - 保证离线状态下基本功能可用

2. **异步同步机制**：
   - 使用消息队列（如RabbitMQ/Kafka）进行异步数据同步
   - 网络恢复后自动将本地缓存数据上传到服务端
   - 采用幂等性设计，防止重复提交

3. **冲突解决**：
   - 使用版本号或时间戳进行乐观并发控制
   - 冲突时以服务端数据为准，提示用户确认

4. **健康检测**：
   - 定期心跳检测网络状态
   - 自动切换在线/离线模式

## 试题三（选做）：Redis缓存 — 缓存击穿的解决方案

### 题目描述

某系统采用Redis+MySQL架构，在高并发场景下遇到了缓存击穿问题。团队中王工和李工分别提出了不同的解决方案。

王工提出采用**物理过期+互斥锁**方案：当缓存过期时，通过互斥锁确保只有一个线程去重建缓存，其他线程等待。

李工提出采用**逻辑过期+异步更新**方案：缓存中存储逻辑过期时间，即使逻辑过期也返回旧数据，同时异步更新缓存。

### 问题1

请分别说明王工和李工方案的实现原理。

**参考答案**:

**王工方案 — 物理过期+互斥锁**：
1. 缓存设置物理TTL过期时间
2. 当缓存过期（物理删除），请求访问时发现缓存不存在
3. 尝试获取分布式互斥锁（如Redis的SETNX）
4. 获取锁成功的线程：查询MySQL → 重建缓存 → 释放锁
5. 获取锁失败的线程：等待一段时间后重试获取缓存
6. 保证同一时间只有一个线程重建缓存

**李工方案 — 逻辑过期+异步更新**：
1. 缓存不设置物理TTL（永不过期），但在value中存储逻辑过期时间
2. 请求访问时检查逻辑过期时间
3. 如果未过期：直接返回缓存数据
4. 如果已过期：仍然返回旧的缓存数据（保证可用性）
5. 同时开启一个异步线程去查询MySQL并更新缓存
6. 后续请求获取到新数据

### 问题2

请对比两种方案的优缺点。

**参考答案**:

| 对比维度 | 王工方案（物理过期+互斥锁） | 李工方案（逻辑过期+异步更新） |
|---------|--------------------------|--------------------------|
| **一致性** | ✅ 强一致性：重建完成后才返回 | ⚠️ 最终一致性：可能短暂返回旧数据 |
| **可用性** | ⚠️ 较低：等待锁期间请求被阻塞 | ✅ 高：始终有数据返回 |
| **性能** | ⚠️ 锁等待导致延迟增加 | ✅ 无锁等待，响应快速 |
| **复杂度** | 较低 | 较高（需管理逻辑过期+异步线程） |
| **适用场景** | 数据强一致性要求高、并发不高 | 高并发场景、允许短暂数据不一致 |

### 问题3

请给出推荐方案。

**参考答案**:

综合考虑系统高可用和高并发要求，**推荐优先采用李工的逻辑过期方案**，原因：

1. **高可用保障**：任何时候都能返回数据，不会出现线程阻塞
2. **高并发友好**：无锁竞争，不会成为性能瓶颈
3. **用户体验好**：响应时间稳定，不受缓存重建影响

**推荐的折衷优化方案**：在逻辑过期方案基础上增加超时兜底机制：
- 异步更新设置超时时间（如5秒）
- 如果异步更新超时，标记为需更新，下次请求重试
- 对关键数据增加互斥锁保护，防止缓存雪崩
- 通过消息队列实现缓存更新的异步解耦

## 试题四（选做）：嵌入式系统 — AI与嵌入式系统融合

### 题目描述

某公司开发一款嵌入式智能设备，探讨AI技术在嵌入式系统中的应用，涉及模型部署、资源约束、边缘计算等技术。

### 问题

请说明AI与嵌入式系统融合的关键趋势和技术挑战。

**参考答案**:

**AI与嵌入式系统融合的关键趋势**：

1. **端侧推理普及**：越来越多的AI推理任务从云端迁移到边缘设备
2. **专用AI芯片**：NPU（神经网络处理单元）集成到嵌入式SoC中
3. **模型轻量化**：模型压缩技术（量化、剪枝、蒸馏）使大模型可在端侧运行
4. **TinyML生态**：针对微控制器的机器学习框架（如TensorFlow Lite Micro）日趋成熟
5. **联邦学习**：在保护隐私的前提下实现分布式模型训练

**技术挑战**：

| 挑战 | 说明 | 应对策略 |
|------|------|---------|
| **算力约束** | 嵌入式设备CPU/内存有限 | 模型量化（INT8/INT4）、算子优化 |
| **功耗限制** | 电池供电设备对功耗敏感 | 动态频率调节、模型休眠策略 |
| **实时性要求** | 需要毫秒级推理响应 | 硬件加速（NPU/DSP）、模型剪枝 |
| **存储限制** | Flash/RAM空间有限 | 模型压缩、参数共享、分块加载 |
| **散热限制** | 密封设备散热困难 | 热感知调度、降频策略 |
| **可靠性** | 工业环境对稳定性要求高 | 冗余设计、看门狗机制、OTA更新 |

**典型应用场景**：
- 智能摄像头：端侧人脸检测+云端人脸识别
- 工业质检：端侧缺陷检测+云端模型训练
- 智能家居：端侧语音唤醒+云端语义理解

## 试题五（选做）：Web应用架构

### 题目描述

某Web应用系统的架构设计与优化问题。

> ⚠️ 注：本题考生回忆内容较少，以下为基于搜索摘要推断的框架性内容。

### 问题

Web应用架构的设计与数据库优化。

**参考答案**:

**Web应用分层架构**：

| 层次 | 职责 | 技术选型 |
|------|------|---------|
| **表现层** | 用户界面展示 | HTML/CSS/JS、Vue/React |
| **控制层** | 请求路由与分发 | Spring MVC、Nginx |
| **业务逻辑层** | 核心业务处理 | Spring Boot、微服务 |
| **数据访问层** | 数据持久化操作 | MyBatis、JPA |
| **数据层** | 数据存储 | MySQL、Redis、MongoDB |

**数据库优化策略**：
1. **读写分离**：主库写、从库读，减轻主库压力
2. **分库分表**：水平拆分应对数据量增长
3. **索引优化**：合理建立索引，避免全表扫描
4. **SQL优化**：避免子查询、减少JOIN、使用批量操作
5. **缓存层**：Redis缓存热点数据，减少数据库访问
6. **连接池**：使用HikariCP等连接池管理数据库连接
