## 试题一（必答题）：架构评估 — 在线大模型训练平台

### 题目描述

某公司拟开发一个在线大模型训练平台，该平台支持用户提交Python模型代码，系统自动解析代码并分配计算资源进行训练。平台需要满足以下需求：

（1）平台应在用户提交训练任务后1分钟内分配计算资源并启动训练
（2）当数据库服务器出现故障时，系统能自动切换到备份服务器，确保服务不间断
（3）当训练服务器出现故障时，系统能自动将训练任务迁移到其他服务器继续运行
（4）在训练服务器出现故障时，系统仍然可以正常运行，并能上报故障信息，同时为用户提供训练日志、资源使用日志等各类日志
（5）平台界面应适配不同分辨率的终端设备
（6）平台应支持多语言界面，方便不同国家和地区的用户使用
（7）平台应提供快捷键功能，方便高级用户快速操作
（8）平台应支持远程自动化测试框架，方便进行功能和性能测试
（9）平台应支持多种终端（PC、平板、手机）的访问
（10）平台需要在一周内完成新功能模块的开发和集成
（11）当用户提交的模型代码包含语法错误时，系统应在10秒内给出错误提示
（12）平台应支持从第三方数据源导入训练数据

| （4） | 可用性 | 故障时仍可运行并上报，体现容错、降级与运行可观测性 |

该平台采用解释器风格架构，架构如图所示。请根据题干描述，将下图空白处补充完整，并说明该平台为什么适合用解释器风格。

- （1）程序执行的当前状态
- （2）解释器引擎

**解释器风格适用于该平台的原因：**

解释器架构风格由四个核心组件构成：解释器引擎、被解释的程序、解释器引擎的内部状态、程序执行的当前状态。

该大模型训练平台适合使用解释器风格的原因包括：

1. **动态执行**：用户提交的Python模型代码需要动态解析和执行，解释器能够逐步解析代码，无需预先编译，适合处理多样化的用户代码
2. **硬件抽象**：解释器可以屏蔽底层计算资源（GPU/CPU）的差异，为用户提供统一的编程接口
3. **即时优化**：解释器可以在执行过程中进行资源分配优化和代码优化
4. **扩展性**：通过修改解释器引擎，可以方便地支持新的模型框架和编程范式
5. **跨平台兼容性**：解释器提供了一层抽象，使得同一份代码可以在不同的硬件平台上运行

## 试题二（选做）：知识图谱 — 医药信息知识图谱构建与智能问答

### 题目描述

某研究团队基于医疗领域的大量文本与结构化数据，构建了一个面向医疗知识图谱的问答系统。该系统面向医学科研人员与临床医生，支持实体识别、关系抽取、知识检索与答案生成等功能。由于系统设计较为复杂，包含数据预处理、自然语言处理、知识图谱构建、后端服务、前端服务等模块，系统采用分层架构设计。

系统分为数据层、知识层、业务层三层。

### 问题1

根据知识图谱分层架构，补充完善架构图中的空白处。涉及业务层和知识层内部细节，如知识获取、知识管理、结构化数据、数据清洗、文本向量化等。

**参考答案**:

知识图谱系统分层架构：

| 层次 | 主要模块 | 功能说明 |
|------|---------|---------|
| **数据层** | 结构化数据、半结构化数据、非结构化数据、数据清洗、文本向量化 | 负责数据的采集、清洗和预处理 |
| **知识层** | 知识获取（实体识别、关系抽取）、知识表示、知识存储、知识管理、知识推理 | 负责知识图谱的构建与维护 |
| **业务层** | 智能问答、知识检索、可视化展示、用户管理 | 面向用户的业务功能 |

### 问题2

补充Scrapy爬虫框架的流程图中的空白处。说明异步I/O的概念及其好处。

**参考答案**:

**Scrapy爬虫流程**：

```
Start → Scheduler（调度器）→ Downloader（下载器）→ Middleware（中间件）
→ Spider（解析器）→ Pipeline（数据处理管道）→ Item输出 → End
```

核心组件：
- **Scheduler**：负责URL调度，管理待爬取的请求队列
- **Downloader**：负责发送HTTP请求，获取网页内容
- **Spider**：解析下载的网页，提取数据和新的URL
- **Pipeline**：处理Spider提取的数据，进行清洗、验证和存储

**异步I/O的介绍和好处**：

Scrapy的异步I/O架构采用事件驱动+非阻塞操作模式（基于Twisted框架），支持毫秒级响应切换和高并发请求处理。

好处包括：
1. **高并发处理**：不需要为每个请求创建线程，可以同时处理大量请求
2. **资源利用最大化**：在等待I/O操作时可以处理其他任务，CPU利用率更高
3. **网络请求高效调度**：通过事件循环实现请求的高效管理
4. **错误隔离与灵活扩展**：单个请求的失败不影响其他请求的处理

### 问题3

请说一下知识图谱的构建和智能问答如何存储实现。

**参考答案**:

**知识图谱的存储实现**：

推荐采用**图数据库**（如Neo4j）存储知识图谱：
- 图数据库以节点（Node）、关系（Relationship）、属性（Property）为核心数据模型
- 天然适合表达实体之间的复杂关系
- 支持高效的图遍历查询（如多跳关系查询）
- 使用Cypher查询语言进行灵活查询

**知识图谱的构建流程**：
1. 数据采集：通过Scrapy爬虫异步I/O爬取医疗领域文本数据
2. 数据清洗：去除噪声数据、标准化格式
3. 文本向量化：将文本转化为向量表示
4. 实体识别：从文本中抽取医疗实体（疾病、药物、症状等）
5. 关系抽取：识别实体之间的语义关系
6. 知识融合：整合多源数据，消除冲突
7. 知识存储：将三元组（实体-关系-实体）存入图数据库

**智能问答实现**：
- 通过向量检索进行语义相似度匹配
- 结合图推理在知识图谱中进行多跳推理
- 整合检索结果和推理结果生成答案
- 提升响应速度与准确度

## 试题三（选做）：Redis缓存 — 主从复制与数据持久化

### 题目描述

某电商系统采用Redis+MySQL的缓存架构，为了提高系统的高可用性，采用了Redis主从复制机制。

### 问题1

请补充全量复制的流程图中的空白处。

**参考答案**:

Redis全量复制流程：

1. 从节点（Slave）向主节点（Master）发送 `PSYNC` 命令
2. 主节点收到命令后执行 `BGSAVE` 命令，在后台生成RDB快照文件
3. 主节点在生成RDB期间，将所有新接收的写命令缓存到**复制缓冲区**
4. `BGSAVE` 完成后，主节点将RDB文件发送给从节点
5. 从节点接收并加载RDB文件，恢复数据状态
6. 主节点将复制缓冲区中积累的写命令发送给从节点
7. 从节点执行这些写命令，与主节点达到数据一致状态

关键点：
- `BGSAVE` 命令通过fork子进程生成快照，不阻塞主进程
- 两次数据同步：第一次是RDB快照，第二次是缓冲区增量命令

### 问题2

请补充增量复制（部分重同步）的流程图中的空白处。

**参考答案**:

Redis增量复制流程：

1. 从节点断线重连后，向主节点发送 `PSYNC <runid> <offset>` 命令
2. 主节点检查从节点的**复制偏移量（offset）**是否在**复制积压缓冲区（repl_backlog）**范围内
3. 如果偏移量在范围内，主节点从复制积压缓冲区中取出从节点缺失的数据
4. 主节点将增量数据发送给从节点
5. 从节点执行增量数据，与主节点达到一致

关键点：
- 增量复制依赖于复制积压缓冲区（repl_backlog），是一个固定大小的FIFO队列
- 如果偏移量超出缓冲区范围，则退化为全量复制

### 问题3

请介绍Redis的两种持久化方法，说明各自的优缺点。

**参考答案**:

**1. RDB（Redis Database Backup）持久化**

定时生成内存数据的二进制快照（默认文件dump.rdb）。

| 维度 | 说明 |
|------|------|
| **优点** | ①文件紧凑，恢复速度快（直接加载二进制数据）；②适合备份、容灾和快速重启；③对Redis性能影响较小（BGSAVE通过fork子进程执行） |
| **缺点** | ①可能丢失最后一次快照后的数据（依赖备份频率）；②BGSAVE的fork操作在大内存场景下可能短暂阻塞主线程 |

**2. AOF（Append Only File）持久化**

记录所有写命令（文本格式追加到文件appendonly.aof）。

| 维度 | 说明 |
|------|------|
| **优点** | ①数据安全性高（最多丢失1秒数据，若配置everysec）；②可读性强（日志文件可人工修复或分析） |
| **缺点** | ①文件体积大，恢复速度慢（需重放所有命令）；②长期运行后文件膨胀，需定期执行BGREWRITEAOF重写优化 |

**对比总结**：

| 对比维度 | RDB | AOF |
|---------|-----|-----|
| 数据安全性 | 较低（可能丢失分钟级数据） | 较高（最多丢失1秒数据） |
| 恢复速度 | 快 | 慢 |
| 文件大小 | 小（二进制压缩） | 大（文本命令） |
| 适用场景 | 定期备份、容灾恢复 | 数据安全要求高的场景 |

## 试题四（选做）：嵌入式系统 — 云AI与端AI辨析

### 题目描述

某公司正在开发一款智能设备产品，需要集成AI能力。产品团队在讨论采用云端AI还是终端AI（端AI）方案，或者两者结合的方案。

### 问题1

请分别说明云AI和端AI的优势与局限。

**参考答案**:

**云AI的优势与局限**：

| 维度 | 说明 |
|------|------|
| **优势** | ①算力强大：支持复杂模型训练与大规模数据处理；②全局优化：多设备数据聚合，适合宏观分析与策略制定；③模型更新方便：中心化部署，可快速更新模型 |
| **局限** | ①延迟与网络依赖：实时性差，网络波动影响服务稳定性；②隐私风险：数据传输易被截获，合规成本高；③带宽消耗：大量数据上传下载消耗网络带宽 |

**端AI的优势与挑战**：

| 维度 | 说明 |
|------|------|
| **优势** | ①低延迟：本地处理实现毫秒级响应，提升用户体验；②隐私保护：数据无需上传，符合GDPR等隐私法规要求；③成本优化：减少云端算力租赁与带宽费用；④离线可用：不依赖网络连接 |
| **挑战** | ①算力限制：需模型压缩（如蒸馏、量化）以适应终端硬件；②碎片化部署：不同设备需定制化适配，开发复杂度高；③模型更新困难：需要OTA推送，更新周期较长 |

### 问题2

请说明云端协同AI的设计思路。

**参考答案**:

云端协同AI架构设计思路：

1. **分层处理**：
   - 端侧：运行轻量级模型，处理实时性要求高的任务（如语音唤醒、简单图像识别）
   - 云端：运行大型模型，处理复杂推理和模型训练任务

2. **数据协同**：
   - 端侧预处理数据，仅上传关键特征或摘要信息到云端
   - 云端汇聚多端数据进行全局优化和模型迭代

3. **模型协同**：
   - 云端训练完整模型
   - 通过模型压缩（知识蒸馏、量化、剪枝）生成端侧轻量模型
   - 通过OTA推送模型更新

4. **弹性调度**：
   - 网络良好时，优先使用云端强大算力
   - 网络不佳时，端侧模型独立运行保证基本功能

## 试题五（选做）：区块链

### 题目描述

某场景涉及区块链技术的应用，考察区块链基础概念和应用设计。

> ⚠️ 注：本题回忆内容较少，以下为基于考生回忆整理的框架性内容。

### 问题1

区块链的核心技术和特点。

**参考答案**:

**区块链核心技术**：
1. **分布式账本**：所有节点共同维护一份完整的交易记录
2. **共识机制**：通过PoW、PoS、PBFT等算法达成节点间共识
3. **密码学技术**：哈希函数、数字签名、非对称加密保障数据安全
4. **智能合约**：自动执行的程序化合约

**区块链核心特点**：
1. **去中心化**：无需信任第三方机构
2. **不可篡改**：链式哈希结构保证数据完整性
3. **可追溯**：所有交易记录永久保存，可溯源
4. **透明性**：所有参与节点可验证数据
5. **匿名性**：通过公私钥体系实现用户匿名
