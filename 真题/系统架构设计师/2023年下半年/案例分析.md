## 试题一（必答题）：大数据架构（Lambda架构与Kappa架构）

### 题目描述

某网作为某电视台在互联网上的大型门户入口，成为某奥运会中国大陆地区的特权转播商，独家全程直播了某奥运会全部的赛事，积累了庞大稳定的用户群。这些用户在使用各类服务过程中产生了大量数据，对这些海量数据进行分析与挖掘，将会对节目的传播及商业模式变现起到重要的作用。

该奥运期间需要对增量数据在当日概览和赛事回顾两个层面上进行分析：
- **当日概览模块**：需要秒级刷新直播在线人数、网站的综合浏览量、页面停留时间、视频的播放次数和平均播放时间等千万级数据量的实时信息。传统的分布式架构采用重新计算的方式分析实时数据，在不扩充以往集群规模的情况下无法在几秒内分析出重要的信息。
- **赛事回顾模块**：需要展现自定义时间段内的历史最高在线人数、逐日播放走势、直播最高在线人数和点播视频排行等海量数据的统计信息。由于该奥运期间产生的数据通常不需要被经常索引、更新，因此要求采用不可变方式存储所有的历史数据以保证历史数据的准确性。

### 问题1（8分）

请根据Lambda架构和Kappa架构特点填写表格。

**参考答案**:

| 特征维度 | Lambda架构 | Kappa架构 |
|---------|-----------|----------|
| 数据处理方式 | 同时支持批处理和流处理两条数据链路 | 仅通过流处理一条数据链路计算并产生视图 |
| 实时性 | 速度层提供低延迟实时视图，批处理层提供高精度历史视图 | 通过流处理统一实现，实时性好 |
| 复杂度 | 需要维护两套代码（批处理+流处理），复杂度高 | 只维护一套流处理代码，复杂度低 |
| 数据一致性 | 批处理层保证最终一致性，服务层合并视图 | 通过重放(replay)机制保证一致性 |
| 适用场景 | 需要同时处理历史数据和实时数据的场景 | 对实时性要求高、历史数据重计算需求少的场景 |

大数据架构包括Lambda架构和Kappa架构。Lambda架构分解为三层：即(1)\_\_\_\_\_\_层、(2)\_\_\_\_\_\_层和(3)\_\_\_\_\_\_层。Kappa架构不同于Lambda同时计算流计算和批计算并合并视图，Kappa只会通过流计算一条的数据链路计算并产生视图。

## 试题二：Web系统架构（JWT认证与Web安全）

### 题目描述

某公司拟开发一套基于微服务架构的在线业务系统，系统包含多个独立的服务模块，用户在不同服务间进行操作时需要保持登录状态。团队在讨论用户认证与授权方案时，李工建议采用传统的基于Session的认证方案，即用户登录后在服务端创建Session并将Session ID通过Cookie返回客户端。王工则建议采用基于JWT（JSON Web Token）的认证方案。

### 问题1（8分）

请简要说明基于Session的认证方案和基于JWT的认证方案的基本原理，并从可扩展性、安全性和服务端存储三个方面对两种方案进行比较分析。

**参考答案**:

**基于Session的认证**：用户登录后，服务端创建Session对象并存储用户信息，将Session ID通过Cookie返回客户端。后续请求中客户端携带Session ID，服务端查找对应Session验证身份。

**基于JWT的认证**：用户登录后，服务端生成包含用户信息的JWT令牌返回客户端。后续请求中客户端在Authorization头携带JWT，服务端通过验证签名来确认身份，无需查询存储。

| 比较维度 | Session方案 | JWT方案 |
|---------|-----------|--------|
| **可扩展性** | 需要Session共享（Redis集群或Session复制），扩展性**较差** | 无状态，天然支持分布式部署，扩展性**好** |
| **安全性** | Session存储在服务端，客户端只持有Session ID，安全性**较好** | 令牌存储在客户端，存在被窃取风险；一旦签发无法主动作废，安全性需额外措施保障 |
| **服务端存储** | 需要服务端存储Session数据，占用内存资源 | 服务端**无需存储**，令牌自包含用户信息 |

---

### 问题2（9分）

JWT由哪三个部分组成？请分别说明每个部分的作用。

**参考答案**:

JWT由三部分组成，用"."号分隔：**Header.Payload.Signature**

1. **Header（头部）**：包含令牌类型（typ: "JWT"）和使用的签名算法（alg），如HMAC SHA256或RSA。经过Base64Url编码形成JWT的第一部分。
2. **Payload（载荷）**：包含声明(Claims)，即关于实体（通常是用户）和其他数据的声明。声明分为三种类型：
   - **注册声明**：如iss（签发者）、exp（过期时间）、sub（主题）、aud（受众）
   - **公共声明**：可自定义，但建议使用IANA注册的声明名
   - **私有声明**：双方协商约定的自定义声明
   经过Base64Url编码形成JWT的第二部分。
3. **Signature（签名）**：使用编码后的Header、编码后的Payload、一个密钥，以及Header中指定的算法进行签名，用于验证消息的完整性和发送者身份。公式：`HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)`

---

### 问题3（8分）

请说明在微服务架构中使用JWT认证时可能遇到的问题及解决方案。

**参考答案**:

1. **令牌无法主动撤销**：JWT一旦签发，在过期前无法主动使其失效。
   - **解决方案**：引入Redis黑名单机制，将需要撤销的令牌加入黑名单；或设置较短的过期时间配合Refresh Token刷新机制。

2. **令牌体积较大**：Payload中包含用户信息，相比Session ID体积大，增加网络开销。
   - **解决方案**：精简Payload中的声明，只包含必要信息。

3. **敏感信息泄露**：Payload仅经过Base64编码而非加密，可被解码查看。
   - **解决方案**：不在Payload中存放敏感信息，或使用JWE（JSON Web Encryption）对令牌进行加密。

4. **令牌续期问题**：用户长时间操作时令牌可能过期。
   - **解决方案**：采用双令牌机制（Access Token + Refresh Token），Access Token短期有效，Refresh Token长期有效用于刷新。

## 试题三：数据库与缓存（Redis数据丢失、读写分离）

### 题目描述

某电商公司拟开发一套高并发的商品管理系统，系统使用MySQL作为主要持久化数据库，Redis作为缓存层。随着业务增长，系统面临数据丢失风险和读写性能瓶颈等问题。团队围绕Redis持久化策略和数据库读写分离方案展开了讨论。

### 问题1（8分）

Redis提供了两种持久化机制：RDB快照和AOF日志。请分别说明这两种持久化机制的基本原理，并分析在什么场景下各自更为适用。

**参考答案**:

**RDB（Redis Database Backup）快照**：
- **原理**：在指定的时间间隔内，Redis将内存中的数据集快照写入磁盘，创建一个二进制的RDB文件。Redis通过fork子进程完成快照写入，父进程继续处理请求。
- **适用场景**：适合大规模数据恢复，对数据完整性要求不高的场景；适合做定期备份。
- **缺点**：两次快照之间的数据在意外宕机时会丢失。

**AOF（Append Only File）日志**：
- **原理**：以日志形式记录Redis服务器所处理的每一个写操作命令，Redis重启时通过重新执行AOF文件中的命令来恢复数据。支持三种同步策略：always（每次写）、everysec（每秒）、no（操作系统决定）。
- **适用场景**：对数据完整性要求高的场景，最多只丢失1秒数据（everysec策略）。
- **缺点**：AOF文件体积较大，恢复速度比RDB慢。

**综合建议**：生产环境通常同时开启RDB和AOF，利用RDB进行快速恢复，AOF保证数据完整性。

---

### 问题2（9分）

请说明Redis在主从复制过程中可能出现数据丢失的场景，并给出相应的解决方案。

**参考答案**:

**场景一：异步复制导致的数据丢失**
- **原因**：Redis主从之间采用异步复制，主节点接收写命令后即返回客户端成功，但数据可能尚未同步到从节点。若此时主节点宕机，未同步的数据将丢失。
- **解决方案**：配置`min-slaves-to-write`和`min-slaves-max-lag`参数，确保至少有N个从节点且延迟不超过指定秒数时才接受写入。

**场景二：脑裂（Split-Brain）导致的数据丢失**
- **原因**：在网络分区情况下，哨兵（Sentinel）将从节点提升为新主节点，而旧主节点可能仍在接收写入。网络恢复后旧主节点变为从节点，其新数据被新主节点的数据覆盖。
- **解决方案**：同样配置`min-slaves-to-write`和`min-slaves-max-lag`参数，当旧主节点无法与足够多的从节点通信时自动拒绝写入。

**场景三：AOF重写期间宕机**
- **原因**：AOF重写期间如果发生宕机，重写缓冲区中的数据可能丢失。
- **解决方案**：使用`aof-use-rdb-preamble`开启混合持久化，减少AOF重写风险。

---

### 问题3（8分）

请说明数据库读写分离的基本方案和实现方式，并分析其优缺点。

**参考答案**:

**基本方案**：
- **主库（Master）**负责写操作（INSERT、UPDATE、DELETE）
- **从库（Slave）**负责读操作（SELECT）
- 通过MySQL主从复制机制将数据从主库同步到从库

**实现方式**：
1. **应用层实现**：在应用代码中根据SQL类型路由到不同数据库连接
2. **中间件实现**：使用数据库中间件（如MyCat、ShardingSphere）自动进行读写分离路由
3. **代理层实现**：使用数据库代理（如ProxySQL）在代理层完成读写分离

**优点**：
- 分担主库读压力，提高系统吞吐量
- 提升系统可用性，主库故障时从库可快速切换
- 读操作水平扩展方便

**缺点**：
- 存在主从延迟，可能出现数据不一致（读到旧数据）
- 架构复杂度增加
- 需要处理主从切换的数据一致性问题

## 试题四：系统设计（Hibernate ORM架构）

### 题目描述

某公司拟开发一套企业级信息管理系统，系统采用Java技术栈，使用Hibernate作为数据持久化框架。架构师在进行技术选型和系统设计时，需要对Hibernate框架的架构、核心组件和工作机制进行充分理解。

### 问题1（8分）

请说明Hibernate框架的核心架构和主要组件，并简要描述各组件的作用。

**参考答案**:

Hibernate框架核心架构采用分层设计，主要组件包括：

1. **Configuration（配置对象）**：负责读取Hibernate配置文件（hibernate.cfg.xml）和映射文件（*.hbm.xml或注解），建立数据库连接属性和映射关系。是使用Hibernate的第一步。

2. **SessionFactory（会话工厂）**：
   - 由Configuration创建，是线程安全的重量级对象
   - 负责创建Session实例
   - 通常整个应用只有一个SessionFactory实例（对应一个数据库）
   - 内部维护了二级缓存和连接池

3. **Session（会话）**：
   - 轻量级非线程安全对象，是应用程序与数据库之间的主要接口
   - 提供了创建、读取、更新和删除操作的方法
   - 包含一级缓存（Session级别缓存），是必需的
   - 生命周期短，用完即关

4. **Transaction（事务）**：管理数据库事务，确保数据操作的原子性

5. **Query/Criteria**：
   - Query接口：使用HQL（Hibernate Query Language）或原生SQL进行查询
   - Criteria接口：提供面向对象的条件查询API

---

### 问题2（9分）

请解释Hibernate的一级缓存和二级缓存的概念、区别和适用场景。

**参考答案**:

**一级缓存（Session Cache）**：
- **概念**：与Session绑定的缓存，也称为Session级缓存
- **特点**：默认开启，不可关闭；生命周期与Session一致；Session关闭时一级缓存销毁
- **适用场景**：同一Session内的重复查询，避免对数据库的重复访问

**二级缓存（SessionFactory Cache）**：
- **概念**：与SessionFactory绑定的缓存，也称为进程级缓存或应用级缓存
- **特点**：默认关闭，需手动配置开启；可跨Session共享；需要第三方缓存实现（如EhCache、Redis）
- **适用场景**：读多写少的数据，如系统配置表、字典表等

| 对比维度 | 一级缓存 | 二级缓存 |
|---------|---------|---------|
| 作用范围 | Session级别 | SessionFactory级别（应用级） |
| 默认状态 | 默认开启 | 默认关闭 |
| 共享性 | 不可跨Session共享 | 可跨Session共享 |
| 生命周期 | Session关闭即销毁 | 应用运行期间有效 |
| 并发安全 | 线程私有，无并发问题 | 需要考虑并发安全 |

---

### 问题3（8分）

请简要说明Hibernate中对象的三种状态及其转换关系。

**参考答案**:

Hibernate中持久化对象有三种状态：

1. **瞬时态（Transient）**：
   - 通过new创建的对象，尚未与任何Session关联
   - 数据库中没有对应记录
   - 没有OID（对象标识符）

2. **持久态（Persistent）**：
   - 已经与Session关联，在Session缓存中存在
   - 数据库中有对应记录
   - 对其修改会在事务提交时自动同步到数据库（脏检查机制）

3. **游离态/脱管态（Detached）**：
   - 曾经与Session关联，但Session已关闭
   - 数据库中可能有对应记录
   - 有OID，但不在任何Session缓存中

**状态转换关系**：
- 瞬时态 → 持久态：调用`session.save()`或`session.persist()`
- 持久态 → 游离态：调用`session.evict()`、`session.close()`或`session.clear()`
- 游离态 → 持久态：调用`session.update()`或`session.merge()`
- 持久态 → 瞬时态：调用`session.delete()`
- 游离态 → 瞬时态：调用`session.delete()`

## 试题五：系统建模（SysML需求图）

### 题目描述

某航空电子公司拟为某型号飞机设计飞行控制系统。在需求分析阶段，系统工程师采用SysML（Systems Modeling Language，系统建模语言）对系统需求进行建模。SysML是UML的扩展，专门用于系统工程领域，支持复杂系统的规格说明、分析、设计、验证和确认。

SysML定义了需求图（Requirement Diagram）来捕获和管理系统需求，需求图能表达需求之间以及需求与其他模型元素之间的多种关系。

### 问题1（8分）

请简要说明SysML需求图中需求元素的基本结构，并列举SysML需求图中定义的主要关系类型及其含义。

**参考答案**:

**需求元素的基本结构**：
SysML中的需求用矩形框表示，包含以下属性：
- **id（标识符）**：需求的唯一标识
- **text（文本）**：需求的自然语言描述
- **name（名称）**：需求的简称

**SysML需求图的七种主要关系**：

1. **包含关系（Containment）**：一个需求包含另一个需求，表示需求的层次分解。用嵌套或树形结构表示。
2. **派生关系（DeriveReqt）**：一个需求从另一个需求派生而来，表示原始需求到派生需求的推导关系。用带`<<deriveReqt>>`标签的虚线箭头表示。
3. **满足关系（Satisfy）**：设计元素满足某个需求。用带`<<satisfy>>`标签的虚线箭头表示。
4. **验证关系（Verify）**：测试用例验证某个需求是否被满足。用带`<<verify>>`标签的虚线箭头表示。
5. **精化关系（Refine）**：模型元素精化某个需求的描述，使之更具体。用带`<<refine>>`标签的虚线箭头表示。
6. **追踪关系（Trace）**：在需求和其他模型元素之间建立通用的追踪关系。用带`<<trace>>`标签的虚线箭头表示。
7. **复制关系（Copy）**：将一个需求复制到另一个上下文中。用带`<<copy>>`标签的虚线箭头表示。

### 问题2（9分）

### 问题3（8分）

请简要比较SysML与UML的主要区别，并说明SysML在系统工程中的优势。

**参考答案**:

**SysML与UML的主要区别**：

| 比较维度 | UML | SysML |
|---------|-----|-------|
| 定位 | 面向软件系统建模 | 面向复杂系统工程建模（软件+硬件+过程+人员） |
| 图的种类 | 14种图 | 9种图（复用UML部分图，新增需求图和参数图） |
| 需求建模 | 没有专门的需求图 | 提供专门的需求图(Requirement Diagram) |
| 参数约束 | 不支持 | 提供参数图(Parametric Diagram)支持物理约束建模 |
| 结构建模 | 使用类图 | 使用块定义图(BDD)和内部块图(IBD)替代类图 |

**SysML在系统工程中的优势**：
1. 支持对软硬件混合系统进行统一建模
2. 提供需求图实现需求的可追踪性管理
3. 参数图支持系统性能和物理约束的建模与分析
4. 与UML兼容，便于软件工程师和系统工程师协作
5. 是OMG和INCOSE联合推出的标准，具有广泛的工具支持
