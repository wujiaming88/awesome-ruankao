## 试题一（必答题）：软件架构设计与评估

### 题目背景

某公司拟开发一套系统，采用微服务架构。需要对系统进行架构评估，使用质量属性效用树和质量属性场景分析。

### 【问题1】（7分）

微服务的介绍，并说明微服务的优缺点。

#### 参考答案

**微服务架构**是一种软件架构风格，将应用程序拆分为多个小型、独立部署的服务，每个服务负责特定功能，通过轻量级通信协议（如HTTP/REST）进行交互。这种架构允许各服务独立开发、测试、部署和扩展，促进敏捷开发和持续交付。

**优点**：
1. **独立部署**：各服务可独立部署，减少对其他模块的影响，提升发布频率和灵活性
2. **技术多样性**：不同服务可以使用不同的技术栈，最适合解决各自的问题
3. **可扩展性**：根据需要单独扩展某个服务，提高资源利用效率
4. **故障隔离**：一个服务出现故障不会导致整个系统崩溃
5. **团队自治**：每个团队可以独立负责一个服务

**缺点**：
1. **复杂性增加**：系统拆分成多个服务后，增加了运维和管理复杂度
2. **网络通信开销**：服务之间的通信依赖网络，可能带来延迟和性能问题
3. **数据一致性**：分布式系统的数据一致性管理更加复杂，需要额外的机制保障数据同步
4. **分布式事务**：跨服务的事务处理难度大
5. **测试复杂性**：集成测试和端到端测试更加困难

### 【问题2】（6分）

质量效用树填空。

#### 参考答案

质量效用树根节点为"效用(Utility)"，第二层为四大质量属性：

| 质量属性 | 典型场景示例 |
|---------|------------|
| **性能** | 正常负载响应时间、高峰负载处理能力 |
| **可用性** | 故障恢复时间、备用系统切换 |
| **安全性** | 支付安全、数据授权 |
| **可修改性** | 添加中间件、修改接口 |

将题干中给定的描述（如可用性、安全性、可修改性等）填入对应位置即可。

### 【问题3】（12分）

质量属性场景中，使用**六元素**来描述题目中可用性的两个场景。

#### 参考答案

**质量属性场景六元素**：

| 元素 | 定义 |
|------|------|
| **刺激源(Source)** | 某个生成该刺激的实体（人、计算机、其它任何刺激器） |
| **刺激(Stimulus)** | 指当刺激达到系统时需要考虑的条件 |
| **环境(Environment)** | 指该刺激在某些条件下发生 |
| **制品(Artifact)** | 某个制品被激励，可能是整个系统，也可能是系统的一部分 |
| **响应(Response)** | 指在激励达到后所采取的行动 |
| **响应度量(Response Measure)** | 当响应发生时，应当能够以某种方式对其进行度量 |

**可用性场景示例1**（如：故障恢复）：
- 刺激源：系统内部
- 刺激：服务器硬件故障
- 环境：正常运行时
- 制品：主服务节点
- 响应：系统检测到故障并切换到备用节点
- 响应度量：在XX秒内完成故障转移

**可用性场景示例2**（如：高可用）：
- 刺激源：外部网络
- 刺激：网络连接中断
- 环境：业务高峰期
- 制品：网络通信模块
- 响应：系统启用备用通信链路
- 响应度量：在XX分钟内恢复服务

## 试题二：系统设计与建模（UML序列图）

### 考点内容

- UML序列图中的三种消息类型
- 序列图补全填空
- 系统分析设计过程中两种交互图的选取

### 【问题1】

序列图的哪三种消息及其概念。

#### 参考答案

UML序列图中的**三种消息类型**：

1. **同步消息(Synchronous Message)**：发送者发出消息后必须等待接收者处理完毕并返回结果后，才能继续执行。用实心箭头表示。
2. **异步消息(Asynchronous Message)**：发送者发出消息后不需要等待接收者的响应，可以继续执行自己的操作。用开放箭头（线条箭头）表示。
3. **返回消息(Return Message)**：表示从被调用者返回到调用者的消息，通常携带返回值。用虚线箭头表示。

### 【问题2】

序列图补全填空题（根据题目给定的场景，将对象和消息填入序列图中空白位置）。

### 【问题3】

系统分析设计过程中两种交互图的选取。

#### 参考答案

UML中两种主要的交互图：

1. **序列图（Sequence Diagram）**：
   - 强调消息的**时间顺序**
   - 适合描述对象之间消息传递的时间序列
   - 适用于分析**单个用例**的行为

2. **通信图/协作图（Communication Diagram）**：
   - 强调对象之间的**组织结构**
   - 适合描述对象之间的关联关系
   - 适用于分析**多个对象间的协作关系**

**选择原则**：
- 关注消息的时间顺序 → 选择序列图
- 关注对象的组织结构和协作关系 → 选择通信图
- 两者在语义上是等价的，可以互相转换

## 试题三：数据库与缓存（分布式锁）

### 题目背景

某系统使用分布式架构，需要解决分布式锁问题。

### 【问题1】（9分）

使用基于数据库的分布式锁所存在的缺陷。

#### 参考答案

基于数据库实现分布式锁的**主要缺陷**：

1. **性能瓶颈**：数据库操作需要IO读写，在高并发场景下性能较差，容易成为系统瓶颈
2. **复杂性高**：需要处理锁超时、死锁检测、锁的可重入等复杂问题
3. **可靠性问题**：数据库如果是单点部署，存在单点故障风险；主从切换时可能导致锁失效
4. **扩展性差**：数据库连接数有限，大规模分布式系统中不易扩展
5. **一致性问题**：数据库主从复制延迟可能导致锁状态不一致

### 【问题2】（6分）

Redis的几种操作命令。

#### 参考答案

Redis有序集合（Sorted Set / ZSet）的常用命令：

| 操作 | 命令 | 说明 |
|------|------|------|
| **写入/添加** | `ZADD key score member` | 向有序集合中添加成员及其分数 |
| **查询** | `ZSCORE key member` | 获取成员的分数 |
| **查询** | `ZRANGE key start stop` | 按分数从低到高返回指定范围内的成员 |
| **查询** | `ZCARD key` | 获取有序集合的成员数 |
| **删除** | `ZREM key member` | 移除有序集合中的一个或多个成员 |

其他常用Redis命令：
- **SET key value**：设置键值对（字符串写入）
- **GET key**：获取键的值（字符串查询）
- **DEL key**：删除键（删除操作）

### 【问题3】（10分）

**问题一**：基于Redis的分布式锁也会存在死锁场景，举例说明。

#### 参考答案

**Redis分布式锁死锁场景举例**：

客户端A获取到了锁（SET key value NX PX timeout），在执行业务逻辑时：
1. **业务执行时间超过锁超时时间**：客户端A的业务还未执行完，锁已自动过期释放。此时客户端B获取到锁开始执行，客户端A执行完后执行DEL释放锁，实际上释放的是客户端B的锁，导致锁失效混乱。
2. **客户端崩溃未释放锁**：如果没有设置锁的过期时间，客户端获取锁后崩溃，锁永远不会被释放，造成死锁。
3. **Redis主从切换**：客户端在Master节点获取到锁后，Master宕机，锁信息未同步到Slave节点，Slave提升为新Master后，其他客户端也能获取到同一把锁。

**问题二**：基于数据库的分布式锁和基于Redis的分布式锁都存在问题，还有哪些其它的分布式锁的类型？

#### 参考答案

其他分布式锁类型：

1. **基于ZooKeeper的分布式锁**：
   - 利用ZooKeeper的临时顺序节点实现
   - 优点：可靠性高，有天然的锁超时机制（会话断开临时节点自动删除）
   - 缺点：性能不如Redis

2. **基于etcd的分布式锁**：
   - 利用etcd的Lease和Watch机制实现
   - 优点：强一致性（基于Raft协议），可靠性高
   - 缺点：部署和运维相对复杂

3. **基于Redlock算法的分布式锁**：
   - Redis作者Antirez提出的分布式锁算法
   - 需要多个独立的Redis实例，在大多数节点上获取锁才算成功

## 试题五：Web架构（数据库类型选择与MongoDB）

### 题目背景

某Web系统采用分层架构，需要选择合适的数据库类型存储不同类型的数据。

### 【问题1】（11分）

数据库类型选择。从下列选项中填入(1)到(7)。

可供选择的选项有：接口层、处理层、数据层、MySQL、HDFS、HBase、ES（Elasticsearch）

#### 参考答案

根据Web系统分层架构：
- **接口层**：对外提供API接口服务
- **处理层**：业务逻辑处理
- **数据层**：数据存储

数据库选择：
| 数据类型/需求 | 推荐数据库 | 理由 |
|-------------|-----------|------|
| 结构化关系数据 | **MySQL** | 支持ACID事务，适合关系型数据 |
| 大规模文件存储 | **HDFS** | 分布式文件系统，适合海量文件存储 |
| 大规模非结构化数据/随机读写 | **HBase** | 列族存储，适合海量数据的随机读写 |
| 全文搜索/日志检索 | **ES** | 倒排索引，适合全文检索和日志分析 |

### 【问题2】（10分）

说明MongoDB是如何把非关系型数据存入的，以及MongoDB对于存储非关系型数据的优势。

#### 参考答案

**MongoDB数据存储方式**：

MongoDB使用**文档（Document）**作为基本存储单元，采用**BSON（Binary JSON）**格式存储数据。数据组织为：
- **数据库(Database)** → **集合(Collection)** → **文档(Document)**
- 文档是键值对(key-value)的有序集合
- 同一集合中的文档可以有不同的字段结构（模式自由/Schema-free）

**存储非关系型数据的优势**：

1. **灵活的文档模型**：无需预定义Schema，文档结构可以动态变化，适合非结构化和半结构化数据
2. **嵌套文档**：支持文档嵌套，可以将相关数据存储在同一文档中，减少JOIN操作
3. **高性能**：内置内存映射机制，读写性能优异
4. **水平扩展**：原生支持分片（Sharding），可轻松进行水平扩展
5. **丰富的查询语言**：支持丰富的查询表达式，包括文档内嵌查询、正则表达式查询等
6. **高可用性**：支持副本集（Replica Set），自动故障转移
7. **支持多种数据类型**：字符串、整数、数组、嵌入文档、二进制等

### 【问题3】（4分）

说明使用热数据、温数据、冷数据分级存储的原因。

#### 参考答案

**分级存储的原因**：

数据根据访问频率和时效性分为三个层级：

| 数据级别 | 特征 | 存储方式 | 示例 |
|---------|------|---------|------|
| **热数据** | 访问频率高，时效性强 | 高性能存储（SSD/内存） | 最近的用户行为数据、实时交易数据 |
| **温数据** | 访问频率中等 | 普通磁盘存储 | 近期历史数据、月报数据 |
| **冷数据** | 访问频率低，主要用于归档 | 低成本大容量存储（HDD/对象存储） | 历史归档数据、合规留存数据 |

**分级存储的好处**：
1. **降低存储成本**：将不常访问的数据迁移到低成本存储介质
2. **提高系统性能**：热数据使用高性能存储，保证关键业务的响应速度
3. **优化资源利用**：合理分配存储资源，避免高成本存储的浪费
4. **满足合规需求**：冷数据长期归档，满足数据留存的法律法规要求
