## 一、软件架构设计（第1-15题）

### 第1题
软件架构风格定义了一组设计规则，用于识别提供所需功能的组件类型及其拓扑布局。以下关于常见架构风格的描述中，**错误**的是（）。

- A. 管道-过滤器风格中，每个过滤器都是独立的实体，各过滤器之间不共享状态
- B. 数据仓库（仓库）风格中，中央数据结构的当前状态触发进程执行时，称为黑板系统
- C. 解释器风格的核心思想是用虚拟机仿真硬件的执行过程，其优点是执行效率高
- D. 隐式调用风格中，组件不直接调用过程，而是触发或广播事件

**答案**：C

**解析**：解释器风格通过虚拟机仿真硬件执行，其**缺点**恰恰是执行效率较低（因为存在解释执行的开销），但优点是灵活性高、可移植性好。A正确，管道-过滤器各过滤器独立，不共享状态。B正确，仓库风格中若由数据状态变化驱动进程选择，即为黑板系统。D正确，隐式调用（事件驱动）风格中组件通过事件触发交互。

### 第2题
在基于架构的软件开发方法（ABSD）中，架构需求过程包括需求获取、标识构件和（）三个活动。

- A. 架构评审
- B. 需求评审
- C. 架构设计
- D. 生成类图

**答案**：B

**解析**：ABSD方法的架构需求过程由三个活动组成：需求获取、标识构件、需求评审。架构评审属于架构设计阶段的验证活动，架构设计和生成类图属于后续设计阶段的活动。

### 第3题
特定领域软件架构（DSSA）的活动包括领域分析、领域设计和领域实现。其中，领域分析阶段的主要目标是（）。

- A. 获得领域架构
- B. 获得领域模型
- C. 开发和组织可复用信息
- D. 构建领域特定的开发环境

**答案**：B

**解析**：DSSA三个阶段的目标分别为：领域分析——获得领域模型；领域设计——获得DSSA（即领域架构）；领域实现——开发和组织可复用信息，以及构建领域特定的开发和维护环境。A是领域设计的目标，C和D是领域实现的目标。

### 第4题
架构权衡分析方法（ATAM）是一种常用的架构评估方法。以下关于ATAM的描述中，**正确**的是（）。

- A. ATAM主要关注系统的功能性需求
- B. ATAM的评估结果是一个精确的架构质量度量值
- C. ATAM通过分析质量属性之间的权衡关系来评估架构
- D. ATAM只适用于已经完全实现的系统

**答案**：C

**解析**：ATAM（Architecture Tradeoff Analysis Method）的核心特点是揭示质量属性之间的权衡关系（Tradeoff）和敏感点。A错误，ATAM主要关注质量属性（非功能性需求），如性能、可用性、安全性等。B错误，ATAM不给出精确的度量值，而是识别风险点、敏感点和权衡点。D错误，ATAM可以在架构设计阶段（系统实现之前）进行评估。

### 第5题
在软件质量属性中，"系统在正常负载和突发负载下的响应能力"属于（）质量属性。

- A. 可靠性
- B. 可用性
- C. 性能
- D. 可修改性

**答案**：C

**解析**：性能（Performance）关注的是系统对事件的响应时间、吞吐量和资源利用率等。"正常负载和突发负载下的响应能力"直接描述的是性能质量属性。可靠性关注系统在规定条件下维持功能的能力；可用性关注系统正常运行的时间比例；可修改性关注系统变更的代价。

### 第6题
某电商系统采用微服务架构，为保证各服务之间的松耦合，服务间通信适合采用（）模式。

- A. 共享数据库直接访问
- B. 同步远程过程调用（RPC）
- C. 基于消息队列的异步通信
- D. 全局变量共享

**答案**：C

**解析**：微服务架构的核心原则之一是服务间松耦合。基于消息队列的异步通信能够实现服务间的解耦——发送方和接收方不需要同时在线，不直接依赖对方的接口。A违反了微服务"数据库独立"的原则；B虽然可用但属于同步耦合，服务间紧密依赖；D全局变量共享在分布式环境中不可行且高度耦合。

### 第7题
以下关于面向服务架构（SOA）与微服务架构区别的描述中，**不正确**的是（）。

- A. SOA通常使用ESB（企业服务总线）进行服务编排，微服务倾向于去中心化治理
- B. SOA服务粒度通常较粗，微服务粒度较细
- C. 微服务架构中每个服务可以独立部署，SOA中的服务通常需要统一部署
- D. SOA和微服务都不支持跨语言开发

**答案**：D

**解析**：D错误。SOA和微服务都支持跨语言开发——SOA通过标准的WSDL/SOAP协议实现跨语言互操作，微服务通过REST/gRPC等轻量级协议实现跨语言通信。A正确，ESB是SOA的核心组件，微服务倾向去中心化。B正确，SOA服务粒度粗（业务域级别），微服务粒度细（单一职责）。C正确，微服务独立部署是其核心优势。

### 第8题
在架构评估中，"敏感点"是指（）。

- A. 影响所有质量属性的架构决策
- B. 为了实现某一质量属性而影响另一质量属性的架构决策
- C. 对某一质量属性有显著影响的架构决策或组件参数
- D. 架构中存在的潜在缺陷

**答案**：C

**解析**：在ATAM架构评估中，敏感点（Sensitivity Point）是指对某一特定质量属性有显著影响的架构决策或设计参数。例如，增加消息加密的密钥长度是安全性的一个敏感点。B描述的是权衡点（Tradeoff Point），D描述的是风险点（Risk），A的描述过于笼统且不准确。

### 第9题
某系统需要支持多种数据格式（JSON、XML、CSV）的导入和导出功能，且未来可能增加新的格式。从架构设计角度，最适合采用的架构风格是（）。

- A. 主程序/子程序风格
- B. 管道-过滤器风格
- C. 客户端/服务器风格
- D. 分层架构风格

**答案**：B

**解析**：管道-过滤器风格的特点是每个过滤器独立处理数据流的转换，过滤器之间通过管道连接。对于数据格式转换场景，可以将不同格式的解析器和生成器设计为独立的过滤器，新增格式时只需添加新的过滤器组件，具有良好的可扩展性和复用性。主程序/子程序风格缺乏灵活性；C/S风格关注的是网络交互模式；分层架构风格虽有一定扩展性，但不如管道-过滤器适合数据流转换场景。

### 第10题
软件架构复用的层次由低到高依次为（）。

- A. 代码复用、设计复用、架构复用、领域复用
- B. 代码复用、架构复用、设计复用、领域复用
- C. 领域复用、架构复用、设计复用、代码复用
- D. 设计复用、代码复用、领域复用、架构复用

**答案**：A

**解析**：软件复用的层次从低到高为：代码复用（最基本，复用代码片段/函数/类库）→ 设计复用（复用设计方案、设计模式）→ 架构复用（复用软件体系结构）→ 领域复用（最高层次，复用整个领域的分析和设计成果）。层次越高，复用的抽象程度越高，复用效益也越大。

### 第11题
在"4+1"视图模型中，用于描述系统的并发与同步方面的是（）。

- A. 逻辑视图
- B. 开发视图
- C. 过程视图
- D. 物理视图

**答案**：C

**解析**：Kruchten的"4+1"视图模型中：逻辑视图关注功能需求，面向最终用户；开发视图关注软件模块的组织和管理，面向程序员；过程视图关注系统的并发、同步和通信，面向系统集成人员；物理视图关注系统的拓扑、安装和通信，面向系统工程师。场景视图（+1）是核心，用于验证其他4个视图。

在软件产品线中，核心资源包括（）。

- B. 软件架构和一组软件构件
- C. 软件架构、可复用的软件构件、领域模型、需求说明书、测试用例等

### 第13题
REST架构风格的核心约束**不包括**（）。

- A. 无状态
- B. 统一接口
- C. 分层系统
- D. 有状态会话管理

**答案**：D

**解析**：REST（Representational State Transfer）架构风格的6个核心约束是：客户端-服务器、无状态、可缓存、统一接口、分层系统、按需代码（可选）。REST要求通信必须是无状态的，服务器端不保存客户端的会话状态，因此"有状态会话管理"与REST的基本原则相矛盾。

### 第14题
某系统要求在单个服务实例出现故障时，系统仍能继续提供服务。以下架构策略中，直接支持该需求的是（）。

- A. 增加缓存层以提高性能
- B. 采用服务冗余和故障转移机制
- C. 采用代码混淆提高安全性
- D. 使用持续集成/持续部署（CI/CD）

**答案**：B

**解析**：该需求属于**可用性**质量属性——系统在组件故障时继续服务。服务冗余（Redundancy）和故障转移（Failover）是提高可用性的核心架构策略。A关注性能，C关注安全性，D关注开发效率和部署速度，均不直接满足可用性需求。

### 第15题
在架构描述语言（ADL）中，用于表示系统中组件之间交互关系的基本元素是（）。

- A. 组件（Component）
- B. 连接件（Connector）
- C. 配置（Configuration）
- D. 端口（Port）

**答案**：B

**解析**：ADL的三个基本元素为组件（Component）、连接件（Connector）和架构配置（Configuration）。组件表示计算或数据存储单元；**连接件**表示组件之间的交互关系和规则，是描述组件间通信、协调、协作的核心元素；配置描述组件和连接件的拓扑结构。端口是组件与外界交互的接口点，不是ADL的基本元素。

在需求工程中，需求验证（Validation）的主要目的是（）。

- A. 确认需求文档中的需求是否正确反映了用户的真实意图
- B. 验证系统实现是否符合需求规格说明

### 第17题
在UML中，用于描述系统内部对象之间的交互顺序（强调消息的时间序列）的图是（）。

- A. 类图
- B. 活动图
- C. 序列图（顺序图）
- D. 组件图

**答案**：C

**解析**：序列图（Sequence Diagram）是UML交互图的一种，通过垂直的时间轴展示对象之间消息传递的时间顺序。类图描述系统的静态结构（类及其关系）；活动图描述工作流或业务流程；组件图描述系统的物理组件及其依赖关系。

### 第18题
面向对象设计原则中，"高层模块不应依赖于低层模块，两者都应依赖于抽象"描述的是（）。

- A. 开闭原则（OCP）
- B. 里氏替换原则（LSP）
- C. 依赖倒置原则（DIP）
- D. 接口隔离原则（ISP）

**答案**：C

**解析**：依赖倒置原则（Dependency Inversion Principle, DIP）有两层含义：① 高层模块不应依赖低层模块，两者都应依赖抽象；② 抽象不应依赖细节，细节应依赖抽象。OCP指对扩展开放、对修改封闭；LSP指子类应能替换父类而不影响程序正确性；ISP指客户端不应依赖它不需要的接口。

### 第19题
在UML状态图中，一个对象从"空闲"状态接收到"请求"事件后变为"忙碌"状态，其中"请求"称为（）。

- A. 动作（Action）
- B. 转换（Transition）
- C. 触发事件（Trigger）
- D. 守卫条件（Guard）

**答案**：C

**解析**：UML状态图中，状态转换的要素包括：源状态、触发事件（Trigger）、守卫条件（Guard）、动作（Action）和目标状态。"请求"引起了状态从"空闲"到"忙碌"的转换，是触发这次状态转换的事件，即触发事件。转换是整个状态变化过程；动作是转换过程中执行的操作；守卫条件是转换发生的布尔条件。

敏捷开发方法Scrum中，每个Sprint结束时进行的会议中，用于团队回顾和改进的是（）。

- B. 每日站会

- D. Sprint回顾会议

### 第23题
UML中，表示类与类之间"整体-部分"关系，且部分可以独立于整体存在的是（）。

- A. 组合关系（Composition）
- B. 聚合关系（Aggregation）
- C. 泛化关系（Generalization）
- D. 依赖关系（Dependency）

**答案**：B

**解析**：聚合（Aggregation）是"弱拥有"的整体-部分关系，部分可以独立于整体存在。例如，部门与员工的关系，部门撤销后员工仍然存在。组合（Composition）是"强拥有"关系，部分不能脱离整体独立存在（如房屋与房间）。泛化是继承关系；依赖是"使用"关系。

### 第24题
在软件开发过程中，"将需求规格说明转化为软件内部结构（模块划分、接口定义等）"属于（）阶段的工作。

- A. 需求分析
- B. 概要设计
- C. 详细设计
- D. 编码实现

**答案**：B

**解析**：概要设计（也称总体设计/架构设计）阶段的主要任务是将需求转化为软件的体系结构，包括模块划分、模块间的接口定义、数据结构设计等。需求分析阶段确定"做什么"；详细设计阶段关注模块内部的算法和数据结构细节；编码实现阶段是将设计转化为代码。

### 第25题
以下关于软件测试的McCabe环路复杂度的描述中，正确的是（）。

- A. McCabe复杂度 = 边数 - 节点数 + 1
- B. McCabe复杂度 = 边数 - 节点数 + 2
- C. McCabe复杂度 = 节点数 - 边数 + 2
- D. McCabe复杂度 = 节点数 + 边数 - 2

**答案**：B

- A. 一定会
- B. 可能会
- C. 不会
- D. 无法判断

**答案**：C

**解析**：分析最坏情况：每个进程已分配3个资源（最大需求4 - 1 = 3），此时3个进程共占用 3 × 3 = 9个资源，系统中还剩 10 - 9 = 1个资源。这1个资源可以分配给任意一个进程使其满足最大需求（3+1=4），该进程可以执行完毕并释放全部4个资源，从而其他进程也可以继续执行。因此**不会**发生死锁。一般规律：当资源总数 ≥ 进程数 ×（最大需求 - 1）+ 1 时，不会死锁。10 ≥ 3 × 3 + 1 = 10，满足条件。

        if (instance == null) {
            synchronized (Singleton.class) {

                }
            }
        }
        return instance;
    }
}
```

public class AddStrategy implements Strategy {
    public double calculate(double a, double b) { return a + b; }
}
public class MultiplyStrategy implements Strategy {
    public double calculate(double a, double b) { return a * b; }
}
public class Context {
    private Strategy strategy;
    public Context(Strategy strategy) { this.strategy = strategy; }
    public double execute(double a, double b) {
        return strategy.calculate(a, b);
    }
}
```

- A. 模板方法模式（Template Method）
- B. 观察者模式（Observer）
- C. 策略模式（Strategy）
- D. 命令模式（Command）

**答案**：C

**解析**：策略模式（Strategy）定义一系列算法，将每个算法封装起来，并使它们可以互相替换。代码中`Strategy`接口定义了算法族，`AddStrategy`和`MultiplyStrategy`是具体策略，`Context`持有策略引用并委托执行。客户端可以在运行时选择不同策略。模板方法在父类定义算法骨架；观察者处理对象间的依赖通知；命令将请求封装为对象。
