### 问题2 参考答案

**（1）主备模式与多活模式的架构特点**

**主备模式（Active-Standby）**：
- 一个数据中心为主（Active），承担所有读写流量；另一个为备（Standby），实时同步主中心数据但不对外服务
- 数据同步方式：数据库主从复制（同步/半同步），Binlog实时同步
- 故障切换：主中心故障后，通过DNS切换或负载均衡器将流量切换到备中心，备中心升级为主

**多活模式（Active-Active）**：
- 两个或多个数据中心同时对外提供服务，各中心可独立处理读写请求
- 数据同步方式：双向数据复制（如MySQL Group Replication、OceanBase等分布式数据库）
- 流量调度：通过全局负载均衡（GSLB）按地域/权重将用户流量分配到就近的数据中心

**（2）四维度对比**

| 维度 | 主备模式 | 多活模式 |
|------|---------|---------|
| **可用性** | 故障切换需要30秒-数分钟（DNS生效+数据库切换）；切换期间有短暂不可用 | 任一中心故障，其他中心自动承接流量，秒级切换；可用性更高 |
| **数据一致性** | 主从复制延迟可控（半同步模式下接近0延迟）；一致性保障较好 | 双向复制存在冲突风险（两中心同时修改同一数据）；需要冲突检测与解决机制；一致性挑战大 |
| **成本** | 备中心资源大部分时间闲置，资源利用率约50% | 两个中心均在线服务，资源利用率接近100%；但需要更复杂的数据同步和冲突处理组件 |
| **运维复杂度** | 较低：主从架构成熟，运维经验丰富 | 较高：需要处理数据冲突、全局路由、多中心配置同步等问题 |

**（3）推荐方案**

**推荐：主备模式（Active-Standby）+ 读写分离**

理由：
1. **数据一致性优先**：医疗系统对数据一致性要求极高（病历、检验报告不允许出错），主备模式的数据一致性保障更强
2. **RPO需求可满足**：采用半同步复制，RPO可控制在1分钟内，满足需求（d）
3. **业务特性匹配**：医疗系统以查询为主（读写比约9:1），备中心可承担部分只读流量（报表查询、历史数据检索），提高资源利用率
4. **运维成本可控**：主备架构技术成熟，团队经验丰富，运维风险低
5. **逐步演进**：未来可在主备基础上逐步向多活演进，先解决当前需求

### 考点分析

- **质量属性效用树**：构建方法、优先级标注（重要程度+实现难度）
- **灾备架构**：主备模式 vs 多活模式的设计与对比
- **医疗信息化标准**：HL7 FHIR、EMPI、数据分类分级
- **架构评估**：结合业务特性选择合适的架构方案

## 试题二

### 问题1 参考答案

**（1）三种分布式事务方案的实现原理**

**2PC（两阶段提交）**：
- **阶段一（Prepare/投票）**：协调者向所有参与者发送Prepare请求，参与者执行本地事务但不提交（写入Undo/Redo日志），然后向协调者回复Yes/No
- **阶段二（Commit/提交）**：若所有参与者回复Yes，协调者发送Commit指令，参与者正式提交事务；若任一参与者回复No或超时，协调者发送Rollback指令，参与者回滚事务
- **协调者**：通常为单独的事务管理器（如Seata Server）

**TCC（Try-Confirm-Cancel）**：
- **Try阶段**：尝试执行业务操作，完成资源预留（如冻结账户余额），不执行真正的业务变更
- **Confirm阶段**：所有Try成功后，执行业务确认操作（如扣减冻结金额），幂等操作
- **Cancel阶段**：任一Try失败后，执行补偿操作（如解冻账户余额），幂等操作
- 各阶段逻辑由业务方自行实现，框架负责编排调用

**Saga模式**：
- 将长事务拆分为一系列本地事务（T1, T2, ... Tn），每个本地事务有对应的补偿事务（C1, C2, ... Cn）
- 正向流程：依次执行T1→T2→...→Tn，每个事务执行后立即提交
- 回滚流程：若Ti失败，依次执行Ci-1→...→C2→C1（反向补偿已完成的事务）
- 最终一致性：不保证隔离性，但通过补偿机制保证最终一致

**（2）四维度对比**

| 维度 | 2PC | TCC | Saga |
|------|-----|-----|------|
| **一致性** | 强一致性（阻塞式同步） | 最终一致性（业务层面保证） | 最终一致性（补偿机制） |
| **性能** | 低：全局锁阻塞，等待所有参与者响应 | 中：Try阶段只锁定部分资源 | 高：无全局锁，本地事务独立提交 |
| **耦合度** | 低：对业务代码侵入小 | 高：需要编写Try/Confirm/Cancel三套逻辑 | 中：需要编写正向操作+补偿操作 |
| **适用场景** | 强一致性且参与者少的场景（如数据库层面） | 短流程、高一致性要求的金融场景 | 长流程、多参与者、允许最终一致的场景 |

### 问题2 参考答案

**Saga事务流程设计（转账业务）**

**推荐：编排式（Orchestration）**

**正向流程**：

```
Saga Orchestrator（转账编排器）
  │
  ├─ Step 1: 风控服务.checkTransaction()
  │          → 检查交易合规性（反洗钱、限额等）
  │
  ├─ Step 2: 账户服务.debit()
  │          → 扣除转出方账户余额
  │
  ├─ Step 3: 账户服务.credit()
  │          → 增加转入方账户余额
  │
  ├─ Step 4: 交易服务.recordTransaction()
  │          → 记录交易流水（状态=成功）
  │
  └─ Step 5: 通知服务.sendNotification()
             → 发送交易通知（短信/APP推送）
```

**补偿流程**（任一步骤失败时反向补偿）：

| 失败步骤 | 触发的补偿操作 |
|---------|--------------|
| Step 1失败（风控不通过） | 无需补偿，直接拒绝交易 |
| Step 2失败（扣款失败） | 无需补偿，标记交易失败 |
| Step 3失败（入账失败） | C2: 账户服务.refundDebit() — 退回转出方扣款金额 |
| Step 4失败（流水记录失败） | C3: 账户服务.reverseCredit() — 撤销转入方入账；C2: 账户服务.refundDebit() — 退回转出方扣款 |
| Step 5失败（通知失败） | 不触发补偿（通知为非关键操作），记录失败日志后重试发送即可 |

**选择编排式（Orchestration）的理由**：

1. **流程可控**：转账涉及资金操作，需要严格的步骤顺序和统一的事务管理，编排器可集中控制流程
2. **补偿逻辑清晰**：编排器掌握全局状态，能精确知道需要补偿哪些步骤，避免协调式中因事件丢失导致的补偿遗漏
3. **可观测性**：编排器记录每个步骤的执行状态，便于排障和审计
4. **金融场景要求**：金融转账对可靠性要求极高，集中式编排的确定性优于去中心化协调

### 问题3 参考答案

**熔断器模式（Circuit Breaker）工作原理**

熔断器有三种状态：

```
        失败率超过阈值
  ┌─────────────────────┐
  │                     ▼
[关闭] ──────────→ [打开]
  ▲                   │
  │     探测成功       │ 超时后进入半开
  │  ┌───────────── [半开]
  │  │                │
  └──┘    探测失败     │
          ┌───────────┘
          ▼
        [打开]
```

| 状态 | 说明 |
|------|------|
| **关闭（Closed）** | 正常状态，请求正常转发到目标服务。统计失败率，当失败率超过阈值（如50%），切换到打开状态 |
| **打开（Open）** | 熔断状态，所有请求立即失败返回（快速失败），不再调用目标服务。经过超时时间（如30秒）后，自动进入半开状态 |
| **半开（Half-Open）** | 探测状态，允许少量探测请求通过。若探测请求成功，恢复到关闭状态；若失败，回到打开状态 |

**风控服务熔断策略配置**

```yaml
# 风控服务调用的熔断配置
circuit-breaker:
  # 滑动窗口统计最近10秒的请求
  sliding-window-type: TIME_BASED
  sliding-window-size: 10  # 10秒

  # 触发熔断的条件
  failure-rate-threshold: 60        # 失败率超过60%触发熔断
  slow-call-rate-threshold: 80      # 慢调用率超过80%触发熔断
  slow-call-duration-threshold: 500ms  # 超过500ms视为慢调用
  minimum-number-of-calls: 20       # 至少20次调用后才开始统计

  # 熔断持续时间
  wait-duration-in-open-state: 30s  # 打开状态持续30秒后进入半开

  # 半开状态允许的探测请求数
  permitted-number-of-calls-in-half-open-state: 5

  # 降级策略
  fallback:
    strategy: DEFAULT_PASS  # 风控服务熔断时，默认放行低风险交易
    # 高风险交易（金额>5万）进入人工审核队列
    high-risk-threshold: 50000
```

**风控服务的特殊考虑**：
1. **降级策略**：风控服务熔断时不能简单拒绝所有交易（影响业务），而是采用"降级放行+限额控制"——小额交易（<5000元）降级放行，大额交易进入人工审核队列
2. **超时设置**：风控调用超时设置为500ms（而非默认的数秒），因为转账的总响应时间不能太长
3. **隔离策略**：为风控服务分配独立的线程池（Bulkhead隔离），避免风控慢调用拖垮整个系统

### 考点分析

- **分布式事务**：2PC、TCC、Saga三种方案原理与对比
- **Saga模式**：编排式 vs 协调式，补偿机制设计
- **熔断器模式**：三种状态及转换条件、降级策略
- **微服务治理**：服务容错、限流、隔离

## 试题三

### 问题1 参考答案

**（1）集中式编排 vs 去中心化协调**

| 维度 | 集中式编排（Orchestrator） | 去中心化协调（Choreography） |
|------|--------------------------|---------------------------|
| **核心思想** | 一个中央编排器（Orchestrator）控制所有Agent的任务分配和执行顺序 | 没有中央控制器，各Agent通过监听事件/消息自主决定下一步行动 |
| **流程可见性** | 高：编排器掌握全局状态，可清晰追踪任务进度 | 低：流程分散在各Agent中，需要聚合各Agent日志才能还原全局流程 |
| **耦合度** | Agent与编排器耦合；Agent之间解耦 | Agent之间通过事件松耦合；但隐式依赖（通过事件类型） |
| **单点故障** | 编排器是单点，需要高可用保证 | 无单点，但事件总线（消息队列）是关键依赖 |
| **复杂流程支持** | 强：天然支持条件分支、并行、循环、超时等复杂控制流 | 弱：复杂条件判断需要Agent自行处理，难以实现复杂流程 |
| **可扩展性** | 增加新Agent需修改编排器逻辑 | 增加新Agent只需订阅相关事件，扩展性好 |
| **适用场景** | 有明确流程顺序的复杂任务；需要集中监控和管理 | 简单事件驱动的松耦合系统；各参与方相对独立 |

**（2）推荐方案与通信协议设计**

**推荐：集中式编排（Orchestrator模式）**

理由：
1. 软件开发任务有明确的阶段和依赖关系（需求分析→编码→测试→审查→部署），需要严格的流程控制
2. 需要支持并行处理（如编码和测试用例生成可并行）、条件分支（审查不通过需返回编码Agent修改）
3. 需要人工审批节点，编排器可方便地在流程中插入等待人工审批的节点
4. 全过程可追溯是核心需求，集中式编排天然满足

**Agent通信协议设计**：

采用基于消息队列的异步通信 + HTTP同步回调混合模式：

**消息格式（JSON）**：

```json
{
  "message_id": "msg_20260506_001",
  "task_id": "task_feature_user_register",
  "from_agent": "orchestrator",
  "to_agent": "coding_agent",
  "message_type": "TASK_ASSIGN",  // TASK_ASSIGN | TASK_RESULT | TASK_FEEDBACK | HUMAN_APPROVAL
  "timestamp": "2026-05-06T10:30:00Z",
  "payload": {
    "sub_task_id": "subtask_003",
    "action": "implement_code",
    "context": {
      "requirement": "实现用户注册功能...",
      "tech_stack": "Spring Boot + MySQL",
      "related_artifacts": ["spec_doc_url", "api_design_url"]
    },
    "constraints": {
      "timeout_seconds": 300,
      "max_retries": 3,
      "priority": "HIGH"
    }
  },
  "trace_id": "trace_20260506_feature_001",
  "parent_message_id": null
}
```

**通信机制**：
- **任务分派**：Orchestrator → RabbitMQ/Kafka → Agent（异步，支持削峰）
- **结果回报**：Agent → RabbitMQ → Orchestrator（异步）
- **状态查询**：Orchestrator ↔ Agent（HTTP同步，用于健康检查和进度查询）
- **人工审批**：Orchestrator → 审批系统（Webhook通知）→ 审批结果回调Orchestrator

### 问题2 参考答案

**（1）Agent故障检测与恢复策略**

| 机制 | 实现方式 |
|------|---------|
| **心跳检测** | 每个Agent每30秒向Orchestrator发送心跳消息；连续3次未收到心跳（90秒）判定Agent故障 |
| **任务超时检测** | Orchestrator为每个子任务设置超时时间（如编码任务5分钟），超时未返回结果视为任务失败 |
| **健康检查** | Agent暴露HTTP `/health` 端点，Orchestrator定期主动探测 |
| **自动恢复** | Agent以容器形式运行（Kubernetes Pod），故障后由K8s自动重启；重启后向Orchestrator重新注册 |
| **任务重新分配** | Agent故障后，Orchestrator将该Agent的未完成任务重新分配给同类型的备用Agent |

**（2）任务重试与幂等性保证**

- **重试策略**：采用指数退避重试（1s→2s→4s→8s），最多重试3次。超过重试次数后将任务转入死信队列
- **幂等性保证**：
  - 每个任务分配唯一的`sub_task_id`，Agent在执行前检查该任务是否已经执行过
  - Agent将任务执行结果与`sub_task_id`绑定存储，重复投递时直接返回已有结果
  - Orchestrator在接收结果时按`message_id`去重，防止重复处理

**（3）降级策略**

| 场景 | 降级方案 |
|------|---------|
| 编码Agent不可用 | 将任务降级为"生成代码框架+人工补充"，输出带有TODO标记的骨架代码 |
| 测试Agent不可用 | 跳过自动测试，标记为"需要人工测试"，流程继续推进 |
| 审查Agent不可用 | 降级为基础的静态代码分析（如SonarQube自动扫描），不进行架构级审查 |
| LLM服务不可用 | 切换到备用LLM模型（如从GPT-4o切换到本地部署的Qwen2.5）；若所有LLM均不可用，暂停任务等待恢复 |

**（4）死信队列（DLQ）的应用**

- **进入DLQ的条件**：任务重试3次仍然失败；任务执行超时（5分钟）；Agent返回不可重试的错误（如"代码无法编译，需求描述不清"）
- **DLQ处理流程**：
  1. 任务进入DLQ后，Orchestrator发送告警通知（企业IM/邮件）
  2. 运维人员在管理界面查看DLQ中的任务详情（失败原因、重试记录、Agent日志）

**（2）数据存储方案**

**选择时序数据库（TSDB）的理由**：
1. **数据模型匹配**：工业设备数据天然是时序数据——以时间戳为主键，附带设备ID和多个指标值
2. **高写入吞吐**：TSDB针对写入密集场景优化，支持每秒百万级数据点写入（本系统需5万/秒）
3. **高效压缩**：时序数据的时间戳和相邻数据点变化小，TSDB的列式存储+专用压缩算法可实现10:1甚至更高压缩比，PB级数据的存储成本可控
4. **时间范围查询优化**：天然支持按时间范围的快速查询、降采样（Downsampling）和聚合运算
5. **数据生命周期管理**：支持自动过期（TTL）和冷热分层存储

**时序数据库对比**：

| 维度 | TDengine | TimescaleDB | InfluxDB |
|------|----------|-------------|----------|
| **架构** | 分布式原生（C语言） | 基于PostgreSQL扩展 | 独立架构（Go） |
| **写入性能** | 极高（单节点100万点/秒） | 高（依赖PG优化） | 高 |
| **SQL支持** | 类SQL + 超级表概念 | 完全兼容PostgreSQL SQL | InfluxQL / Flux |
| **集群能力** | 原生分布式，水平扩展 | 社区版单节点，企业版分布式 | 企业版支持集群 |
| **生态** | 国产，适合物联网场景 | 依托PG庞大生态 | 监控领域标准 |
| **推荐场景** | 海量IoT设备数据（本系统推荐） | 需要关系查询+时序混合的场景 | 运维监控 |

**（1）数据清洗和异常检测**

Flink流处理Job实现：
- **数据清洗**：过滤空值/畸形数据、数值范围校验（如温度-50~500°C）、时间戳校正、单位标准化
- **异常检测**：
  - **阈值检测**：单指标超过预设上下限（如温度>85°C）
  - **变化率检测**：指标在短时间内剧烈变化（如1分钟内振动值增长200%）
  - **统计检测**：基于滑动窗口的均值和标准差，偏离3σ以上视为异常

**（2）滑动窗口聚合统计**

```java
// Flink滑动窗口：5分钟窗口，1分钟滑动步长
dataStream
  .keyBy(record -> record.getDeviceId())
  .window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
  .aggregate(new DeviceMetricsAggregator())
  // 输出：每台设备每分钟的avg/max/min/std指标值
```

- 窗口大小5分钟，滑动步长1分钟：每分钟计算最近5分钟的聚合统计
- 聚合维度：按设备ID分组，计算温度/振动/压力的平均值、最大值、最小值、标准差
- 输出结果写入TSDB，供大屏展示和趋势分析

**（3）复杂告警规则引擎**

支持多条件组合的告警规则：

```
规则示例：
  IF  设备类型 = "数控机床"
  AND 温度.5min_avg > 80°C
  AND 振动.变化率 > 50%
  AND 上一次告警时间 > 10分钟前  // 告警抑制，防止重复告警
  THEN 触发"设备过热+异常振动"二级告警
       → 推送工厂维护人员（企业微信/短信）
       → 写入告警表
       → 关联该设备近1小时的原始数据，生成告警上下文
```

实现方式：
- 使用Flink CEP（Complex Event Processing）引擎定义复杂事件模式
- 告警规则存储在数据库中，支持动态加载和热更新
- 告警等级分为四级：提示→警告→严重→紧急，不同等级触发不同响应流程
- 告警抑制：同一设备同一规则在抑制时间内不重复告警

**应对百万级并发的架构策略**：

1. **试卷预加载**：考试开始前10分钟，通过CDN预分发加密的试卷数据到客户端缓存，考试开始时仅需下发解密密钥（极小数据量），避免开始瞬间的流量洪峰

3. **分区隔离**：按地区或考试场次将考生分配到不同的服务集群，避免单点过载

6. **限流降级**：API Gateway对非核心接口（如排行榜、历史记录）限流；核心接口（答题提交）保障优先

**（2）答案实时保存机制**

**多级保存策略**：

| 层级 | 机制 | 触发时机 | 恢复方式 |
|------|------|---------|---------|
| **L1: 浏览器本地** | LocalStorage + IndexedDB自动保存 | 每10秒自动保存 + 每次答题切换 | 刷新页面/重新打开时从本地恢复 |
| **L2: 服务端缓存** | WebSocket实时同步到Redis | 每30秒增量同步 + 答案变更时 | 换设备登录时从Redis恢复 |
| **L3: 持久化存储** | 异步写入MySQL（消息队列削峰） | 每2分钟批量持久化 | 服务端故障恢复时从MySQL恢复 |

**具体实现**：
1. **前端**：答案变更时先写入LocalStorage（同步）+ IndexedDB（异步），再通过WebSocket推送到服务端
2. **WebSocket长连接**：维持考生与服务端的长连接，断线自动重连（指数退避），重连后增量同步本地未同步的答案
3. **服务端**：WebSocket收到答案后先写Redis（毫秒级），然后通过Kafka异步写入MySQL
4. **断网保护**：检测到离线时，切换为纯本地模式，所有答案保存在LocalStorage/IndexedDB；恢复网络后自动同步
5. **最终一致**：交卷时比对本地数据和服务端数据的版本号，以最新版本为准

### 问题2 参考答案

**OJ安全沙箱架构**

**（1）核心安全限制**

| 限制类型 | 具体措施 |
|---------|---------|
| **时间限制** | 每次代码执行限制CPU时间（如5秒）和Wall时间（如10秒），超时强制终止 |
| **内存限制** | 限制进程内存使用（如256MB），超过限制OOM Kill |
| **文件系统限制** | 只读挂载系统文件，禁止写入（除临时目录）；禁止访问 /proc, /sys, /dev 等敏感目录 |
| **网络限制** | 完全禁止网络访问（drop所有出入流量），防止数据泄漏或攻击外部系统 |
| **进程限制** | 限制最大进程/线程数（如10个），禁止fork炸弹；禁止创建子进程 |
| **系统调用限制** | 使用seccomp-bpf白名单机制，仅允许必要的系统调用（read/write/mmap/exit等），禁止exec/socket/kill等危险调用 |

**（2）实现技术方案**

采用**容器隔离 + 安全沙箱**双层架构：

- **外层：容器隔离（Docker/gVisor）**
  - 每个代码执行任务启动独立的Docker容器（或gVisor沙箱容器）
  - 容器使用精简的基础镜像（Alpine + 语言运行时），无额外工具
  - 容器设置资源限制（--cpus, --memory, --pids-limit）
  - 容器网络设置为none（无网络）

- **内层：seccomp + namespace**
  - Linux Namespace隔离（PID/NET/MNT/USER namespace）
  - seccomp-bpf白名单过滤系统调用
  - 以无权限用户（nobody）运行代码
  - chroot到最小化文件系统

**（3）防止恶意代码攻击**

| 攻击类型 | 防御措施 |
|---------|---------|
| **Fork炸弹** | pids-limit限制进程数量；cgroup限制 |
| **死循环/CPU耗尽** | CPU时间限制 + Wall时间限制，超时SIGKILL |
| **内存耗尽** | memory限制 + OOM Killer |
| **读取敏感文件** | chroot隔离 + 只读文件系统 + 最小化目录 |
| **网络攻击/数据泄漏** | 网络完全隔离（network=none） |
| **提权攻击** | 非root用户运行 + 禁用capabilities + seccomp过滤 |
| **编译炸弹（如C的#include嵌套）** | 编译阶段同样有时间和资源限制 |

### 问题3 参考答案

**前端监控措施**

| 措施 | 说明 |
|------|------|
| **切屏检测** | 监听`visibilitychange`和`blur`事件，记录考生切换到其他窗口/标签页的次数和持续时间；超过阈值（如3次）发出警告 |
| **摄像头监控** | 调用WebRTC API开启摄像头，通过AI人脸检测判断考生是否在位、是否有多人画面；异常时截图上报 |
| **剪贴板禁用** | 禁用Ctrl+C/V、右键菜单、拖拽操作，防止外部复制粘贴答案 |

**网络传输措施**

| 措施 | 说明 |
|------|------|
| **试卷加密传输** | 试卷内容使用AES-256加密传输，密钥通过安全通道（TLS+考试开始时下发）分发，防止提前泄题 |
| **答案签名** | 每次答案提交附带HMAC签名（含时间戳+考生ID+答案hash），防止篡改和重放攻击 |
| **IP+设备绑定** | 考试开始后绑定IP和设备指纹，中途更换设备需要重新认证 |

**后端分析措施**

| 措施 | 说明 |
|------|------|
| **答案相似度检测** | 考试结束后，对所有考生的主观题答案进行文本相似度分析（如SimHash、TF-IDF余弦相似度），相似度超过阈值（如85%）标记为可疑 |
| **答题行为分析** | 分析答题时间模式（如选择题5秒内完成、编程题复制粘贴痕迹）、答题顺序异常（正确率与时间不匹配）等行为特征 |
| **IP关联分析** | 检测同一IP/网段下的多个考生，分析是否存在集体作弊 |

**平衡防作弊与隐私保护**

1. **最小化采集**：摄像头仅在考试期间开启，仅保存异常时刻的截图而非全程录像；考试结束后按时删除
2. **知情同意**：考试前明确告知考生将采集的数据类型和用途，获得书面（电子）同意
3. **数据脱敏**：行为分析系统使用脱敏的考生ID，与真实身份信息隔离存储
4. **合规审计**：建立数据访问审计机制，只有授权人员可查看监控数据
5. **分级策略**：普通考试采用轻量级防作弊（切屏检测+答案相似度），高等级考试（如认证考试）才启用摄像头监控

### 考点分析

- **高并发架构**：流量洪峰应对（预加载、弹性伸缩、限流降级）
- **数据安全**：实时保存机制、多级保障
- **安全沙箱**：容器隔离、seccomp、namespace
- **安全架构**：防作弊与隐私保护的平衡
