## 七、英语阅读（第71-75题）

**Read the following passage and answer Questions 71-75.**

Service Mesh has emerged as a critical infrastructure layer for managing service-to-service communication in microservices architectures. Unlike traditional approaches where networking logic is embedded within application code, a service mesh deploys a network of lightweight proxies (known as "sidecars") alongside each service instance. These proxies intercept all inbound and outbound traffic, providing capabilities such as load balancing, service discovery, encryption, authentication, and observability without requiring changes to the application code.

The architecture of a service mesh typically consists of two planes: the **data plane** and the **control plane**. The data plane is composed of the sidecar proxies that handle the actual network traffic between services. The control plane provides the management interface for configuring and monitoring the behavior of the data plane proxies. For example, in Istio—one of the most widely adopted service mesh implementations—Envoy serves as the data plane proxy, while istiod acts as the unified control plane component.

One of the key benefits of a service mesh is that it enables **zero-trust networking** within the cluster. By enforcing mutual TLS (mTLS) between all services, the mesh ensures that all inter-service communication is encrypted and authenticated, regardless of whether the services are running in a trusted network. This approach aligns with modern security principles where no implicit trust is granted based solely on network location.

However, adopting a service mesh introduces additional complexity and resource overhead. Each sidecar proxy consumes CPU and memory, and the additional network hop can increase latency. Organizations must carefully evaluate whether the operational benefits—improved observability, security, and traffic management—justify the increased infrastructure costs, particularly for smaller deployments where simpler alternatives may suffice.
