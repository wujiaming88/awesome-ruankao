## 试题一（必答题）：架构评估 — 智慧供应链协同平台

### 【说明】

某大型制造企业拟构建一个**智慧供应链协同平台**，用于整合上游供应商、自有工厂、下游经销商及物流服务商的业务数据，实现从原材料采购到成品交付的全链路数字化管理。平台将采用微服务架构，部署在混合云环境（私有云+公有云）上，核心交易数据存储在私有云，弹性计算资源使用公有云。

在系统需求分析阶段，项目组识别出以下关键需求：

（a）供应商提交的采购订单应在500ms内得到系统确认响应。
（b）系统应支持同时接入5000家供应商在线协同操作，峰值并发请求达到每秒2万次。
（c）当私有云核心交易服务器发生故障时，系统应在30秒内完成主备切换，确保交易数据零丢失。
（d）系统应对所有跨企业的数据传输进行端到端加密，并实施基于角色的细粒度访问控制。
（e）平台需要在3个工作日内完成新供应商的对接集成，包括API适配和数据格式转换。
（f）系统提供多维度的供应链数据可视化看板，支持自定义报表和数据钻取功能。
（g）平台支持移动端（Android/iOS）和PC端同时访问，并保持操作体验一致。
（h）系统应支持在不停机的情况下对单个微服务进行独立升级和回滚。
（i）当公有云弹性计算节点出现故障时，系统应在2分钟内自动将任务迁移到健康节点上继续执行。
（j）系统应支持通过标准EDI（电子数据交换）协议和RESTful API两种方式与外部系统对接。
（k）物流追踪模块应每5秒更新一次货物位置信息，在正常网络条件下用户查看物流轨迹的延迟不超过3秒。
（l）系统日志应完整记录所有关键业务操作，支持审计追溯，日志保留期不少于3年。

在架构评审会上，架构师王工提出采用**事件驱动架构（EDA）**来实现供应链各环节的协同。但李工认为应该采用**面向服务的架构（SOA）**，理由是SOA的服务治理能力更成熟，更适合企业级集成场景。

## 试题二（选做）：多租户SaaS平台数据隔离架构

### 【说明】

某软件公司拟构建一个**多租户SaaS项目管理平台**，面向中小企业提供项目管理、任务协作、文档管理和工时统计等功能。平台计划支持3000+企业租户，每个租户的用户数从10人到5000人不等。不同租户的数据必须严格隔离，确保A企业的项目数据对B企业完全不可见。

系统采用Spring Boot + MySQL技术栈，部署在公有云Kubernetes集群上。在技术方案讨论会上，关于多租户的数据隔离方案，团队成员提出了三种候选方案：

**方案A（独立数据库模式）**：为每个租户分配一个独立的MySQL数据库实例，应用层通过租户ID路由到对应的数据库。

**方案B（共享数据库、独立Schema模式）**：所有租户共享同一个MySQL实例，但每个租户使用独立的Schema（数据库名），各Schema内的表结构相同。

**方案C（共享数据库、共享Schema模式）**：所有租户共享同一个MySQL实例和同一套表，通过在每张表中增加`tenant_id`字段来区分不同租户的数据。

平台的核心数据表设计如下：
- 项目表（project）：预计总数据量500万行
- 任务表（task）：预计总数据量5000万行
- 文档表（document）：预计总数据量2000万行
- 用户表（user）：预计总数据量100万行

系统在试运行阶段发现以下问题：某些大租户（用户数超过2000人）的复杂查询（如跨项目的工时统计报表）响应时间超过10秒，严重影响用户体验，且这些大租户的慢查询对同一数据库实例上的其他小租户也产生了性能影响。

### 【问题1】（8分）

请从**数据隔离性、运维复杂度、资源利用率、扩展性**四个维度对比方案A、B、C三种多租户数据隔离方案的优缺点，并以表格形式展示。

### 【问题2】（8分）

针对该平台的业务场景（3000+租户、租户规模差异大），请推荐你认为最合适的数据隔离方案（可以是单一方案或组合方案），并说明理由。

### 【问题3】（9分）

针对试运行中发现的大租户慢查询影响其他租户的性能问题（"噪声邻居"问题），请给出至少3种具体的优化措施，并说明每种措施的实现原理和预期效果。

## 试题三（选做）：Serverless与事件驱动架构 — 在线教育实时互动平台

### 【说明】

某在线教育公司拟构建一个**实时互动教学平台**，支持直播授课、实时答题、课堂互动、课后作业批改和学习数据分析等功能。平台的核心特点是**流量波动极大**：上课时段（工作日晚18:00-22:00、周末全天）并发用户可达50万，而非上课时段用户量不足1000。为了降低成本并提高弹性，架构师决定采用**Serverless架构+事件驱动架构**的组合方案。

系统整体架构如下：
- **API Gateway**：接收客户端请求，路由到对应的Function
- **FaaS函数层**：使用云函数（如AWS Lambda/阿里云FC）处理业务逻辑，包括用户认证函数、课程管理函数、作业批改函数（调用AI模型）、学习报告生成函数等
- **事件总线**：使用EventBridge作为统一事件总线，所有业务事件通过事件总线进行分发
- **事件存储**：使用Event Store记录所有领域事件，支持事件溯源
- **数据层**：热数据使用DynamoDB/TableStore，冷数据使用S3/OSS，搜索使用Elasticsearch

在系统上线后，团队发现以下问题：
1. **冷启动延迟**：在非上课时段，函数实例被回收。当上课时段突然来临时，大量函数需要冷启动，首次请求延迟达到5-8秒，严重影响直播互动体验。
2. **事件顺序性**：课堂互动中的答题事件需要严格按照时间顺序处理（先提交先处理），但事件总线的并发消费导致事件乱序。
3. **事件溯源查询效率**：随着事件数据累积到亿级，通过事件重放来重建聚合状态的性能急剧下降。

### 【问题1】（8分）

请简述Serverless架构的核心特征和优势，并说明FaaS（Function as a Service）的冷启动产生的原因及其对系统性能的影响。

### 【问题2】（9分）

针对上述三个问题（冷启动延迟、事件顺序性、事件溯源查询效率），请分别给出具体的技术解决方案，并说明每种方案的实现原理。

### 【问题3】（8分）

请说明事件溯源（Event Sourcing）模式的核心思想、与传统CRUD模式的区别，以及在该在线教育平台中事件溯源的具体应用场景。

## 试题四（选做）：数字孪生系统架构 — 智慧工厂设备管理

### 【说明】

某大型制造企业拟构建一个**智慧工厂数字孪生平台**，对工厂内的2000+台关键生产设备（包括数控机床、工业机器人、传送带、环境传感器等）进行数字化建模，在虚拟空间中构建物理工厂的精确映射。平台的核心目标是实现：
- **实时监控**：通过数字孪生体实时展示每台设备的运行状态、工艺参数和环境数据
- **预测性维护**：基于设备历史数据和AI模型，预测设备故障并提前安排维修计划
- **仿真优化**：在虚拟环境中模拟调整生产参数，优化生产工艺后再应用到物理设备

系统的数据采集层通过工业物联网网关连接各类设备，采集频率从1Hz（环境传感器）到1000Hz（振动传感器）不等，每秒产生的原始数据量约500MB。数据经过边缘节点预处理后上传至平台。

在架构设计评审中，团队面临以下关键技术挑战：

1. **物理-虚拟映射**：如何建立物理设备与数字孪生模型之间的精确映射关系？孪生模型应该包含哪些维度的信息？如何保持模型的实时更新？

2. **实时同步延迟**：从物理设备产生数据到数字孪生体完成状态更新，端到端延迟要求不超过500ms。但在高频数据采集（1000Hz）场景下，传统的"采集→传输→处理→渲染"链路延迟难以满足要求。

3. **海量时序数据存储与查询**：系统每天产生约40TB的时序数据，需要支持长达5年的历史数据查询（用于训练预测性维护AI模型），同时要保证最近7天数据的毫秒级查询响应。

### 【问题1】（8分）

请说明数字孪生系统的核心概念和分层架构，并描述物理实体层、数字孪生层和应用服务层各自的职责与关键技术。

### 【问题2】（9分）

针对上述三个技术挑战（物理-虚拟映射、实时同步延迟、海量时序数据），请分别给出具体的架构设计方案和技术选型建议。

### 【问题3】（8分）

在该平台中，预测性维护模块需要使用历史设备数据训练AI故障预测模型。请说明预测性维护系统的数据流转架构（从数据采集到预测告警的完整链路），并给出关键技术选型。

## 试题五（选做）：软件建模与设计模式 — 智能文档审批工作流系统

### 【说明】

某政务服务中心拟开发一个**智能文档审批工作流系统**，用于处理企业开办、项目审批、资质认证等各类政务审批事项。系统需要支持灵活的审批流程配置，不同的审批事项有不同的审批环节和流转规则。

系统核心功能包括：
- **流程定义**：管理员可以通过可视化界面定义审批流程模板（包括审批节点、流转条件、回退规则等）
- **任务分派**：根据审批规则将任务自动分派给对应的审批人或审批组
- **状态流转**：审批文档在不同状态（草稿→待审核→审核中→已通过/已驳回→已归档）之间流转，每个状态下允许的操作不同
- **多级审批**：支持串行审批（逐级）、并行审批（会签）、条件分支审批
- **智能辅助**：集成OCR识别和AI审核辅助功能，自动提取关键信息并进行合规性预检

在设计评审中，架构师发现当前的审批流转逻辑使用大量的if-else嵌套来判断当前状态和可执行操作，代码复杂度高、可维护性差。例如：

```java
if (status == "DRAFT") {
    if (action == "submit") { /* 提交逻辑 */ }
    else if (action == "delete") { /* 删除逻辑 */ }
    else { throw new Exception("当前状态不允许该操作"); }
} else if (status == "PENDING_REVIEW") {
    if (action == "approve") { /* 通过逻辑 */ }
    else if (action == "reject") { /* 驳回逻辑 */ }
    else if (action == "withdraw") { /* 撤回逻辑 */ }
    // ...更多状态和操作的组合
}
```

同时，审批过程中需要执行多种检查（格式检查→权限验证→业务规则校验→合规性检查→AI辅助审核），每种检查的逻辑独立，且不同审批事项需要的检查组合不同。

### 【问题1】（10分）

针对审批状态流转的if-else嵌套问题，请说明**状态模式（State Pattern）**的设计思想、核心角色和类结构，并给出该审批系统应用状态模式的UML类图描述（文字描述即可）。

### 【问题2】（8分）

针对审批过程中多种检查的组合执行问题，请说明**责任链模式（Chain of Responsibility Pattern）**的设计思想，并描述在该审批系统中如何应用责任链模式来实现灵活的检查链配置。

### 【问题3】（7分）

除状态模式和责任链模式外，请再列举该系统中可以应用的2种设计模式，分别说明每种模式的应用场景及其带来的设计优势。
