#### 1.2 高优先级质量属性场景

##### 场景1：需求(a) - 10秒内完成调度规划（性能）

**六要素描述：**

| 要素 | 内容 |
|------|------|
| **刺激源 (Source)** | 外部用户/订单系统 |
| **刺激 (Stimulus)** | 用户提交订单，系统接收订单请求 |
| **制品 (Artifact)** | 调度规划引擎、路由算法模块、数据库查询 |
| **环境 (Environment)** | 正常业务运行时段（日均2000万订单） |
| **响应 (Response)** | 系统完成以下操作：① 查询仓库库存；② 计算最优配送路线；③ 计算预计送达时间；④ 返回规划结果给用户 |
| **响应度量 (Response Measure)** | 端到端响应时间 ≤ 10秒，成功率 ≥ 99.5%；平均响应时间 < 5秒 |

**核心技术方案：**
- 采用缓存策略（Redis）预加载仓库库存、实时库存快照
- 使用预计算的配送区域网格划分，加快路由搜索
- 调度算法采用启发式算法（如地理位置聚类+贪心或遗传算法），避免精确求解的高时间复杂度
- 建立多级查询缓存，热点数据放内存
- 异步化非关键路径（如风险评分、反欺诈可异步处理）

##### 场景2：需求(c) - 调度引擎故障自动切换（可用性）

**六要素描述：**

| 要素 | 内容 |
|------|------|
| **刺激源 (Source)** | 故障检测系统/监控告警 |
| **刺激 (Stimulus)** | 主调度引擎发生故障（如进程崩溃、数据库连接丢失、CPU异常飙升） |
| **制品 (Artifact)** | 主调度引擎、备用调度引擎、故障转移机制、分布式任务队列、共享存储 |
| **环境 (Environment)** | 调度系统正在处理实时订单；已提交但未执行的调度任务存在于任务队列中 |
| **响应 (Response)** | ① 故障检测系统在10秒内发现故障；② 自动触发故障转移；③ 备用引擎接管；④ 从共享队列重新加载已提交的调度任务；⑤ 恢复调度处理 |
| **响应度量 (Response Measure)** | 切换时间 ≤ 10秒；任务丢失率 = 0；RTO ≤ 10秒；RPO = 0 |

**核心技术方案：**
- **主备架构**：主调度引擎+备用调度引擎（Hot-Standby或Warm-Standby）
- **故障检测**：心跳检测（Heartbeat）+ 健康检查接口，检测周期 2-3 秒
- **分布式任务队列**：使用 Kafka、RabbitMQ 或 Redis Queue 存储待调度任务
  - 任务写入队列时采用两阶段提交（或本地+远程双写）确保可靠性
  - 备用引擎监听同一队列，可立即接管
- **共享状态存储**：Redis/ZooKeeper 存储当前调度状态、进行中的任务ID
- **自动转移机制**：通过 VIP 漂移或 DNS 重定向实现客户端的自动切换
- **任务重放**：故障转移后，扫描任务队列中的未处理任务，重新调度

##### 场景3：需求(f) - 基于角色的访问控制（安全性）

**六要素描述：**

| 要素 | 内容 |
|------|------|
| **刺激源 (Source)** | 系统用户（仓库管理员、区域经理、配送员等） |
| **刺激 (Stimulus)** | 用户发起数据查询请求（查看仓库数据、配送站点信息、订单状态等） |
| **制品 (Artifact)** | 权限管理系统、认证服务、授权决策引擎、审计日志系统 |
| **环境 (Environment)** | 系统正常运营；用户已完成身份认证；请求包含用户身份和请求资源信息 |
| **响应 (Response)** | ① 系统识别用户身份；② 查询用户角色和权限列表；③ 对请求数据进行行级过滤（Row-Level Security）；④ 返回用户有权限访问的数据；⑤ 记录访问审计日志 |
| **响应度量 (Response Measure)** | 权限判定延迟 < 100ms；错误授权率 = 0；所有访问操作均被审计 |

**核心技术方案：**
- **RBAC 模型**：定义角色（仓库管理员、区域经理、司机等）→ 权限映射表
  ```
  用户 →(belongs to)→ 角色 →(has)→ 权限 →(on)→ 资源
  ```
- **权限矩阵设计**：
  ```
  角色             查询权限范围        操作权限
  ─────────────────────────────────────────
  仓库管理员        所辖仓库数据        库存调整、出入库
  区域经理         所辖区域聚合数据     分配订单、查看报表
  配送员           个人订单、配送路线   完成签收、上报异常
  运营总监         全局数据            所有查询、策略制定
  ```
- **行级安全（RLS）实现**：
  - 数据库层：使用 PostgreSQL RLS、Oracle VPD（Virtual Private Database）等
  - 应用层：在 ORM 查询时自动添加 WHERE 子句进行过滤
    ```sql
    SELECT * FROM orders WHERE warehouse_id IN (SELECT warehouse_id FROM user_warehouses WHERE user_id = ?)
    ```
- **权限缓存**：使用 Redis 缓存用户角色和权限（TTL 5-10分钟），减少频繁查询
- **审计日志**：
  - 记录所有数据访问行为（用户、操作、资源、时间、结果）
  - 存储至审计日志表或专门的审计系统
  - 支持事后追溯和合规审计

### 问题2：架构风格对比与选择（11分）

#### 2.1 两种架构风格的核心特点与适用场景（4分）

##### 方案A：事件驱动架构（EDA - Event Driven Architecture）

**核心特点：**
- 各模块通过发布-订阅模式解耦
- 消息中间件（Kafka、RabbitMQ）作为事件总线
- 异步非阻塞通信
- 模块间依赖关系隐式（通过事件名称）
- 扩展新功能时，只需新增事件订阅者

**事件驱动流程示例：**
```
订单系统 → 发布"订单已创建"事件 → 事件总线(Kafka)
                                    ├→ 调度系统订阅 → 触发调度
                                    ├→ 支付系统订阅 → 处理支付
                                    ├→ 库存系统订阅 → 扣减库存
                                    └→ 通知系统订阅 → 发送短信/推送
```

**优势：**
- 高度解耦，新增功能无需修改核心业务
- 天然异步，不阻塞主流程
- 高并发处理能力强
- 易于实现事件溯源（Event Sourcing）

**劣势：**
- 调试困难（异步链路追踪复杂）
- 事件一致性保证困难
- 消息中间件引入额外复杂度

**适用场景：**
- 高并发实时处理（数据流、物流、电商平台）
- 跨系统异步协作
- 对响应时间无严格要求的业务

##### 方案B：微服务+同步API网关

**核心特点：**
- 各微服务通过 RESTful API 同步调用
- API 网关统一入口，负责路由和聚合
- 请求-响应模式
- 强一致性保证
- 调用链路显式且易于理解

**微服务架构流程示例：**
```
客户端 → API 网关 → 调度服务 ──┬→ 仓库服务 (REST)
                               ├→ 路由服务 (REST)
                               └→ 库存服务 (REST)
                                  （同步阻塞调用）
```

**优势：**
- 链路明确，易于调试
- 强一致性，事务控制简单
- 每个服务独立可部署

**劣势：**
- 级联故障风险（A调用B，B调用C，任何一个故障都会影响整体）
- 响应延迟较长（多个同步调用堆积）
- 高并发下容易产生长链路阻塞

**适用场景：**
- 对一致性要求高的业务
- 业务流程简单明确的系统
- 查询操作较多、写操作较少的系统

#### 2.2 两种方案在需求(b)和(k)上的优劣分析（4分）

##### 需求(b)：实时处理5000辆车GPS数据（每5秒更新一次）

| 对比维度 | 方案A：事件驱动 | 方案B：微服务API |
|--------|---------------|-----------------|
| **处理吞吐量** | 高（★★★★★）- 异步非阻塞，消息队列天然支持高吞吐 | 中（★★★） - 同步调用易形成阻塞 |
| **延迟** | 低（★★★★★） - 事件异步处理，不阻塞发送方 | 中等（★★★） - 同步等待响应 |
| **资源利用率** | 高（★★★★★） - 线程池异步处理，不占用大量连接 | 中（★★★） - 同步调用占用连接资源 |
| **技术适配** | 极优 - Kafka天生设计用于处理这类场景 | 一般 - API网关需要配置连接池等 |
| **可靠性** | 高（★★★★） - 消息可持久化，支持重试 | 中等（★★★） - 需要熔断、重试等机制 |
| **场景示例** | IoT数据流处理的标准方案 | 不推荐用于此场景 |

**分析：** 
- 5000辆车×每5秒1条数据 = 1000个GPS事件/秒
- 方案A用Kafka接收GPS事件，异步分发给多个消费者（地理位置索引、热力图、实时追踪），自然满足要求
- 方案B则需要GPS系统同时调用地理位置、热力图、追踪多个微服务，链路过长，延迟累积

##### 需求(k)：弹性扩容至3倍处理能力

| 对比维度 | 方案A：事件驱动 | 方案B：微服务API |
|--------|---------------|-----------------|
| **扩容粒度** | 细粒度（★★★★★） - 独立扩容消费者 | 粗粒度（★★★）- 需要扩容多个微服务实例 |
| **扩容难度** | 易（★★★★★） - 新增消费者实例自动加入消费组 | 中等（★★★★） - 需要修改API网关配置 |
| **资源利用率** | 高 - 按需增减消费者，消息队列自动负载均衡 | 中等 - 可能出现负载不均 |
| **成本** | 低 - 只需扩容消费层 | 中等 - 需要扩容整个调用链路 |
| **秒级响应** | 可能（基于Kubernetes HPA） | 需要更多时间（更复杂的依赖关系） |

**分析：**
- 方案A：增加消费者实例→消息队列自动分配分片→立即提高吞吐量
- 方案B：增加一个微服务实例，需要同时考虑上游路由调整和下游依赖关系

#### 2.3 推荐方案与核心架构设计（3分）

**推荐方案：事件驱动架构（EDA）主导 + 微服务网关混合**

**理由：**
1. 需求(b)和(k)都明显倾向于EDA
2. 物流系统本质是"流处理"系统，适合EDA
3. GPS更新、订单创建、派单等都是事件源

**核心架构设计：**

```
┌─────────────────────────────────────────────────────────────┐
│                        系统边界                               │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │ 订单系统     │  │ GPS终端      │  │ 调度APP      │       │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘       │
│         │                  │                  │               │
│         └──────────────────┼──────────────────┘               │
│                            │                                  │
│                 ┌──────────▼──────────┐                      │
│                 │   API 网关          │                      │
│                 │ (请求验证、限流)    │                      │
│                 └──────────┬──────────┘                      │
│                            │                                  │
│         ┌──────────────────▼──────────────────┐              │
│         │   事件总线 (Kafka / Redis Streams)   │              │
│         │                                     │              │
│         │  Topics:                            │              │
│         │  - order.created                    │              │
│         │  - gps.updated                      │              │
│         │  - route.planned                    │              │
│         │  - dispatch.assigned                │              │
│         └─────┬─┬─┬──────────────────────────┘              │
│               │ │ │                                          │
│      ┌────────┘ │ │                                          │
│      │          │ │                                          │
│  ┌───▼────┐ ┌──▼─┴────┐ ┌──────────────┐ ┌─────────────┐   │
│  │调度    │ │库存     │ │地理位置索引  │ │通知系统     │   │
│  │消费者  │ │消费者   │ │消费者        │ │消费者       │   │
│  └────────┘ └─────────┘ └──────────────┘ └─────────────┘   │
│                                                               │
│  ┌──────────────────────┐  ┌───────────────────┐            │
│  │ Redis缓存            │  │ TimescaleDB       │            │
│  │ (实时库存/位置)      │  │ (时序数据存储)    │            │
│  └──────────────────────┘  └───────────────────┘            │
│                                                               │
└─────────────────────────────────────────────────────────────┘

核心数据流：
订单创建 → order.created事件 → Kafka topic
          ├→ 调度消费者(处理路线规划，异步，10s内返回)
          ├→ 库存消费者(扣减库存)
          └→ 通知消费者(推送客户)

GPS更新 → gps.updated事件 → Kafka topic
        ├→ 实时位置索引(更新geohash)
        ├→ 热力图消费者
        └→ 异常预警消费者
```

**架构优势：**
1. **性能**：异步处理，10秒规划目标通过预缓存+启发式算法达成
2. **扩容**：独立扩容消费者实例，3倍扩容轻松实现
3. **可靠性**：事件持久化（Kafka retention），故障时可重放
4. **解耦**：新增功能（如异常预警）只需新增消费者
5. **实时性**：GPS数据流处理延迟 < 1秒

**关键技术选项：**
- **消息队列**：Kafka（高吞吐、持久化、分布式消费）
- **实时计算**：Flink / Spark Streaming 做复杂事件处理
- **缓存**：Redis（库存快照、热点数据）
- **容错**：主备调度引擎 + 事件重放机制

## 试题二 - 保险核心系统现代化改造

### 问题1：数据库国产化迁移方案（10分）

#### 1.1 从Oracle到国产数据库迁移的关键因素（5分）

**五大关键因素：**

##### 1. 应用兼容性评估

**内容：**
- SQL方言差异：Oracle特有语法（如 DECODE、ROWNUM、分析窗口函数等）与国产DB的兼容程度
  ```sql
  -- Oracle特有语法
  SELECT DECODE(status, 1, '生效', 0, '失效') FROM policy;
  SELECT * FROM orders WHERE ROWNUM <= 10;
  
  -- 国产DB需要改写为标准SQL或使用兼容模式
  SELECT CASE WHEN status=1 THEN '生效' ELSE '失效' END FROM policy;
  SELECT * FROM orders LIMIT 10;
  ```
- 存储过程、触发器、函数的迁移复杂度
- 数据类型映射（如BLOB、CLOB、NUMBER精度）
- 序列(Sequence)、物化视图等特殊对象的处理

**评估工具：** Oracle到国产DB兼容性扫描工具（如 TiDB Migration Assistant）

##### 2. 性能基准测试

**关键指标：**
- 单表查询性能（特别是大表 > 1亿行）
- 复杂JOIN性能（200+应用模块可能有复杂查询）
- 事务处理能力（TPS - Transactions Per Second）
- 并发连接数（5000+的连接数需求）
- 批量INSERT/UPDATE性能（报表生成、数据导入）

**测试场景：**
```
场景1：单表查询 - SELECT * FROM big_table WHERE id=?
场景2：多表JOIN - SELECT ... FROM orders o JOIN customers c JOIN products p WHERE ...
场景3：事务测试 - BEGIN; INSERT; UPDATE; COMMIT;
场景4：并发测试 - 100/500/2000并发连接，持续1小时
场景5：批量操作 - INSERT 10M rows in 5 minutes
```

**目标：** 国产DB性能应达到Oracle的 ≥ 80%，关键业务 ≥ 95%

##### 3. 数据一致性和事务ACID保证

**关键评估点：**
- 事务隔离级别：国产DB是否支持 Read Committed / Serializable 等隔离级别
- 分布式事务：如果国产DB采用分布式架构（如TiDB分片）是否支持全局一致性
- 锁机制：行锁、表锁、死锁检测和恢复
- 崩溃恢复：故障重启后数据一致性保证

**特别关注：**
- 保险系统涉及财务数据，ACID要求极高
- 需要验证：多个UPDATE操作在并发下的隔离性
- 测试场景：在高并发下进行转账操作，验证金额平衡

```sql
-- 关键测试：转账事务隔离性
BEGIN;
  UPDATE accounts SET balance = balance - 1000 WHERE account_id = '001';
  UPDATE accounts SET balance = balance + 1000 WHERE account_id = '002';
COMMIT;

-- 验证：即使崩溃，也不会出现两个账户都扣款或都入账的情况
```

##### 4. 运维工具和生态成熟度

**需要评估：**
- **备份恢复工具**：是否有完整的备份方案、PITR（Point-In-Time Recovery）
- **监控告警**：国产DB的监控系统、告警机制、日志分析
- **性能优化工具**：执行计划分析、索引建议、SQL优化
- **高可用方案**：主从复制、多副本、自动failover
- **技术支持**：供应商支持响应时间、社区活跃度

**选型参考：**
- OceanBase（阿里）：完整的备份恢复、高可用方案成熟
- TiDB（PingCAP）：生态工具丰富（TiDB Dashboard、TiDB Lightning等）
- 达梦（武汉）：国产替代，支持有保障

##### 5. 成本与ROI分析

**成本构成：**

```
    - 国产DB → 同步回Oracle（double-write）
    - 应用层对比Oracle和国产DB的查询结果

迁移中（灰度21-100%）：
    应用系统 → 国产DB（主）
                ├→ [READ/WRITE] → 国产DB

```

**详细流程：**

| 阶段 | 比例 | 时间 | 操作 | 风险 | 回滚方案 |
|------|------|------|------|------|--------|
| 灰度1 | 10% | 1周 | 选择非关键业务模块（如报表系统）先做迁移 | 低 | 路由切回Oracle |
| 灰度2 | 20% | 1周 | 扩大到核心非金融模块 | 低 | 路由切回Oracle |
| 灰度3 | 50% | 2周 | 50%金融交易流量 | 中 | 立即回滚所有写操作 |
| 灰度4 | 80% | 2周 | 80%金融交易流量 | 中 | 立即回滚所有写操作 |
| 灰度5 | 100% | 2周 | 完全迁移 | 低 | Oracle变为只读备份 |

##### 阶段3：数据一致性保证机制

**双写策略实现：**

```java
// 应用层面实现双写
public void updatePolicyData(Policy policy) {
    try {
        // 写Oracle
        oracleDAO.update(policy);
        
        // 同步写国产DB
        tidbDAO.update(policy);
        
        // 写入同步日志（用于对账）
        syncLog.record("UPDATE", policy.getId(), "SUCCESS");
    } catch (Exception e) {
        // 处理国产DB写入失败
        if (isGrayedTraffic()) {
            // 如果是灰度流量，进行补偿
            asyncRetry.schedule(() -> tidbDAO.update(policy));
        } else {
            // 非灰度流量，只写Oracle，国产DB异步补齐
            asyncSync.schedule(() -> tidbDAO.update(policy));
        }
    }
}

// 读取时的验证
public Policy queryPolicy(String policyId) {
    Policy oracleData = oracleDAO.query(policyId);
    
    if (isGrayedTraffic()) {
        Policy tidbData = tidbDAO.query(policyId);
        
        // 验证数据一致性
        if (!dataConsistent(oracleData, tidbData)) {
            log.warn("Data inconsistency detected: " + policyId);
            // 返回Oracle数据（保证正确性）
            // 触发修复任务
            fixInconsistency(policyId);
        }
    }
    
    return oracleData;
}
```

**数据对账机制：**

```
每天凌晨 2:00 启动对账任务：
1. 提取Oracle和国产DB的对账数据（按业务主键）
2. 逐条比对：SELECT COUNT(*) WHERE ... 不一致
3. 生成对账差异报告
4. 自动修复（如果可确定哪一方正确）
5. 人工审核（复杂情况）
6. 入库修复结果
7. 再次验证一致性

对账SQL示例：
SELECT 
    oracle.id,
    oracle.balance,
    tidb.balance,
    CASE WHEN oracle.balance != tidb.balance THEN '差异' ELSE '一致' END
FROM oracle_accounts oracle
FULL OUTER JOIN tidb_accounts tidb ON oracle.id = tidb.id
WHERE oracle.balance != tidb.balance OR tidb.id IS NULL;
```

##### 阶段4：验证与回退计划

**关键验证点：**
1. 数据总数对比（≠0则有问题）
2. 关键字段求和对比（如总交易额）
3. 关键业务指标对比（如保费总额、赔付总额）
4. 事务处理能力对比（TPS、响应延迟）

**回退条件：**
- 任何数据一致性问题 → 立即回退
- 性能下降超过 20% → 回退并优化
- 故障发生 → 立即回退

**回退步骤：**
```
1. 路由器立即切回Oracle
2. 停止国产DB的写操作
3. 检查灰度期间的写入，补齐Oracle缺失数据
4. 启动对账流程，确保最终一致性
5. 根因分析，解决问题后再次尝试迁移
```

### 问题2：绞杀者模式（Strangler Fig Pattern）微服务迁移（8分）

#### 2.1 如何识别和选择第一批拆分的服务？（3分）

**识别标准（RICE排序）：**

| 标准 | 说明 | 评分 |
|------|------|------|
| **Reach（影响范围）** | 多少用户受益 | 高影响=高分 |
| **Impact（业务价值）** | 对业务增长的贡献 | 直接收入贡献=高分 |
| **Confidence（实现确定性）** | 技术风险有多大 | 风险低=高分 |
| **Effort（工程量）** | 需要多少人天 | 工程量少=高分 |

**RICE评分公式：**
```
RICE Score = (Reach × Impact × Confidence) / Effort

示例：
服务A：(100×3×90%) / 10 = 27分（优先级高）
服务B：(50×5×50%) / 20 = 6.25分（优先级低）
```

**第一批拆分候选：**

1. **推荐**：登录认证服务
   - Reach: 100（所有用户都要登录）
   - Impact: 3（不直接产生收入，但基础）
   - Confidence: 90%（功能相对独立）
   - Effort: 5人天
   - RICE = 54分 ✓ 优先

2. **推荐**：支付结算服务
   - Reach: 80（交易用户）
   - Impact: 5（直接关系收入）
   - Confidence: 80%
   - Effort: 10人天
   - RICE = 32分 ✓ 优先

3. **推荐**：规则引擎服务
   - Reach: 70（保费计算、核保等）
   - Impact: 4（影响核心业务）
   - Confidence: 70%（需要单独部署测试）
   - Effort: 8人天
   - RICE = 24.5分 ✓ 优先

4. **不推荐**：报表生成服务
   - Reach: 20（仅内部使用）
   - Impact: 2（无直接收入）
   - Confidence: 80%
   - Effort: 15人天
   - RICE = 2.13分 ✗ 后期

**分析维度：**

```
复杂度 vs 收益 矩阵：

高收益
  ↑     │ 登录服务  支付服务      报表服务
  │     │ (优先)  (优先)         (后期)
  │  低 │
  │复杂 │
  │度   │
  └─────→ 高复杂度

优先顺序：
1. 登录认证（低耦合、易实现、基础性强）
2. 支付结算（核心业务、收益高）
3. 规则引擎（降低核心系统复杂度、高复用）
```

### 问题1：数据湖仓一体架构（10分）

#### 1.1 Lakehouse核心概念与优势（5分）

**核心概念：**

Lakehouse = Data Lake（数据湖）+ Data Warehouse（数据仓库）

```
传统架构：
源系统 → 数据湖(低成本存储) → ETL → 数据仓库(高性能查询) → BI/分析
          │ 非结构化/嵌套    │           │ 结构化          │
          └─ 灵活无约束 ─────┘           └─ 严格模式 ──────┘

Lakehouse架构：
源系统 → 统一存储(数据湖仓) ←→ 元数据管理 ←→ 查询引擎
              ├─ 低成本(对象存储)
              ├─ 灵活(支持多种格式)
              ├─ 结构化(ACID事务)
              └─ 高性能(索引/分区)
              
结果：
- 读写一体：不需要ETL抽取，直接查询
- 成本优化：消除数据冗余
- 实时性：数据一写入就可查询
```

**Data Lake vs Data Warehouse vs Lakehouse对比：**

| 特性 | 数据湖 | 数据仓库 | Lakehouse |
|------|-------|---------|----------|
| 存储成本 | 低★★★★★ | 高★★ | 低★★★★★ |
| 数据质量 | 低 | 高 | 高 |
| 数据格式 | 灵活 | 结构化 | 灵活+结构化 |
| 查询性能 | 慢 | 快 | 快 |
| ACID事务 | 无 | 有 | 有 |
| ETL复杂度 | 低 | 高 | 低 |
| 更新延迟 | 高 | 中 | 低 |
| 适用场景 | 原始数据存储 | 历史分析 | **实时+分析** |

**Lakehouse核心优势（针对风控平台）：**

1. **实时-离线一体**
   ```
   实时特征计算：
   DataFrame df = spark.readStream()
     .format("kafka")
     .load()
   result = mlModel.predict(df)  // 100ms内返回
   
   离线回填：
   DataFrame hist = spark.read()
     .format("delta")  // 使用Delta Lake格式
     .load("/feature_store")
   result = mlModel.fit(hist)  // 训练模型
   
   同一份代码，既支持实时，又支持离线
   ```

2. **数据一致性**
   - 原生支持ACID事务
   - 同一特征值的实时-离线计算结果一致（避免训练-服务偏差）
   
3. **成本优化**
   - 数据只存一份（对象存储如S3/OSS）
   - 无需维护多套系统（Data Lake+Data Warehouse）
   - 成本可降低 60-80%

4. **血缘追踪**
   - 元数据管理完善
   - 支持数据来源追踪、计算逻辑追踪

### 问题2：实时-离线特征一致性（8分）

#### 2.1 避免训练-服务偏差（5分）

**问题描述：**

```
常见偏差场景：

1. 离线：特征计算在Spark中
   user_visits_count = 用户过去7天的访问次数
   计算逻辑：SELECT COUNT(*) FROM events WHERE user_id=? AND date>=DATE_SUB(TODAY(), 7)
   
2. 实时：特征在应用中计算（Python）
   user_visits_count = redis.get('user:%s:visits'%user_id)
   
问题：
- 离线可能包含昨天、今天的数据
- 实时Redis中的计数可能是半小时前更新的
- 结果不一致！
```

**一致性保证方案：**

##### 方案A：统一特征定义（推荐）

```python
# shared_feature_definitions.py
# 所有特征定义集中在一个共享库中
# Python、Scala、Java等都可以导入这个库

class FeatureDefinition:
    """统一特征定义"""
    
    def __init__(self, name, depends_on, window_size):
        self.name = name
        self.depends_on = depends_on  # 依赖的原始数据
        self.window_size = window_size
    
    def compute_offline(self, spark_df):
        """离线计算逻辑"""
        pass
    
    def compute_realtime(self, event):
        """实时计算逻辑"""
        pass
    
    def validate_consistency(self, offline_result, realtime_result):
        """验证一致性"""
        assert offline_result == realtime_result, \
            f"Inconsistent: offline={offline_result}, realtime={realtime_result}"

# 特征定义示例
user_visits_7d = FeatureDefinition(
    name="user_visits_7d",
    depends_on=["user_visit_event"],
    window_size="7 days"
)

# 离线计算（Spark）
@user_visits_7d.compute_offline
def calc_offline(spark_df, user_id, current_date):
    from datetime import timedelta
    window_start = current_date - timedelta(days=7)
    return spark_df.filter(
        (spark_df.user_id == user_id) &
        (spark_df.event_date >= window_start) &
        (spark_df.event_date < current_date)
    ).count()

# 实时计算（Flink）
@user_visits_7d.compute_realtime
def calc_realtime(feature_store, user_id, current_timestamp):
    # 从特征存储中读取最新的统计
    # 保证计算逻辑与离线一致
    window_start = current_timestamp - timedelta(days=7)
    return feature_store.get_feature_value(
        user_id=user_id,
        feature_name="user_visits_7d",
        timestamp=current_timestamp
    )
```

```
┌────────────────────────────────────────────────┐
│        Feature Store 架构                      │

   ├─ 位置：S3 / OSS（低成本）
   ├─ 更新频率：每日凌晨批量计算
   ├─ 用途：模型训练、离线评估
   └─ 示例表：
       user_id | feature_date | feature_name | feature_value | ...
       001     | 2026-05-10   | user_visits_7d | 42          | ...

3. 在线特征库 (Online Feature Store)
   ├─ 存储：Redis / DynamoDB（低延迟）
   ├─ Key结构：{user_id}:{feature_name}:{timestamp}
   ├─ 更新频率：实时更新（Flink消费流）

       L1: 本地内存（热数据）
       L2: Redis（全量）
```

```
    realtime_value,

   - 重新发布

3. 如果一致性良好（ < 1%）：
   - 将离线特征值同步到在线特征库
   - 实时查询继续使用在线库
   - 记录同步成功日志
```

### 问题3：数据血缘追踪系统（7分）

#### 3.1 核心概念和追踪粒度

**数据血缘定义：**
```
从数据源到最终结果的完整链路追踪
        ↓
源数据 → 清洗 → 转换 → 特征计算 → 模型预测 → 决策
 │       │      │      │        │       │
 └──────────────────────┬────────────────┘
           血缘关系
```

**追踪粒度（3个层级）：**

| 粒度 | 说明 | 用途 | 示例 |
|------|------|------|------|
| **系统级** | 数据在哪些系统间流转 | 高层架构理解 | 交易系统→风控系统→决策系统 |
| **表级** | 数据在哪些表间流转 | 问题排查、权限控制 | trade_events → user_statistics |
| **字段级** | 最细粒度，精确到列 | 影响分析、测试覆盖 | trade_events.amount → user_statistics.total_volume |

**示例：风控特征的血缘追踪**

```
特征：user_fraud_score
  ↑
  ├─ 来源1：用户访问频次
  │  ├─ 源：events表.event_type='login'
  │  ├─ 转换：时间窗口聚合（7天）
  │  └─ 规则：访问频次异常时↑风险
  │
  ├─ 来源2：交易金额
  │  ├─ 源：trade表.amount
  │  ├─ 转换：求和、归一化
  │  └─ 规则：超过用户历史平均3倍时↑风险
  │
  └─ 来源3：设备信息
     ├─ 源：第三方设备指纹库
     ├─ 转换：匹配黑名单、风险评分
     └─ 规则：命中黑名单时↑风险

血缘分析优势：
- 追踪：fraud_score为什么高？→ 找到所有贡献因素
- 影响：修改login事件定义会影响哪些特征？→ 自动列出
- 数据质量：upstream数据有问题，能快速定位
```

## 试题四 - AI模型推理服务平台

### 问题1：大模型推理优化技术（10分）

#### 1.1 五种常见优化技术（6分）

##### 1. 模型量化（Quantization）

**原理：** 将高精度参数（Float32）降低到低精度（Int8/Int4），减少显存占用和计算量

```
原始：70B模型 × 4字节/参数 = 280GB显存需求（单卡A100仅80GB）
INT8：70B模型 × 1字节/参数 = 70GB显存
INT4：70B模型 × 0.5字节/参数 = 35GB显存

性能对比：
┌──────────┬──────────┬────────┬─────────────┐
│精度      │显存占用  │推理速度│精度损失     │
├──────────┼──────────┼────────┼─────────────┤
│FP32      │280GB     │1.0x    │无(基准)     │
│INT8      │70GB      │2-3x    │0.5-1%       │
│INT4      │35GB      │4-5x    │2-3%         │
│FP16      │140GB     │2x      │0.1%         │
└──────────┴──────────┴────────┴─────────────┘

适用场景：
- INT8：对精度要求不高（如内容审核、分类任务）
- INT4：要求极致推理速度和显存节省（如移动端推理）
- FP16：精度和速度平衡（推荐）
```

##### 2. 层融合（Layer Fusion）

**原理：** 将多个计算步骤合并为一个操作，减少数据传输和中间结果

```
优化前：
input → Linear → ReLU → Linear → Attention → output
         ↓        ↓      ↓         ↓
      存储中间结果，需要4次内存访问

优化后：
input → [Linear+ReLU+Linear+Attention] → output
         （一个融合算子，只需1次内存访问）

性能提升：
- 内存访问减少 75%
- 推理速度提升 20-30%
```

##### 3. KV缓存优化（KV Cache Quantization）

**原理：** Transformer模型在生成长序列时，需要存储所有已生成token的KV值。优化策略：
- 只保留高价值的KV（剪枝）
- 量化KV缓存为低精度
- 使用环形缓冲减少显存拷贝

```
问题场景：生成长文本（如2000 tokens）
  Seq len: 1   → KV缓存: 144MB
  Seq len: 128 → KV缓存: 18GB
  Seq len: 2048 → KV缓存: 288GB (爆炸！)

解决：
- KV缓存量化为INT8 → 降低75%显存
- 删除低相关KV → 进一步降低50%
- 最终显存需求可控在合理范围
```

##### 4. 算子优化 & 自定义CUDA核

**原理：** 用高效的自定义CUDA核替代通用算子

```
原始：使用PyTorch/TensorFlow通用算子
  Attention = softmax(Q@K^T/√d) @ V
  需要调用4个通用算子，数据往复GPU显存

优化：写自定义CUDA核（如FlashAttention）
  一个核函数完成整个Attention计算
  - 减少显存访问 10倍
  - 速度提升 5-8倍
  
常见优化库：
- TensorRT（Nvidia）：自动生成优化CUDA核
- vLLM：LLM推理优化库
- DeepSpeed-Inference：微软大模型推理框架
```

##### 5. 批处理和请求融合（Batching & Request Fusion）

**原理：** 合并多个独立请求，充分利用GPU的向量化计算

```
单请求推理：
Request 1 → GPU处理 → 返回 → GPU利用率 20%

批处理推理：
Request 1  ─┐
Request 2  ─┼→ GPU批处理 → 返回 → GPU利用率 80%
Request 3  ─┤
Request 4  ─┘

吞吐量提升：
- 单请求：100 req/s
- 批大小8：600 req/s（提升6倍）

权衡：
- 增加等待延迟（等待凑齐一批请求）
- 但总体吞吐量大幅提升
- 对低延迟应用有影响，需要权衡

动态批处理策略：
- 收集到N个请求或等待M毫秒，就立即执行
- 例：等待8个请求或100ms超时，先到的执行
```

##### 6. 推测解码（Speculative Decoding）

**原理：** 使用较小的草稿模型快速生成候选token，大模型进行验证

```
传统解码：
Token 1: 大模型生成 → 1ms延迟
Token 2: 大模型生成 → 1ms延迟
...
Token 128: 大模型生成 → 1ms延迟
总延迟：128ms

推测解码：
Token 1-4: 草稿模型快速生成 → 0.1ms
验证: 大模型一次验证4个候选 → 1.2ms
通过的token直接输出，失败的重新生成

总延迟：30-40%的减少
```

**技术选型指南：**

```
按应用场景选择：

低延迟场景（如实时对话）：
  ├─ 层融合 ✓
  ├─ 算子优化 ✓
  └─ 量化 ✓ (INT8可接受)

高吞吐场景（如批量处理）：
  ├─ 批处理 ✓
  ├─ 量化 ✓ (INT4可用)
  └─ KV缓存优化 ✓

长文本生成：
  ├─ KV缓存优化 ✓
  ├─ 推测解码 ✓
  └─ 量化 ✓

显存受限：
  ├─ 量化 ✓ (INT4必须)
  ├─ KV缓存优化 ✓
  └─ 层融合 ✓
```

#### 1.2 模型量化详解 - INT8 vs INT4（4分）

| 对比维度 | INT8量化 | INT4量化 |
|--------|--------|--------|
| **精度** | 0.5-1%精度损失 | 2-3%精度损失 |
| **显存** | 原来1/4 | 原来1/8 |
| **推理速度** | 2-3倍提升 | 4-5倍提升 |
| **实现复杂度** | 中等 | 高（容易量化失败） |
| **适用模型** | 分类、翻译、摘要 | 边缘设备、移动端 |
| **缓存友好性** | 好 | 中等 |

**量化过程：**

```
Step 1: 数据范围校准 (Calibration)
  对100个典型样本进行前向推理
  记录每个权重和激活值的范围
  Example: weight范围 [-0.5, 0.8]

Step 2: 计算缩放因子 (Scale Factor)
  对于INT8 (范围 -128 ~ 127)：
  scale = (max_value - min_value) / 255
  
  Example: scale = (0.8 - (-0.5)) / 255 = 0.0051

Step 3: 量化
  quantized_value = ROUND((value - min_value) / scale)
  
  Example: 
    原始值 0.3 → 量化 ROUND((0.3-(-0.5))/0.0051) = 157
    量化后用INT8存储：157

Step 4: 反量化（推理时）
  original_value ≈ quantized_value * scale + min_value
  157 * 0.0051 + (-0.5) ≈ 0.3

量化损失来源：
- 四舍五入误差
- 权重范围超出INT8范围
- 激活值分布不均匀
```

**INT8 vs INT4精度对比：**

```
测试数据集：MNIST手写数字分类
                原始(FP32)  INT8      INT4
精度            99.2%      99.1%     97.5%
损失            -          -0.1%     -1.7%

结论：
- INT8可用于大多数任务，精度影响<1%
- INT4需要在低精度应用中使用
- 对于金融、医疗等高精度场景，INT8仍可能不够
```

**路由策略：**

```python
def route_request(request):
    # 根据请求特征决定路由
    
    if request.latency_requirement == 'LOW':  # < 100ms
        # 路由到小模型集群（7B，低延迟）
        return route_to('llm_7b_cluster')
    
    elif request.latency_requirement == 'MEDIUM':  # 100-500ms
        # 路由到中等模型（34B）
        return route_to('llm_34b_cluster')
    
    elif request.latency_requirement == 'HIGH':  # > 1s
        # 路由到大模型（70B），高吞吐
        # 使用批处理
        return route_to('llm_70b_cluster', batch=True)
    
    # 负载均衡：选择当前负载最低的GPU实例
    cluster = select_cluster()
    instance = cluster.select_instance_by_load()
    return instance.process(request)
```

**负载均衡算法：**

```
算法1：轮询 (Round Robin)
  GPU1 → GPU2 → GPU3 → GPU1 ...
  优点：简单
  缺点：不考虑GPU实际负载

算法2：最小连接数 (Least Connections)
  选择当前连接数最少的GPU
  优点：相对均衡
  缺点：不同请求复杂度差异大

算法3：加权最小负载 (Weighted Least Load)
  score = (current_load + request_tokens) / capacity
  选择score最低的GPU
  优点：考虑请求大小和GPU容量
  缺点：需要实时计算

推荐：算法3 + 预测
  预测即将到达的请求，提前调整负载分配
```

```
调度器架构：

│ 请求队列         │

┌───────▼──────────────────┐
│ 调度决策                 │
├──────────────────────────┤
│ 1. 优先执行金牌请求     │
│ 2. 剩余容量执行银牌请求 │

│                          │

└───────┬──────────────────┘
        │
┌───────▼──────────────────┐
│ 执行                     │
├──────────────────────────┤
│ GPU1: 执行金牌请求       │
│ GPU2: 执行银牌请求       │
│ GPU3: 执行铜牌请求       │
│ GPU4: 待机               │
└──────────────────────────┘
```

**优先级调度算法：**

```python
class RequestScheduler:
    def schedule(self, request_queue):
        """基于优先级的调度"""
        
        while gpu_available():
            # Step 1: 尝试调度高优先级请求
            for sla_tier in ['Gold', 'Silver', 'Bronze']:
                queue = request_queue[sla_tier]
                
                if queue.empty():
                    continue
                
                # 检查该等级的资源约束
                if self.has_capacity_for(sla_tier):
                    request = queue.popleft()
                    self.execute(request)
                    break
                else:
                    # 资源不足，尝试下一个等级
                    continue
    
    def has_capacity_for(self, sla_tier):
        """检查该SLA等级是否有容量"""
        reserved = self.reserved_capacity[sla_tier]
        used = self.used_capacity[sla_tier]
        available = reserved - used
        
        if sla_tier == 'Gold':
            # Gold优先级高，可以超用
            return available > 0 or self.total_free_capacity() > 0
        
        elif sla_tier == 'Silver':
            # Silver次优先级
            return available > 0
        
        else:  # Bronze
            # Bronze最低，仅当完全空闲时
            return self.total_used_capacity() < 10  # 容量利用率<10%
    
    def enforce_sla(self):
        """监控并强制SLA"""
        for request in executing_requests:
            if request.sla_tier == 'Gold':
                if request.latency > 100ms:
                    # 违反SLA，可能需要扩容
                    alert("Gold SLA violated: latency > 100ms")
            
            elif request.sla_tier == 'Silver':
                if request.latency > 500ms:
                    alert("Silver SLA violated")
            
            elif request.sla_tier == 'Bronze':
                if request.latency > 1s:
                    alert("Bronze SLA violated")
```

```kotlin
class SecureUpdateManager {
    fun verifyUpdate(updateFile: File): Boolean {
        // 1. 验证签名
        if (!verifySignature(updateFile, PUBLIC_KEY)) {
            log("Update signature verification failed!")
            return false
        }
        
        // 2. 检查Checksum
        val expectedHash = "abc123..."
        val actualHash = calculateHash(updateFile)
        if (expectedHash != actualHash) {
            log("Checksum mismatch!")
            return false
        }
        
        // 3. 检查SSL证书
        if (!isSSLCertificateValid(updateURL)) {
            log("SSL certificate invalid!")
            return false
        }
        
        // 4. 版本号检查（防止回滚攻击）
        if (updateVersion < currentVersion) {
            log("Downgrade attempt detected!")
            return false
        }
        
        return true
    }
    
    fun installUpdate(updateFile: File) {

            return
        }
```

#### 3.3 灰度发布和回退机制

```
灰度发布流程：

第0阶段：内部测试
  ├─ 研发测试：全量更新，验证功能
  ├─ 灰度前准备：生成签名、计算Hash
  └─ 如果通过 → 第1阶段

第1阶段：灰度 1%（千位用户）
  ├─ 下发更新给随机1%用户
  ├─ 监控指标（30分钟）：
  │  ├─ Crash率 < 0.5%
  │  ├─ 使用时长无异常
  │  └─ 关键业务指标无变化
  ├─ 如果正常 → 第2阶段
  └─ 如果异常 → 立即回退

第2阶段：灰度 5%
  ├─ 下发更新给5%用户
  ├─ 监控（1小时）
  ├─ 如果正常 → 第3阶段
  └─ 如果异常 → 立即回退

第3阶段：灰度 25%
  └─ 继续监控

第4阶段：灰度 100%
  ├─ 全量发布
  └─ 持续监控（24小时）

回退机制：
  ├─ 自动回退：
  │  ├─ Crash率 > 5% → 立即回退
  │  ├─ ANR率 > 2% → 立即回退
  │  └─ 关键业务异常 → 立即回退
  │
  └─ 手动回退：
     ├─ 运营人员发起
     ├─ 下发降级指令
     └─ 用户App下次启动时回退
```
