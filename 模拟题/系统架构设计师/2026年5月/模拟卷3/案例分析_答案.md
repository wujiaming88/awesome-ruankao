**（1）需求（a）：10秒内完成调度规划**

| 要素 | 内容 |
|------|------|

| **响应度量** | 授权判断准确率100%；越权访问拦截率100%；权限校验响应时间<50ms |

### 问题2 参考答案

**（1）两种架构风格的核心特点和适用场景**

**事件驱动架构（EDA）**：
- **核心特点**：组件之间通过事件（消息）进行异步通信，发布者不关心谁消费事件，消费者不关心谁发布事件；松耦合、高扩展性；支持事件溯源（Event Sourcing）
- **适用场景**：实时数据流处理、IoT数据采集、需要高吞吐和弹性伸缩的场景、复杂事件处理（CEP）

**微服务+同步API网关**：
- **核心特点**：服务之间通过RESTful/gRPC同步调用；请求-响应模式，调用链路清晰；API网关统一入口，负责路由、认证、限流
- **适用场景**：业务流程明确、需要即时响应的场景；对数据一致性要求高的事务性业务；调用链路较短的场景

**（2）满足需求（b）和需求（k）的优劣分析**

| 维度 | 事件驱动架构（EDA） | 微服务+同步API |
|------|-------------------|---------------|
| **需求（b）实时处理** | ✅ **优**：异步消息天然适合高频实时数据流（5000车/5秒=1000条/秒）；消息队列可缓冲峰值，防止下游过载 | ⚠️ **劣**：同步调用模式下，每条GPS数据都需要等待响应，延迟累积；高频调用对API网关压力大 |
| **需求（k）弹性扩容** | ✅ **优**：消费者可独立水平扩展，增加消费者实例即可提升处理能力；生产者和消费者解耦，扩容不影响上游 | ⚠️ **劣**：同步调用链路上的任一环节都需要同步扩容；服务间依赖关系导致扩容复杂度高 |

**（3）推荐方案与核心架构设计**

**推荐方案：事件驱动架构（EDA）为主 + 同步API为辅的混合架构**

理由：
1. 车辆GPS实时数据、订单事件等高频数据流适合EDA处理
2. 大促期间弹性扩容需求，EDA的消费者可独立扩展
3. 部分场景（如用户下单等待调度结果）需要同步响应，通过API网关+同步服务实现

**核心架构设计**：

```
┌─────────────────────────────────────────────────────┐
│                   API Gateway                        │
│    (同步请求入口：下单、查询、管理后台)                 │
└──────────────────────┬──────────────────────────────┘
                       │
       ┌───────────────┼───────────────┐
       │               │               │
┌──────▼─────┐  ┌─────▼──────┐  ┌─────▼──────┐
│ 订单服务    │  │ 调度服务    │  │ 用户服务    │ ← 同步微服务
└──────┬─────┘  └─────┬──────┘  └────────────┘
       │              │
       ▼              ▼
┌─────────────────────────────────────────────────────┐
│              Kafka 事件总线                          │
│  Topics: order-events, vehicle-events, dispatch-events │
└──────────────────────┬──────────────────────────────┘
       ┌───────────────┼───────────────┐
       │               │               │
┌──────▼─────┐  ┌─────▼──────┐  ┌─────▼──────┐
│ 路径规划    │  │ 实时位置    │  │ 异常处理    │ ← 事件消费者
│ 处理器     │  │ 处理器     │  │ 处理器     │   (可独立扩展)
└────────────┘  └────────────┘  └────────────┘
```

### 考点分析

- **质量属性场景六要素**：刺激源、刺激、制品、环境、响应、响应度量
- **架构风格对比**：事件驱动 vs 微服务+同步API
- **混合架构设计**：根据场景选择合适的架构风格组合
- **物流领域架构**：实时位置追踪、调度引擎、高可用设计

## 试题二

### 问题1 参考答案

**（1）数据库国产化迁移关键因素**

| 关键因素 | 具体考虑内容 |
|---------|-------------|
| **SQL兼容性** | Oracle特有语法（如CONNECT BY层次查询、MERGE INTO、分析函数）的兼容程度；存储过程PL/SQL向国产数据库迁移的改造工作量 |
| **数据类型映射** | Oracle特有数据类型（如NUMBER、VARCHAR2、CLOB、BLOB、DATE）与目标数据库的对应关系和精度差异 |
| **事务与隔离级别** | Oracle的读一致性（Read Consistency）机制在国产数据库中的等效实现；分布式事务的支持能力 |
| **性能基线对齐** | 迁移前后的性能对比测试；索引策略、执行计划优化；确保关键SQL性能不低于Oracle |
| **高可用与灾备** | 国产数据库的主从复制、集群方案是否满足RPO/RTO要求；与现有灾备体系的整合 |
| **运维工具链** | 备份恢复工具、监控告警工具、性能诊断工具的成熟度；运维团队的培训和能力建设 |
| **生态兼容** | 应用框架（MyBatis/Hibernate）、ETL工具、BI报表工具对国产数据库的支持程度 |

**（2）迁移步骤与业务连续性保障**

**迁移步骤**：

```
阶段1：评估与准备（2-3个月）
  ├── 数据库对象全量盘点（表、索引、视图、存储过程、触发器）
  ├── SQL语法兼容性分析，识别需改造的SQL
  ├── 搭建目标数据库测试环境
  └── 制定详细迁移计划和回退方案

阶段2：结构迁移与改造（2-4个月）
  ├── 表结构、索引、约束迁移（工具自动+人工调整）
  ├── 存储过程、函数改造（最耗时的工作）
  ├── 应用代码SQL适配改造
  └── 单元测试和功能测试

阶段3：数据迁移与同步（1-2个月）
  ├── 历史数据全量迁移（停机窗口或在线迁移工具）
  ├── 建立双向数据同步通道（Oracle → 国产DB）
  ├── 数据一致性校验（行数、校验和、抽样比对）
  └── 增量同步测试

阶段4：并行运行与切换（1-2个月）
  ├── 双写模式：应用同时写Oracle和国产DB
  ├── 读流量逐步切换：10% → 30% → 50% → 100%
  ├── 性能监控与问题修复
  └── 确认无误后停止Oracle写入，完成切换

阶段5：稳定运行与收尾
  ├── Oracle降级为只读备份（保留1-3个月）
  ├── 彻底下线Oracle
  └── 项目总结与知识沉淀
```

**业务连续性保障**：
1. **双写机制**：切换前应用同时写入新旧数据库，确保数据一致
2. **灰度切流**：读流量按比例逐步切换，发现问题可快速回退
3. **实时同步**：使用数据同步工具（如OGG、Canal）保持新旧库数据同步
4. **回退方案**：保留Oracle运行能力，任何时候都可以切回

### 问题2 参考答案

**绞杀者模式实施方案**

**（1）识别和选择第一批拆分的服务**

选择标准：
- **业务独立性高**：与其他模块耦合少，边界清晰
- **变更频率高**：经常有需求变更，拆分后能加速交付
- **技术风险可控**：复杂度适中，团队有信心交付
- **业务价值明确**：拆分后能直接带来业务收益

**推荐首批拆分的服务**：
1. **用户中心服务**：用户注册、认证、信息管理——边界清晰，复用度高
2. **产品目录服务**：保险产品信息展示——读多写少，风险低
3. **消息通知服务**：短信、邮件、推送——独立性强，可快速见效

**（2）过渡期新旧系统并行运行**

```
                    ┌──────────────┐
                    │   API网关     │
                    │ (路由分发)    │
                    └──────┬───────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
    ┌──────▼──────┐ ┌─────▼──────┐ ┌──────▼──────┐
    │ 新微服务     │ │ 防腐层(ACL) │ │ 旧单体应用   │
    │ (用户中心)   │ │            │ │ (核心业务)  │
    └─────────────┘ └─────┬──────┘ └─────────────┘
                          │
                    ┌─────▼──────┐
                    │ 旧单体应用  │
                    │ (被调用部分)│
                    └────────────┘
```

- **API网关**：统一入口，根据路由规则将请求分发到新微服务或旧系统
- **防腐层（Anti-Corruption Layer）**：新服务调用旧系统时，通过防腐层进行协议转换和数据映射，隔离新旧系统的模型差异
- **数据同步**：新服务有独立数据库时，通过数据同步保持新旧数据一致

**（3）逐步迁移流量**

| 阶段 | 流量分配 | 验证重点 |
|------|---------|---------|
| **金丝雀发布** | 1%流量到新服务 | 功能正确性、异常监控 |
| **小范围灰度** | 10%流量 | 性能指标、错误率 |
| **大范围灰度** | 50%流量 | 稳定性、容量验证 |
| **全量切换** | 100%流量 | 持续监控1-2周 |
| **旧服务下线** | 下线旧模块代码 | 确认无调用后清理 |

### 问题3 参考答案

**规则引擎架构设计**

**核心组件**：

```
┌─────────────────────────────────────────────────────┐
│                  规则管理平台（Web）                   │
│   规则编辑器 │ 版本管理 │ 测试沙箱 │ 发布审批         │
└──────────────────────┬──────────────────────────────┘
                       │ 发布
┌──────────────────────▼──────────────────────────────┐
│                  规则存储（数据库+缓存）               │
│   规则定义（DSL/决策表）│ 版本历史 │ 生效配置          │
└──────────────────────┬──────────────────────────────┘
                       │ 加载
┌──────────────────────▼──────────────────────────────┐
│                  规则执行引擎                         │
│   规则解析器 │ 条件评估器 │ 动作执行器 │ 结果收集器    │
└──────────────────────┬──────────────────────────────┘
                       │ 调用
┌──────────────────────▼──────────────────────────────┐
│                  业务应用层                           │
│   保费计算服务 │ 核保服务 │ 理赔服务                  │
└─────────────────────────────────────────────────────┘
```

**规则存储方式**：

| 存储形式 | 适用场景 | 示例 |
|---------|---------|------|
| **决策表** | 简单条件组合 | 按年龄、性别、保额确定保费系数 |
| **DSL规则脚本** | 复杂逻辑 | Drools DRL、Aviator表达式 |
| **决策树** | 层层判断逻辑 | 核保决策流程 |

**执行流程**：

1. **规则编辑**：业务人员通过可视化界面编辑规则（决策表或DSL）
2. **规则测试**：在沙箱环境用测试数据验证规则正确性
3. **规则发布**：经审批后，规则发布到生产环境（存入数据库+推送到缓存）
4. **规则加载**：执行引擎启动时加载规则；规则变更时通过消息通知实时刷新（热更新）
5. **规则执行**：业务请求到达时，执行引擎根据输入数据评估规则条件，执行匹配的动作

**热更新机制**：
- 规则发布后，通过Redis Pub/Sub或消息队列通知所有执行引擎实例
- 执行引擎收到通知后，异步拉取最新规则并替换内存中的旧规则
- 采用版本号+双缓冲机制，确保规则切换的原子性

- **数据库迁移**：国产化替代、数据同步、灰度切换
- **绞杀者模式**：遗留系统现代化改造的经典模式

- **保险行业架构**：核心系统架构、规则驱动设计

## 试题三

### 问题1 参考答案

**（1）数据湖仓一体（Lakehouse）架构**

**核心概念**：Lakehouse是结合数据湖和数据仓库优势的新型数据架构，在数据湖的低成本存储之上，增加数据仓库的事务支持、Schema管理和查询优化能力。

**与传统架构的对比**：

| 维度 | 数据仓库 | 数据湖 | Lakehouse |
|------|---------|--------|-----------|
| **存储格式** | 专有格式（列式） | 开放格式（Parquet/ORC） | 开放格式+元数据管理 |
| **Schema管理** | 严格Schema（写时Schema） | 无Schema或弱Schema | 支持Schema演进 |
| **事务支持** | 完整ACID | 无事务 | 支持ACID事务 |
| **数据类型** | 结构化为主 | 结构化+半结构化+非结构化 | 全类型支持 |
| **存储成本** | 高（专有存储） | 低（对象存储） | 低（对象存储） |
| **查询性能** | 高（专用优化） | 中（需要优化） | 高（索引+缓存+优化） |
| **实时能力** | 弱 | 支持流式写入 | 流批一体 |

**Lakehouse优势**：
1. **统一存储**：一份数据支持BI、ML、实时分析多种场景
2. **流批一体**：同一套存储和计算引擎处理实时和批量数据
3. **开放格式**：避免厂商锁定，数据可被多种引擎访问
4. **低成本**：基于对象存储，存储成本仅为传统数仓的1/10

### 问题2 参考答案

**（1）避免"训练-服务偏差"的方案**

**核心原则：特征计算逻辑单点定义，多端复用**

```
┌─────────────────────────────────────────────────────┐
│              特征定义仓库（Feature Definition）       │
│   用DSL/SQL定义特征计算逻辑（单一来源）               │
│   示例：user_7d_txn_count = COUNT(txn) WHERE        │
│         txn.user_id = ? AND txn.time > now() - 7d   │
└──────────────────────┬──────────────────────────────┘
                       │ 编译/生成
           ┌───────────┼───────────┐
           │           │           │
    ┌──────▼─────┐ ┌──▼──────┐ ┌──▼──────────┐
    │ Flink任务  │ │ Spark任务│ │ 在线计算逻辑 │
    │ (实时特征) │ │ (离线回填)│ │ (API调用)   │
    └────────────┘ └─────────┘ └─────────────┘
```

**具体措施**：
1. **统一特征定义语言**：用声明式DSL（如Feast的Feature Definition）定义特征逻辑，而非分别用Flink和Spark代码实现
2. **代码生成**：从统一定义自动生成Flink实时计算代码和Spark批量计算代码
3. **特征回放验证**：定期用历史数据回放，对比实时计算结果和离线计算结果，差异超过阈值告警
4. **特征版本管理**：特征定义变更时同步更新实时和离线计算任务

**（2）Feature Store设计**

```
┌─────────────────────────────────────────────────────┐
│                 Feature Store 架构                   │
├─────────────────────────────────────────────────────┤
│                 特征元数据层                         │
│   特征目录 │ 数据源血缘 │ 特征统计 │ 版本管理         │
├─────────────────────────────────────────────────────┤
│                 特征存储层                           │
│   ┌─────────────┐     ┌─────────────┐              │
│   │ 离线特征存储 │     │ 在线特征存储 │              │
│   │ (Delta Lake)│     │ (Redis)     │              │
│   │ 批量训练用   │     │ 实时推理用   │              │
│   └─────────────┘     └─────────────┘              │
├─────────────────────────────────────────────────────┤
│                 特征计算层                           │
│   实时计算(Flink) ←→ 特征同步 ←→ 离线计算(Spark)    │
├─────────────────────────────────────────────────────┤
│                 特征服务层                           │
│   GetOnlineFeatures() │ GetHistoricalFeatures()    │
└─────────────────────────────────────────────────────┘
```

**关键设计**：
- **双存储**：离线特征存Delta Lake（支持时间旅行，训练时可获取历史某时刻的特征）；在线特征存Redis（毫秒级读取）
- **特征同步**：离线计算完成后自动同步到在线存储；在线实时计算结果定期回写到离线存储
- **点查询优化**：在线特征按实体ID（如user_id）为Key存储，支持批量获取多个特征

### 问题3 参考答案

**（1）数据血缘的核心概念和追踪粒度**

**核心概念**：数据血缘（Data Lineage）描述数据从源头到最终消费的完整流转路径，包括数据来源、经过的转换处理、最终去向。

**追踪粒度**：

| 粒度 | 说明 | 示例 |
|------|------|------|
| **表级血缘** | 表与表之间的依赖关系 | 表A JOIN 表B → 表C |
| **字段级血缘** | 字段的来源和转换 | C.amount = A.price * B.quantity |
| **任务级血缘** | 哪个任务产出哪些数据 | Spark Job X → 表C |
| **记录级血缘** | 单条记录的来源（细粒度） | 特征值X来源于交易ID 123456 |

**（2）数据血缘的采集方式**

| 采集方式 | 原理 | 优缺点 |
|---------|------|--------|
| **静态解析** | 解析SQL/代码AST，分析字段引用关系 | 准确度高，但无法处理动态SQL |
| **运行时采集** | 在计算引擎中埋点，执行时记录数据流转 | 能捕获动态逻辑，但有性能开销 |
| **元数据集成** | 从Hive Metastore、Spark History Server等抽取 | 实现简单，但粒度较粗 |
| **手动标注** | 开发者在代码中显式声明血缘 | 灵活但依赖人工，易遗漏 |

**推荐方案**：静态解析（覆盖大部分SQL任务）+ 运行时采集（补充动态逻辑）

**（3）存储和查询方案**

**存储**：使用图数据库（Neo4j）或支持图查询的数据库（TigerGraph）存储血缘关系
- 节点：表、字段、任务、数据源
- 边：依赖关系（produces、consumes、transforms）
- 属性：时间戳、版本、转换逻辑

**查询**：
- 上游追溯：`MATCH (target:Field {name:'user_risk_score'})<-[:TRANSFORMS*]-(source) RETURN source`
- 下游影响：`MATCH (source:Table {name:'txn_raw'})-[:PRODUCES*]->(target) RETURN target`

**问题排查示例**：

场景：发现某用户的风险评分`user_risk_score`异常偏高

排查过程：
1. 查询特征`user_risk_score`的血缘：发现依赖`user_7d_txn_amount`特征
2. 继续追溯`user_7d_txn_amount`：来源于`txn_detail`表的`amount`字段聚合
3. 检查`txn_detail`表：发现某条交易的`amount`值异常（10000应为100.00，小数点错误）
4. 定位根因：上游数据源的数据质量问题

### 考点分析

- **数据湖仓一体**：Lakehouse架构、Delta Lake/Iceberg
- **特征工程**：Feature Store设计、训练服务一致性
- **数据血缘**：血缘采集、图存储、问题排查应用
- **实时数据平台**：流批一体架构

### 问题1 参考答案

**（1）大模型推理优化技术**

| 技术 | 原理 | 适用场景 |
|------|------|---------|
| **模型量化（Quantization）** | 将FP32/FP16权重量化为INT8/INT4，减少显存占用和计算量 | 显存受限、需要降低成本的场景 |
| **KV Cache优化** | 缓存已计算的Key-Value，避免自回归生成时重复计算 | 长文本生成场景 |
| **连续批处理（Continuous Batching）** | 动态组批，新请求随时加入正在处理的批次 | 高并发、请求长度差异大的场景 |
| **投机采样（Speculative Decoding）** | 用小模型快速生成候选token，大模型验证 | 延迟敏感场景 |
| **张量并行（Tensor Parallelism）** | 将模型权重切分到多GPU，并行计算 | 单模型超过单卡显存的场景（70B+模型） |
| **PagedAttention** | 类似虚拟内存分页管理KV Cache，减少显存碎片 | 长上下文、高并发场景 |
| **FlashAttention** | 优化Attention计算的访存模式，减少HBM访问 | 所有Transformer推理场景 |

**（2）模型量化原理与对比**

**基本原理**：将浮点数权重映射到低比特整数表示，推理时用整数运算替代浮点运算

量化公式：`Q = round(W / scale) + zero_point`  
反量化：`W' = (Q - zero_point) * scale`

**INT8 vs INT4对比**：

| 维度 | FP16（基线） | INT8量化 | INT4量化 |
|------|-------------|---------|---------|
| **显存占用** | 100%（70B≈140GB） | 50%（≈70GB） | 25%（≈35GB） |
| **推理速度** | 1x | 1.5-2x | 2-3x |
| **精度损失** | 基线 | 轻微（<1%） | 明显（1-5%） |
| **适用场景** | 精度优先 | 均衡选择 | 成本/速度优先 |

**量化方法**：
- **PTQ（训练后量化）**：无需重新训练，直接量化，简单快速但精度损失较大
- **QAT（量化感知训练）**：训练时模拟量化误差，精度更好但需要重新训练
- **GPTQ/AWQ**：针对LLM优化的量化方法，在INT4下仍保持较好精度

**（3）模型版本管理与无缝更新**

**模型版本管理**：
- 每个模型版本有唯一标识（如`llama-70b-v2.1`）
- 模型文件存储在S3，元数据（版本、配置、状态）存储在数据库
- 支持版本回退：保留最近N个版本的模型文件

**无缝更新机制（蓝绿部署）**：
1. 新版本模型加载到新的GPU实例组（绿组）
2. 新实例预热完成后，加入负载均衡池
3. 流量逐步从旧实例（蓝组）切换到新实例（绿组）：10%→50%→100%
4. 确认新版本稳定后，下线旧实例释放GPU资源
5. 如发现问题，流量立即切回旧版本

**（1）多租户资源隔离**

| 隔离方式 | 实现机制 | 适用场景 |
|---------|---------|---------|
| **命名空间隔离** | Kubernetes Namespace + ResourceQuota限制每租户的GPU/内存/CPU上限 | 基础隔离需求 |

| **队列隔离** | 每个租户独立的请求队列，防止单租户请求风暴影响其他租户 | 所有多租户场景 |
| **模型实例隔离** | 为大客户部署专属模型实例，不与其他租户共享 | 企业专属部署 |

**（2）SLA保障**

| SLA等级 | 响应时间 | 可用性 | 保障措施 |
|---------|---------|--------|---------|
| **铂金级** | P99<500ms | 99.99% | 专属GPU实例、优先队列、多AZ容灾 |
| **金级** | P99<2s | 99.9% | 共享高优先级队列、资源预留 |
| **银级** | P99<5s | 99.5% | 尽力服务、弹性资源 |

**（3）优先级调度**

```python
# 请求调度伪代码
class PriorityScheduler:
    def __init__(self):
        self.queues = {
            'platinum': PriorityQueue(max_size=100),   # 最高优先级
            'gold': PriorityQueue(max_size=500),
            'silver': PriorityQueue(max_size=2000),
        }
    
    def schedule(self, request):
        tier = request.tenant.sla_tier
        # 根据SLA等级入队
        self.queues[tier].enqueue(request)
    
    def get_next_request(self):
        # 按优先级顺序取请求（铂金 > 金 > 银）
        # 但保证低优先级不完全饿死（每处理10个高优先级，处理1个低优先级）
        for tier in ['platinum', 'gold', 'silver']:
            if not self.queues[tier].empty():
                return self.queues[tier].dequeue()
```

**优先级抢占**：当高优先级请求排队时间超过阈值，可中断低优先级的批处理任务，优先处理高优先级请求

|------|---------|
| **代码篡改** | 补丁包必须经过服务端签名，客户端校验签名后才执行 |
| **中间人攻击** | 使用HTTPS传输 + 证书锁定（Certificate Pinning） |
| **恶意补丁** | 补丁发布需要审批流程；补丁代码需经过安全扫描 |
| **版本混乱** | 补丁与APP版本绑定，不匹配的补丁自动跳过 |
| **应用商店合规** | 避免热更新核心功能（仅用于Bug修复），遵守平台政策 |

**（3）灰度发布和回退机制**

**灰度发布**：
```
补丁发布流程：
1. 开发提交补丁 → 安全扫描 → 审批
2. 灰度发布：5%用户（按设备ID哈希）
3. 观察指标：崩溃率、ANR率、用户反馈
4. 指标正常 → 扩大灰度：20% → 50% → 100%
5. 指标异常 → 暂停发布，排查问题
```

**回退机制**：
1. **自动回退**：客户端检测到补丁加载后崩溃次数超过阈值（如连续3次），自动禁用补丁，回退到原版本
2. **远程回退**：服务端下发"回退指令"，客户端删除本地补丁，恢复原版本代码
3. **版本覆盖**：发布新的"空补丁"覆盖有问题的补丁

**监控告警**：
- 实时监控补丁下载率、生效率、崩溃率
- 设置告警阈值：补丁生效后崩溃率上升>0.1%立即告警
