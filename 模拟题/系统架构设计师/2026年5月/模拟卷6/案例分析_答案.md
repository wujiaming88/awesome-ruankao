## 试题二（选做）参考答案

### 问题1（8分）

**1. 文本切片（Chunking）策略（4分）**

**作用**：文本切片将长文档分割成适当大小的文本块，以便于向量化存储和精确检索。切片策略直接影响检索的精度和LLM生成答案的质量——切片太大则检索不精确，切片太小则丢失上下文。

**常见切片方式**：
- **固定大小切片**：按固定字符/Token数切分，简单但可能切断语义
- **基于段落/句子的切片**：按自然段落或句子边界切分，保持语义完整
- **递归切片**：先按大结构（章节）切分，再递归切分超长片段
- **基于语义的切片**：利用NLP模型判断语义边界，在语义转换处切分
- **滑动窗口切片**：相邻切片之间有重叠（overlap），避免边界信息丢失

**2. 向量数据库的作用和产品（4分）**

**作用**：向量数据库存储文档切片的高维向量表示（Embedding），提供高效的近似最近邻（ANN）搜索能力，根据用户问题的向量快速检索出语义最相关的文档片段。

**常见产品**：
- **Milvus**（开源、支持亿级向量、分布式部署）
- **Pinecone**（全托管SaaS向量数据库）
- **Weaviate**（开源、支持混合搜索）
- **Qdrant**（开源、Rust实现、性能优秀）
- **FAISS**（Meta开源的向量检索库）
- **Chroma**（轻量级开源向量数据库）

### 问题2（9分）

**问题A解决方案**：针对复杂问题检索相关性差的问题：
1. **查询分解（Query Decomposition）**：将复杂问题拆分为多个子查询（如分别检索"A产品高温耐久性"和"B产品高温耐久性"），分别检索后合并结果
2. **HyDE（假设性文档嵌入）**：先让LLM生成一个假设性答案，再用该答案进行向量检索，提高检索与答案的语义匹配度
3. **混合检索**：结合关键词检索（BM25）和向量语义检索，互补各自的短板

**问题B解决方案**：针对LLM幻觉问题：
1. **引用溯源（Citation）**：要求LLM在回答中标注每个论述的来源文档编号，便于用户验证
2. **Faithfulness检测**：使用评估模型检查生成内容是否与检索到的上下文一致，不一致则标记或重新生成
3. **约束Prompt设计**：在System Prompt中明确要求"只根据提供的上下文回答，如果上下文不包含相关信息则回答'无法回答'"
4. **降低temperature参数**：减少LLM的随机性，使其更严格地遵循上下文

**问题C解决方案**：针对矛盾信息判断问题：
1. **文档元数据标注**：为每个文档标注发布时间、版本号、权威等级（如国标>企标>部门文件）
2. **时间优先策略**：当文档内容矛盾时，优先采用最新版本的文档
3. **权威度加权**：在检索排序中加入文档权威度权重，权威文档的检索得分更高
4. **冲突提示**：当检测到检索结果中存在矛盾时，在回答中明确提示用户并列出不同来源的观点

### 问题3（8分）

**1. Agentic RAG与传统RAG的主要区别（4分）**

| 维度 | 传统RAG | Agentic RAG |
|------|---------|-------------|
| 检索方式 | 单次检索 | 多次迭代检索，根据中间结果动态调整检索策略 |
| 流程控制 | 固定的线性流水线 | Agent自主决策，动态规划执行步骤 |
| 工具使用 | 仅使用向量检索 | 可调用多种工具（搜索引擎、数据库查询、计算器、API等） |
| 推理能力 | 简单的检索+生成 | 具备规划、推理、反思和自我纠错能力 |
| 适用场景 | 简单的知识问答 | 复杂的多步推理、分析对比、跨领域综合问题 |

**2. Agentic RAG的典型工作流程（4分）**

1. **理解问题**：Agent分析用户问题的意图和复杂度，判断是否需要多步检索
2. **制定计划**：将复杂问题分解为多个子任务，规划检索和推理步骤
3. **执行检索**：根据计划逐步执行检索，每步可选择不同的工具和数据源
4. **中间推理**：对检索到的信息进行分析和推理，评估是否需要补充检索
5. **反思验证**：检查当前已有的信息是否足够回答问题，验证中间结论的可靠性
6. **综合生成**：将所有收集到的信息综合起来，生成最终的结构化回答

## 试题三（选做）参考答案

### 问题1（9分）

**1. Service Mesh数据平面和控制平面（4分）**

**数据平面（Data Plane）**：
- 由部署在每个服务实例旁边的Sidecar代理（如Envoy）组成
- 负责处理服务间的所有网络通信流量
- 功能包括：请求路由、负载均衡、流量加密（mTLS）、超时重试、熔断、限流、请求监控和追踪数据采集

**控制平面（Control Plane）**：
- 集中管理和配置所有Sidecar代理（如Istio的istiod）
- 功能包括：下发路由规则和策略、管理证书和密钥、服务发现、配置分发、策略执行

**2. Service Mesh相比SDK方式的优势（5分）**

1. **语言无关**：Service Mesh通过Sidecar代理处理通信，不限制服务的开发语言；SDK方式通常与特定语言绑定（如Spring Cloud仅限Java）
2. **业务代码零侵入**：服务治理能力（熔断、限流、追踪）由Sidecar透明提供，无需修改业务代码；SDK方式需在代码中集成和配置
3. **统一升级管理**：治理策略和Sidecar版本可统一升级，不需要逐个服务修改和重新部署；SDK升级需要每个服务重新编译发布
4. **细粒度流量控制**：支持基于请求头、路径等条件的精细流量路由，便于灰度发布和A/B测试
5. **开箱即用的安全通信**：自动实现服务间的mTLS加密通信，无需应用层处理

### 问题2（8分）

**可观测性三大支柱**：

| 支柱 | 定义 | 开源工具 |
|------|------|---------|
| **指标（Metrics）** | 对系统运行状态的量化度量，包括请求量、延迟、错误率、CPU/内存使用率等时序数据 | Prometheus（采集）+ Grafana（可视化） |
| **日志（Logs）** | 对系统中离散事件的文本记录，包含时间戳、级别、上下文等信息 | ELK Stack（Elasticsearch + Logstash + Kibana）或 Loki |
| **追踪（Traces）** | 记录请求在分布式系统中跨服务的完整调用链路，包含每个服务的耗时和调用关系 | Jaeger 或 Zipkin；采集标准：OpenTelemetry |

**三者协同**：Metrics发现问题（如某服务P99延迟异常升高）→ Traces定位问题（追踪请求在哪个服务环节耗时异常）→ Logs分析问题（查看具体的错误日志和上下文）。

### 问题3（8分）

**常见容错模式**：

1. **熔断器（Circuit Breaker）**：当下游服务的错误率或响应时间超过阈值时，熔断器打开，直接返回降级响应而不再调用下游服务，避免故障蔓延。一段时间后进入半开状态，尝试恢复。
2. **限流（Rate Limiting）**：限制单位时间内的请求数量，防止突发流量压垮服务。常用算法：令牌桶、漏桶、滑动窗口计数器。
3. **超时控制（Timeout）**：为每次服务调用设置最大等待时间，超时后立即返回错误，释放资源。避免慢调用占用过多连接池资源导致资源耗尽。
4. **重试（Retry）**：对暂时性故障自动重试，通常结合指数退避（Exponential Backoff）策略。需要保证被调用方的幂等性。
5. **舱壁隔离（Bulkhead）**：将系统资源（如线程池、连接池）隔离到不同的"舱壁"中，一个服务的资源耗尽不会影响其他服务。类似船体的水密舱设计。
6. **降级（Fallback/Degradation）**：当服务不可用时，返回预设的降级结果（如缓存数据、默认值），确保核心功能可用。

## 试题四（选做）参考答案

### 问题1（9分）

| 对比维度 | 2PC（两阶段提交） | TCC | 基于消息队列的最终一致性 |
|---------|-------------------|-----|------------------------|
| **原理** | 协调者先向所有参与者发送Prepare请求，所有参与者准备就绪后再发送Commit请求。任一参与者失败则全部Rollback | 每个服务提供Try（资源预留）、Confirm（确认执行）、Cancel（取消回滚）三个接口 | 通过事务消息保证本地事务和消息发送的原子性，下游服务消费消息后执行本地事务，实现最终一致性 |
| **一致性** | 强一致性 | 最终一致性（但较快） | 最终一致性 |
| **性能** | 差（同步阻塞，锁定资源直到提交） | 较好（Try阶段只预留资源，不锁定） | 好（异步解耦，无同步等待） |
| **复杂度** | 低（框架支持，如XA） | 高（每个服务需实现三个接口，补偿逻辑复杂） | 中等（需要可靠消息中间件，需处理消息幂等） |
| **可用性** | 低（协调者单点问题，参与者阻塞） | 高（去中心化） | 高（异步，各服务独立） |
| **适用场景** | 同一数据库或XA兼容的资源管理器，对一致性要求极高 | 金融转账、库存扣减等需要"预留-确认-取消"语义的场景 | 订单创建后异步通知物流、积分等弱依赖场景 |

### 问题2（8分）

**1. Saga模式的基本原理（3分）**

Saga模式将一个长事务拆分为一系列本地事务（T₁, T₂, ..., Tₙ），每个本地事务都有一个对应的补偿事务（C₁, C₂, ..., Cₙ）。正常流程：T₁→T₂→...→Tₙ 依次执行。如果某个Ti执行失败，则反向执行已完成事务的补偿操作：Ci-1→Ci-2→...→C₁，将系统恢复到初始状态。

**2. 编排式与协同式Saga的区别（5分）**

| 维度 | 编排式Saga（Orchestration） | 协同式Saga（Choreography） |
|------|---------------------------|---------------------------|
| **协调方式** | 有一个中央协调器（Orchestrator）控制整个流程 | 无中央协调器，各服务通过事件驱动自主响应 |
| **通信方式** | 协调器依次调用各服务（命令式） | 服务之间通过发布/订阅事件进行通信 |
| **耦合度** | 协调器知道所有步骤，各服务只关心自己的事务 | 各服务需要知道要监听哪些事件，存在隐式耦合 |
| **优点** | 流程清晰可控、容易追踪状态、补偿逻辑集中管理 | 松耦合、无单点、各服务完全自治 |
| **缺点** | 协调器可能成为单点和复杂瓶颈 | 流程分散、难以追踪全局状态、调试困难 |
| **适用** | 步骤较多、流程复杂、需要清晰状态追踪的场景 | 步骤较少、服务间耦合度低的简单场景 |

### 问题3（8分）

**1. 跨地域部署选择最终一致性的原因（3分）**

根据CAP定理，在网络分区（P不可避免，尤其跨地域网络延迟和分区概率更高）存在时，系统只能在一致性（C）和可用性（A）之间二选一。跨地域场景下：
- 强一致性需要跨地域同步复制，网络延迟（通常几十到几百毫秒）会导致写入性能严重下降
- 全球用户需要低延迟的本地读写体验，强一致性的同步等待会影响用户体验
- 电商业务可以容忍短时间的数据不一致（如库存、订单状态），但不能容忍服务不可用

因此选择AP（可用性+分区容错），接受最终一致性。

**2. 最终一致性中的数据冲突处理策略（5分）**

1. **最后写入胜出（Last Write Wins, LWW）**：以时间戳最新的写入为准。简单高效，但依赖全局时钟同步，可能丢失并发写入。适用于对精确性要求不高的场景（如用户昵称更新）。

2. **向量时钟（Vector Clock）/版本向量**：每个节点维护一个逻辑时钟向量，通过比较向量判断事件的因果关系和冲突。检测到冲突时可以：
   - 自动合并（如CRDT数据结构）
   - 提交给应用层或用户手动解决

3. **CRDT（无冲突复制数据类型）**：使用满足交换律、结合律的数据结构（如G-Counter、OR-Set），保证在不同节点独立更新后自动合并为一致状态，无需人工干预冲突。

4. **业务规则仲裁**：根据业务语义制定冲突解决规则。例如：库存冲突时取最小值（保守策略），价格冲突时以主数据中心为准。

### 问题2（9分）

**挑战1：Oracle存储过程迁移（3分）**
1. 使用Oracle迁移工具（如达梦提供的DTS数据迁移工具）自动转换PL/SQL语法
2. 对工具无法自动转换的语法（如Oracle特有的包、类型、管道函数等），编写人工适配脚本
3. 建立语法映射表，逐一比对Oracle和目标数据库的语法差异
4. 将复杂存储过程重构为应用层代码，减少对数据库的依赖

**挑战2：大数据量不停机迁移（3分）**
1. **增量迁移方案**：先全量迁移历史数据（可在夜间低峰执行），再通过数据库日志（如Oracle LogMiner/GoldenGate）捕获增量变更同步到目标库
2. **双写过渡**：在过渡期间应用层同时写入新旧两个数据库，逐步切换读流量
3. **灰度切换**：按业务模块分批切换，先切非核心模块验证，再切核心模块

**挑战3：数据完整性和业务逻辑验证（3分）**
1. **数据校验工具**：开发自动化数据对比工具，逐表比对新旧数据库的行数、校验和（Checksum）
2. **回归测试**：建立完整的自动化测试用例集，覆盖核心业务流程，在新数据库上执行全量回归测试
3. **并行运行**：新旧系统并行运行一段时间，对比两边的业务处理结果是否一致
4. **关键数据抽样验证**：对高价值/高敏感数据进行人工抽样核对

### 问题3（8分）

**分阶段迁移策略**：

**第一阶段：评估和试点（3-6个月）**
- 对现有系统进行全面的技术栈评估和依赖分析
- 选择1-2个非核心子系统作为试点进行国产化迁移
- 验证国产替代产品的兼容性和性能
- 积累迁移经验，建立迁移工具链和方法论

**第二阶段：外围系统迁移（6-9个月）**
- 将非核心的外围系统（如内部OA、文档管理等）迁移到国产平台
- 优化迁移流程，解决第一阶段发现的问题
- 建立国产平台的运维团队和能力

**第三阶段：核心系统迁移（6-12个月）**
- 对核心业务系统进行迁移（数据库、中间件、操作系统）
- 采用灰度切换策略，逐步将流量从旧系统切换到新系统
- 保留旧系统作为回退方案，确保可随时回滚

**第四阶段：优化和稳定（3-6个月）**
- 全面下线旧系统
- 对国产化后的系统进行性能调优
- 完善运维监控和应急预案

**分阶段策略的优势**：
1. **风险可控**：每个阶段的范围有限，即使出现问题也不会影响整体业务
2. **渐进验证**：前期阶段积累的经验可以指导后续迁移，减少返工
3. **业务连续性**：不需要一次性停机切换，对业务影响最小
4. **团队适应**：给开发和运维团队足够的时间学习国产平台技术栈
5. **回退可行**：每个阶段完成后都可以评估，决定是否继续或调整方案
