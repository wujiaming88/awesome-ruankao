## 论文一：论隐私计算架构的设计与应用

### 写作要点

**1. 隐私计算主要技术路线**

| 技术路线 | 原理 | 优点 | 缺点 | 适用场景 |
|---------|------|------|------|---------|
| **联邦学习** | 数据不出本地，各方在本地训练模型，仅交换模型参数/梯度 | 数据不出域，支持大规模数据，工程成熟 | 通信开销大，模型精度可能下降，梯度泄露风险 | 多方联合建模（如银行联合风控） |
| **安全多方计算（MPC）** | 多方联合计算一个函数，每方只知道自己的输入和输出 | 密码学安全保障，精度无损 | 计算和通信开销极大，难以扩展 | 少数参与方的高安全场景（如联合征信查询） |
| **同态加密** | 在密文上直接计算，解密后得到正确结果 | 数据全程加密，安全性最高 | 计算性能极低（全同态），实用性受限 | 简单统计计算（如加法求和） |
| **差分隐私** | 在计算结果中添加噪声，使单个记录的影响不可区分 | 理论保障强，实现简单 | 噪声影响数据可用性，隐私预算有限 | 统计分析发布、模型训练（如Apple、Google） |
| **可信执行环境（TEE）** | 利用硬件（如Intel SGX/TDX、ARM TrustZone）提供隔离计算环境 | 性能好，通用计算 | 依赖硬件，侧信道攻击风险 | 对性能要求高的场景 |

**2. 架构设计考量**
- 数据分级：根据数据敏感等级选择不同的保护级别
- 计算类型：简单统计用同态加密/MPC，复杂模型训练用联邦学习
- 参与方数量：少量参与方用MPC，大量参与方用联邦学习
- 性能需求：对实时性要求高用TEE，可接受延迟用MPC/同态加密
- 合规要求：根据《数据安全法》《个人信息保护法》的要求选择技术方案

**3. 典型架构**
```
数据方A ←→ 隐私计算平台（协调节点） ←→ 数据方B
                    ↓
            计算结果（密文/模型参数）
                    ↓
            结果方（解密/聚合）
```

### 范文框架

1. **项目介绍（300字）**：金融机构联合风控平台、医疗数据共享平台等
2. **技术路线论述（1000字）**：
   - 联邦学习的横向/纵向/迁移学习三种模式
   - MPC的秘密共享和混淆电路
   - 技术选型的决策矩阵
3. **项目实践（1200字）**：
   - 需求：多家银行联合反欺诈建模，数据不出本地
   - 方案：纵向联邦学习+TEE混合架构
   - 挑战：通信效率、模型精度损失、参与方信任
   - 解决：压缩通信、安全聚合协议、第三方审计
   - 效果：欺诈识别率提升15%，数据合规零风险

## 论文二：论软件架构演进策略及其应用

### 写作要点

**1. 常见架构演进策略**

| 策略 | 原理 | 适用场景 | 优缺点 |
|------|------|---------|--------|
| **绞杀者模式（Strangler Fig）** | 在旧系统外围构建新系统，逐步将流量从旧系统切换到新系统，直到旧系统被完全替换 | 单体→微服务的渐进式迁移 | 优：风险低、渐进式；缺：需要维护新旧两套系统 |
| **防腐层模式（Anti-Corruption Layer）** | 在新旧系统之间建立翻译层，隔离旧系统的数据模型和接口对新系统的污染 | 新系统需要与遗留系统集成 | 优：新系统不被旧模型污染；缺：增加了一层间接性 |
| **分支抽象（Branch by Abstraction）** | 引入抽象层，新旧实现共存于同一代码库，通过开关切换 | 替换系统内部组件（如替换ORM框架） | 优：无需独立部署新系统；缺：代码复杂度临时增加 |
| **并行运行（Parallel Run）** | 新旧系统同时运行，对比两者的处理结果，验证新系统的正确性 | 对正确性要求极高的场景（如金融） | 优：验证充分；缺：资源开销大 |
| **修缮者模式（Decorating Collaborator）** | 通过在旧系统上附加新功能（如拦截器、装饰器），逐步改善旧系统 | 旧系统暂时无法替换，但需要增强功能 | 优：改动小；缺：只能增强，不能重构 |

**2. 架构演进的关键原则**
- 渐进式演进：小步快跑，持续交付
- 可逆性：每一步都应可回滚
- 业务连续性：演进过程不能中断业务
- 数据迁移优先：数据是最难迁移的资产
- 可观测性：演进过程需要充分的监控和指标

### 范文框架

1. **项目介绍（300字）**：传统电商系统从单体架构向微服务迁移
2. **策略论述（1000字）**：详细阐述3-4种演进策略
3. **项目实践（1200字）**：
   - 现状分析：单体应用的痛点（部署慢、扩展难）
   - 策略选择：绞杀者模式+防腐层
   - 实施过程：分3个阶段（外围服务→核心服务→数据库拆分）
   - 挑战：数据库拆分困难、分布式事务、团队技能转型
   - 效果：发布频率从月级提升到周级，系统可用性提升到99.95%

## 论文三：论多模态AI系统的架构设计

### 写作要点

**1. 多模态AI系统核心架构层次**

```
┌─────────────────────────────────────┐
│          应用服务层                   │
│  智能客服 | 影像诊断 | 视频分析      │
├─────────────────────────────────────┤
│          推理服务层                   │
│  模型推理引擎 | 模型管理 | A/B测试   │
├─────────────────────────────────────┤
│          模态融合层                   │
│  早期融合 | 晚期融合 | 注意力融合     │
├─────────────────────────────────────┤
│          特征提取层                   │
│  文本编码器 | 视觉编码器 | 音频编码器 │
├─────────────────────────────────────┤
│          数据处理层                   │
│  数据采集 | 预处理 | 标注 | 增强      │
└─────────────────────────────────────┘
```

**2. 关键技术**
- **模态对齐**：CLIP、ALIGN等对比学习方法实现跨模态表示对齐
- **融合策略**：
  - 早期融合：在特征提取前合并原始数据
  - 晚期融合：各模态独立预测后合并结果
  - 中间融合：通过Cross-Attention等机制在中间层融合
- **推理优化**：模型量化、模型蒸馏、TensorRT加速、动态Batch
- **模型服务化**：Triton Inference Server、TorchServe

**3. 架构挑战**
- 多源异构数据的统一存储和管理
- 模态缺失时的鲁棒处理
- 推理延迟与精度的平衡
- GPU资源的高效调度和共享

### 范文框架

1. **项目介绍（300字）**：工业质检（视觉+传感器数据）、智能客服（文本+语音+图像）
2. **架构论述（1000字）**：四层架构的详细设计
3. **项目实践（1200字）**：
   - 需求：工业缺陷检测系统，融合高清图像+X光+超声波
   - 技术选型：视觉编码器用ResNet/ViT，融合用Cross-Attention
   - 挑战：推理延迟高（需<100ms）、GPU资源紧张、标注成本高
   - 解决：模型量化INT8、TensorRT加速、半监督学习减少标注
   - 效果：缺陷检出率98.5%，误检率<1%，单次推理<50ms

## 论文四：论软件系统的可维护性设计

### 写作要点

**1. 可维护性子特性（ISO 25010）**
- 可分析性：定位问题或变更影响范围的难易程度
- 可修改性：修改缺陷或适应需求变更的难易程度
- 稳定性：修改后引入新缺陷的风险程度
- 可测试性：验证修改正确性的难易程度
- 标准符合性：遵循可维护性相关标准的程度

**2. 架构层面的可维护性设计**
- **模块化设计**：高内聚低耦合，清晰的模块边界
- **分层架构**：表示层/业务层/数据层严格分离
- **依赖注入（DI）**：通过IoC容器管理依赖，便于替换组件
- **接口编程**：面向接口而非实现编程，支持灵活替换
- **配置外部化**：将可变参数从代码中分离，通过配置中心管理
- **API版本管理**：支持API的向后兼容和平滑升级

**3. 代码层面的可维护性实践**
- **SOLID原则**：单一职责、开放封闭、里氏替换、接口隔离、依赖倒置
- **Clean Code**：有意义的命名、短小的函数、避免重复
- **设计模式**：策略模式、模板方法模式等提高扩展性
- **代码审查**：建立代码审查机制和编码规范
- **自动化测试**：单元测试覆盖率、持续集成
- **技术债务管理**：定期偿还技术债务，使用SonarQube等工具度量代码质量

### 范文框架

1. **项目介绍（300字）**：大型企业ERP系统或政务服务平台
2. **设计原则论述（1000字）**：架构+代码两个层面的详细实践
3. **项目实践（1200字）**：
   - 背景：系统运行5年，代码量50万行，维护困难
   - 问题：修改一处影响全局、缺乏测试、文档缺失
   - 改进：引入分层架构重构、建立自动化测试体系、代码质量度量
   - 效果：平均缺陷修复时间从5天降到1天，变更引入缺陷率降低60%
