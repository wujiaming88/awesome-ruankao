## 试题一（必答题）：质量属性与架构评估 — 智慧医疗影像AI辅助诊断平台

### 题目描述

某三甲医院拟建设一套**智慧医疗影像AI辅助诊断平台**。该平台接收各科室上传的医学影像数据（CT、MRI、X光等），利用AI模型自动分析影像，生成辅助诊断报告，辅助医生做出临床诊断决策。平台同时需要对接医院现有的HIS（医院信息系统）和PACS（影像存储与传输系统）。

在需求分析阶段，项目组整理了以下需求：

（a）影像上传后，AI模型应在30秒内完成分析并返回初步诊断结果。
（b）平台应支持同时接入多家AI算法厂商的模型，当某个AI模型不可用时，自动切换到备用模型。
（c）当存储服务器出现故障时，系统应在30秒内自动切换到异地灾备中心，确保影像数据不丢失。
（d）所有患者的影像数据和诊断结果必须加密存储，传输过程使用国密SM4算法加密，确保符合《数据安全法》和《个人信息保护法》要求。
（e）平台应支持根据医院科室增长需求，在2个工作日内完成新科室的接入和配置。
（f）系统应提供完整的操作审计日志，记录每一次影像查看、诊断报告生成和修改操作。
（g）AI模型的推理结果应包含置信度评分，当置信度低于阈值时自动标记为"需人工复核"。
（h）在工作日高峰时段（8:00-12:00），平台需支持同时处理500个以上的影像分析请求。
（i）平台应提供标准化的HL7 FHIR接口，支持与院内外其他医疗信息系统的数据交换。
（j）AI诊断模型的更新迭代不应影响在线服务的可用性，要求支持灰度发布和A/B测试。

## 试题二（选做）：RAG系统架构设计 — 企业知识库智能问答平台

### 题目描述

某大型制造企业拟建设一个**基于RAG（检索增强生成）的企业知识库智能问答平台**。该平台整合企业内部的技术文档、产品手册、质量标准、工艺规范等海量非结构化文档，利用大语言模型（LLM）为研发人员、质量工程师和一线工人提供智能问答服务。

系统架构师张工设计了如下RAG架构：

```
用户提问 → 查询预处理模块 → 向量检索模块 → 重排序模块 → 上下文构建模块 → LLM生成模块 → 回答输出
                                    ↑
                            文档处理流水线：
                    原始文档 → 文档解析 → 文本切片 → 向量化 → 向量数据库
```

### 问题1（8分）

请解释RAG架构中以下核心组件的功能和技术选型：
1. 文本切片（Chunking）策略的作用，以及常见的切片方式有哪些？（4分）
2. 向量数据库在RAG中的作用，列举至少2种常见的向量数据库产品。（4分）

### 问题2（9分）

在该系统的实际运行中，发现以下问题：
- **问题A**：用户提出复杂问题时（如"对比A产品和B产品在高温环境下的耐久性"），检索结果的相关性较差。
- **问题B**：LLM生成的回答有时会出现"幻觉"，即回答内容与检索到的文档不一致。
- **问题C**：当多个文档包含矛盾信息时，系统无法判断哪个文档更权威。

请分别给出上述三个问题的技术解决方案。

### 问题3（8分）

李工提出该系统应从传统的"单次检索+单次生成"升级为"Agentic RAG"（智能体RAG）架构。请说明：
1. Agentic RAG与传统RAG的主要区别。（4分）
2. Agentic RAG的典型工作流程。（4分）

## 试题四（选做）：分布式数据库与事务一致性 — 跨境电商订单系统

### 题目描述

某跨境电商公司的**订单系统**需要处理来自全球多个地区的交易。系统采用微服务架构，核心服务包括：订单服务、库存服务、支付服务、物流服务。每笔订单的处理涉及多个服务的协同操作（创建订单→扣减库存→处理支付→创建物流单）。

由于跨服务操作涉及多个数据库实例，传统的本地事务无法保证全局数据一致性。架构师需要选择合适的分布式事务方案。

### 问题1（9分）

请对比以下三种分布式事务方案的原理、优缺点和适用场景：
1. 两阶段提交协议（2PC）
2. TCC（Try-Confirm-Cancel）模式
3. 基于消息队列的最终一致性方案（事务消息）

### 问题2（8分）

架构师最终选择了基于Saga模式来处理该跨境电商的订单事务。请回答：
1. Saga模式的基本原理是什么？（3分）
2. 编排式Saga（Orchestration）和协同式Saga（Choreography）的区别。（5分）

### 问题3（8分）

在跨境场景下，该系统还面临数据跨地域部署的挑战。请说明：
1. 为什么跨地域部署时通常选择最终一致性而非强一致性？（3分）
2. 在最终一致性方案中，如何处理数据冲突？给出至少2种策略。（5分）

## 试题五（选做）：信创替代架构设计 — 政务信息化系统国产化迁移

### 题目描述

某省级政府部门运营的**电子政务综合服务平台**，原系统基于Oracle数据库、IBM中间件、Windows Server操作系统构建。根据国家信创（信息技术应用创新）政策要求，需要在2年内完成核心系统的国产化替代。

原系统技术栈：
- 数据库：Oracle 19c（100+存储过程、大量PL/SQL逻辑）
- 中间件：IBM WebSphere
- 操作系统：Windows Server 2019
- 应用框架：Java EE（Struts2 + Hibernate）

### 问题1（8分）

请为该系统设计信创替代方案，针对数据库、中间件、操作系统、应用框架四个层面分别给出国产化替代产品选型建议，并说明选型理由。

### 问题2（9分）

在数据库从Oracle迁移到国产数据库的过程中，面临以下技术挑战：
1. 100+个Oracle存储过程需要迁移，部分使用了Oracle特有的PL/SQL语法。
2. 数据量超过10TB，迁移过程中不能长时间停机。
3. 需要验证迁移后数据的完整性和业务逻辑的正确性。

请给出应对上述三个挑战的技术方案。

在信创替代过程中，项目团队建议采用"分阶段迁移"策略而非"一步到位"策略。请说明分阶段迁移策略的具体步骤（至少3个阶段），以及这种策略相比一步到位策略的优势。
