### 问题2（11分）

**1. Prompt注入攻击的原理和危害（4分）**

**原理**：Prompt注入攻击是一种针对大语言模型（LLM）的攻击方式，攻击者通过在用户输入中嵌入精心构造的指令性文本，试图覆盖或绕过系统预设的System Prompt（系统提示词），从而操纵大模型的行为。其核心原理在于：LLM无法在语义层面严格区分"系统指令"和"用户数据"，当攻击者将恶意指令伪装成普通输入数据时，模型可能将其当作指令执行。

**危害**（至少答出3点）：
- **风控决策被操纵**：攻击者可能通过注入指令让模型输出"低风险"的判断结果，绕过风控审核，导致欺诈交易通过
- **敏感数据泄露**：攻击者可能诱导模型泄露System Prompt中包含的系统配置信息、业务规则或其他客户的隐私数据
- **模型行为偏离**：攻击者可能让模型忽略原有的角色设定和约束，执行与风控业务无关的指令
- **合规风险**：被攻击后的输出可能违反金融监管要求，导致合规事故

**2. 防范Prompt注入攻击的技术措施（7分，至少4种）**

1. **输入验证与清洗**（2分）：对用户输入进行严格的格式校验和关键词过滤，检测并移除可能的指令性文本（如"忽略上述指令"、"你是一个..."等模式）。使用正则表达式和NLP模型对输入进行异常检测。

2. **Prompt与数据分离**（2分）：将系统提示词（System Prompt）和用户输入数据严格隔离。使用特殊的分隔符或标记明确区分指令区域和数据区域，并在System Prompt中强调"以下内容为用户数据，不包含任何指令"。采用结构化输入格式（如JSON），而非将用户输入直接拼接到Prompt中。

3. **输出检测与过滤**（2分）：对大模型的输出结果进行内容安全检测，使用分类模型判断输出是否偏离预期的格式和内容范围。建立输出白名单/模板，只允许模型输出符合预定义格式的结构化结果（如JSON格式的风险评级结果）。

4. **LLM防火墙/安全网关**（1分）：在大模型的输入和输出端部署专门的安全网关（如Guardrails、NeMo Guardrails），对所有请求和响应进行实时安全检测和拦截。

5. **最小权限原则**（附加）：限制大模型可访问的数据范围和可执行的操作类型。模型只能接收脱敏后的数据，只能输出预定义格式的风控评分，不能执行数据查询或系统操作。

6. **对抗性测试（Red Teaming）**（附加）：定期使用自动化工具和人工团队对系统进行Prompt注入攻击测试，发现和修复潜在的注入漏洞。建立Prompt注入攻击样本库，持续更新防御策略。

### 问题2（9分）

**"被遗忘权"在分布式系统中的技术挑战与解决方案**：

**挑战1：数据副本的全面定位（3分）**

分布式系统中数据通常存在多个副本：主数据库、从库/读副本、缓存（Redis）、搜索索引（Elasticsearch）、消息队列中未消费的消息、日志文件、备份快照、CDN缓存等。确保所有副本都被删除是极大的挑战。

**解决方案**：
- 建立**数据血缘图谱（Data Lineage）**，记录每条数据在系统中的所有流转路径和存储位置
- 实现**级联删除机制**：删除请求通过事件驱动方式广播到所有数据存储节点，每个节点执行本地删除并确认
- 使用**数据主体标识符（Subject ID）**作为全局索引键，支持按主体ID一键定位和删除所有关联数据

**挑战2：备份与归档数据的删除（3分）**

数据库备份快照和归档数据通常以整体形式存储（如全量备份文件、冷存储归档），无法从中删除单条记录而不影响整体完整性。

**解决方案**：
- **加密删除（Crypto-Shredding）**：为每个数据主体使用独立的加密密钥加密其数据，删除时只需销毁对应的密钥，即使备份中仍有密文也无法解密，等效于数据删除
- **延迟删除标记**：在主系统中立即标记为已删除，备份数据在下次备份轮转时自然过期淘汰
- **细粒度备份策略**：将敏感数据和非敏感数据分开备份，敏感数据使用更短的备份保留周期

**挑战3：第三方数据共享的删除传播（3分）**

当数据已经通过API或数据共享协议传输给第三方合作伙伴时，无法直接控制第三方是否真正删除了数据。

**解决方案**：
- **合同约束**：在数据共享协议中明确约定被遗忘权的响应义务和响应时限（GDPR要求无不当延迟，通常30天内）
- **删除通知API**：提供标准化的删除通知接口，当收到删除请求时自动通知所有数据接收方
- **数据使用凭证（Token化）**：不直接共享原始数据，而是提供Token化引用，删除时撤销Token即可使第三方无法再访问原始数据
- **审计与验证**：要求第三方提供删除证明，定期进行合规审计

**核心设计要点**（每点2分）：

1. **数据本地化存储（Data Residency）**：各地区的原始数据（特别是个人信息和重要数据）严格存储在本地节点，不做跨境物理传输。中国的重要数据和个人信息存储在中国大陆节点，欧洲用户数据存储在欧洲节点。

2. **合规网关（Compliance Gateway）**：在每个地区节点部署合规网关，作为数据出境的唯一关口。网关自动检查数据分类级别，L1级数据禁止出境，L2级数据需经过安全评估审批流程，L3/L4级数据根据脱敏/加密规则处理后方可传输。网关还负责记录所有跨境数据流的审计日志。

4. **差异化合规策略**：合规策略中心维护各地区的法规要求映射表。传输至欧洲节点的数据自动触发GDPR合规检查（同意授权、数据主体权利支持）；从中国出境的数据自动触发安全评估流程。支持策略的动态更新以适应法规变化。

## 试题三（选做）参考答案

### 问题1（9分）

**1. 事件溯源（Event Sourcing）的基本原理（3分）**

事件溯源是一种数据持久化模式，其核心思想是：**不存储实体的当前状态，而是存储导致状态变化的所有事件（Event）的有序序列**。实体的当前状态通过按顺序重放（Replay）所有相关事件来重建。

例如，在证券交易清算系统中，不直接存储"账户余额=10000"，而是存储一系列事件：`[买入股票A(-5000), 卖出股票B(+8000), 分红(+200), 手续费(-50)]`，账户的当前状态通过重放这些事件计算得出。

每个事件是不可变的（Immutable），只能追加不能修改或删除，形成完整的变更历史。

**2. CQRS（命令查询职责分离）的基本原理（2分）**

CQRS将系统的数据操作分为两个独立的模型：
- **命令模型（Command/Write Model）**：负责处理写操作（创建、更新、删除），包含完整的业务逻辑和验证规则
- **查询模型（Query/Read Model）**：负责处理读操作（查询），为特定的查询场景优化数据结构，可以使用不同的数据存储

两个模型可以独立扩展、独立优化，读写操作不再相互影响。

**3. 为什么两者经常结合使用（4分）**

- **事件溯源天然支持CQRS**：事件溯源存储的是事件序列，直接查询事件流来获取业务视图效率很低。CQRS的读模型可以通过订阅事件流来构建针对特定查询场景优化的物化视图，解决了事件溯源的查询效率问题。
- **CQRS的读模型需要数据来源**：在CQRS架构中，读模型需要与写模型保持同步。事件溯源产生的事件流是读模型同步数据的天然来源——读模型订阅事件流，消费事件后更新自己的数据结构。
- **互补优势**：事件溯源提供了完整的审计追踪和状态重建能力（解决审计问题），CQRS提供了读写分离和独立优化能力（解决性能问题）。两者结合，既有完整的历史记录，又有高效的查询能力。

### 问题2（8分）

**1. 解决事件重放性能问题（4分）**

**快照（Snapshot）机制**：定期对聚合根的当前状态创建快照。重建状态时，不需要从第一个事件开始重放，而是从最近的快照开始，只重放快照之后的事件。

具体策略：
- **定期快照**：每N个事件（如每1000个事件）或每隔固定时间自动创建快照
- **按需快照**：当检测到重放耗时超过阈值时触发快照创建
- **快照存储**：快照可以存储在独立的快照表/存储中，包含聚合根ID、版本号、序列化的状态和创建时间

以证券清算系统为例：每日日终清算完成后为每个账户创建快照，次日只需重放当天的新事件即可，无需重放历史所有事件。

**分区和归档**：将历史事件按时间分区存储，热数据（近期事件）和冷数据（历史事件）分离，减少活跃事件存储的数据量。

**2. 事件Schema版本兼容处理（4分）**

- **事件版本化（Event Versioning）**：每个事件携带版本号（如v1、v2），事件处理器根据版本号选择对应的反序列化和处理逻辑。

- **事件上转型（Upcasting）**：当读取旧版本事件时，通过Upcaster组件将旧版本事件转换为最新版本的格式后再处理。例如：v1事件中`{name: "张三"}`升级为v2事件中`{firstName: "三", lastName: "张"}`，Upcaster自动完成字段拆分。

- **弱Schema设计**：事件payload使用灵活的格式（如JSON），新增字段使用默认值，删除字段时保持向后兼容。遵循"只增不删不改"的Schema演进原则。

- **事件转换管道（Transformation Pipeline）**：在事件存储和消费者之间建立转换层，对不同版本的事件统一转换为消费者期望的格式。

### 问题3（8分）

**1. 读写模型之间的数据一致性保证（4分）**

CQRS架构中读写模型分离后，两者之间采用**最终一致性（Eventual Consistency）**模型：

- **事件驱动同步**：写模型执行完命令后发布领域事件，读模型订阅事件并异步更新自己的数据视图。通过消息队列（如Kafka）保证事件的可靠投递和顺序消费。

- **一致性延迟窗口**：读模型的更新存在一定延迟（通常毫秒到秒级）。在证券清算系统中，清算命令执行后，查询端的持仓和余额视图可能延迟数百毫秒才更新。

- **补偿机制**：
  - 事件重放：当读模型出现数据不一致时，可以通过重放事件流来重建读模型
  - 版本校验：读模型记录已处理的事件版本号，确保不漏消费和不重复消费
  - 幂等处理：读模型的事件处理器必须保证幂等性，同一事件多次消费结果一致

- **对于强一致性需求**：少数需要强一致的查询场景（如实时余额查询），可直接查询写模型而非读模型，或在写操作返回时附带最新状态。

**2. CQRS如何解决"历史查询影响在线交易"的问题（4分）**

在传统CRUD架构中，历史数据查询（如生成月度报表、合规审计查询）和在线交易处理共用同一数据库，复杂查询会锁表或消耗大量I/O，影响在线交易的性能。

CQRS的解决方案：

- **物理隔离**：读模型和写模型使用完全独立的数据库实例。写模型使用事件存储（如EventStoreDB或关系数据库），专注于高吞吐的事件写入。读模型可以使用适合不同查询场景的多种数据库：
  - 实时持仓查询 → Redis/内存数据库（低延迟）
  - 历史交易报表 → ClickHouse/OLAP数据库（复杂分析）
  - 全文搜索 → Elasticsearch（文本检索）

- **独立扩展**：读模型可以水平扩展多个副本以承载大量查询请求，不会影响写模型的性能。在日终清算高峰期，可以临时缩减读模型资源，集中资源保障写操作。

- **查询优化**：读模型的数据结构可以针对特定查询场景预先聚合和反范式化（如预计算好日/月/年维度的统计数据），避免复杂的JOIN操作。

### 问题1（8分）

**统一可观测性架构设计**：

- **统一数据模型**：所有遥测数据携带统一的上下文标签（service.name、trace_id、span_id、deployment.environment），实现Metrics-Traces-Logs三者的关联
- **统一采集标准**：强制所有微服务使用OpenTelemetry SDK，消除各团队使用不同监控工具的问题

### 问题2（9分）

**1. Trace、Span和SpanContext的含义及关系（5分）**

- **Trace（链路/追踪）**：代表一次完整的请求在分布式系统中的端到端调用链路。例如，用户点击播放视频的一次请求，从API网关开始，经过认证服务→用户服务→内容服务→CDN调度→播放器服务的完整调用过程，构成一个Trace。每个Trace由一个全局唯一的TraceID标识。

- **Span（跨度）**：代表Trace中的一个操作单元，记录了一次具体的服务调用或操作。每个Span包含：SpanID（唯一标识）、操作名称、开始时间、持续时间、状态码、标签（Tags）和日志（Logs）。Span之间通过ParentSpanID形成父子关系，构成树状调用结构。

- **SpanContext（跨度上下文）**：是跨进程传播Trace信息的载体，包含TraceID、SpanID、采样标志等信息。当服务A调用服务B时，SpanContext通过HTTP Header（如`traceparent`）或gRPC Metadata传递给下游服务，使得下游服务能够创建属于同一Trace的子Span。

**关系**：一个Trace由多个Span组成，Span之间通过父子关系形成调用树。SpanContext是Span之间跨进程传播的上下文信息，是实现分布式链路追踪的关键机制。

**2. 常见的采样策略及其优缺点（4分）**

| 采样策略 | 原理 | 优点 | 缺点 |
|---------|------|------|------|
| **头部采样（Head-based Sampling）** | 在请求入口处（如API网关）决定是否采样，整个Trace要么全部采集，要么全部丢弃。常用概率采样（如采样率10%） | 实现简单，资源开销可控 | 可能丢失重要的错误或慢请求Trace |
| **尾部采样（Tail-based Sampling）** | 等整个Trace完成后，根据Trace的特征（如是否包含错误、延迟是否超过阈值）再决定是否保留 | 能确保异常Trace被保留 | 需要缓存完整Trace数据后再决策，内存和计算开销大；实现复杂 |
| **自适应采样（Adaptive Sampling）** | 根据服务的流量和错误率动态调整采样率。流量低时采样率高，流量高时降低采样率 | 在资源和覆盖率之间自动平衡 | 实现复杂，需要实时监控指标来动态调整 |
| **基于规则的采样** | 根据预定义规则决定采样（如特定API路径全采、特定用户全采、健康检查请求不采） | 可针对关键业务保证全量采集 | 规则维护成本高，不够灵活 |

### 问题3（8分）

**基于根因分析（RCA）的智能告警方案设计**：

**1. 工作原理（4分）**

传统告警方案为每个监控指标设置独立阈值，当大规模级联故障发生时，上下游服务同时触发告警，产生"告警风暴"。基于RCA的智能告警方案的核心思路是：**先抑制告警风暴，再定位根因，只发送根因告警**。

工作流程：
1. **告警聚合**：短时间内（如5分钟窗口）爆发的多条告警进行聚合，按时间关联性和服务调用关系分组
2. **拓扑关联**：基于服务调用拓扑图（Service Dependency Graph），分析告警之间的因果关系。如果上游服务A告警，且A的下游服务B、C、D也同时告警，则B、C、D大概率是衍生告警
3. **根因推断**：沿着调用链自底向上或自顶向下分析，找到最早出现异常且处于调用链上游的服务节点，判定为根因
4. **告警收敛**：仅发送根因告警，衍生告警折叠为根因告警的关联信息，大幅减少告警数量

**2. 关键技术（4分）**

- **服务依赖拓扑自动发现**：通过分析分布式链路追踪数据（Trace），自动构建服务之间的实时调用关系图。拓扑图需要实时更新，反映服务间的实际调用关系和调用权重。

- **时序因果分析**：利用格兰杰因果检验（Granger Causality）或基于图神经网络（GNN）的因果推断模型，分析不同服务告警指标之间的时序因果关系，区分"因"和"果"。

- **异常传播建模**：基于服务拓扑图建立故障传播模型。利用随机游走（Random Walk）或PageRank变体算法，计算每个告警节点作为根因的概率得分，得分最高的节点判定为最可能的根因。

- **告警降噪规则引擎**：建立告警去重（相同告警合并）、告警抑制（根因告警触发后抑制衍生告警）、告警静默（维护窗口期静默）等规则，减少运维人员的告警疲劳。

### 问题2（8分）

**1. "货物入库时根据货物类型选择不同的存储策略"——工厂方法模式（Factory Method）或抽象工厂模式（Abstract Factory）（2分）**

**理由**：系统需要根据货物类型（常温、冷链、危化品）创建不同的存储策略对象。工厂方法模式将对象的创建延迟到子类/工厂类，客户端代码无需知道具体的策略类，只需传入货物类型即可获得对应的策略实例。当新增货物类型时，只需新增具体工厂和具体策略类，符合开闭原则。

**2. "出库订单的拣货策略有多种，可动态切换"——策略模式（Strategy Pattern）（2分）**

**理由**：系统有多种拣货策略（FIFO、LIFO、按批次号），且需要在运行时动态切换。策略模式将算法封装为独立的策略类，通过统一的接口调用，支持在运行时根据订单属性或业务规则动态选择和切换拣货策略，且新增策略不影响现有代码。

**3. "库存变动需实时通知相关系统"——观察者模式（Observer Pattern）（2分）**

**理由**：库存变动时需要通知多个系统（ERP、财务系统、看板系统），且未来可能增加新的订阅者。观察者模式定义了一对多的依赖关系，库存服务（Subject/被观察者）维护一个观察者列表，当库存发生变动时自动通知所有注册的观察者。新增订阅系统时只需注册新的观察者，不修改库存服务代码。

**4. "支持对操作指令的撤销和重做"——命令模式（Command Pattern）（2分）**

**理由**：命令模式将操作封装为命令对象（Command），每个命令对象包含execute()和undo()方法。系统维护一个命令历史栈，执行操作时将命令压栈，撤销时弹出栈顶命令并调用其undo()方法，重做时重新执行该命令。这种模式将操作的请求者和执行者解耦，天然支持撤销/重做功能。

### 问题3（8分）

**基于开闭原则的可扩展存储策略框架设计**：

**设计方案**：

**第一步：定义存储策略抽象接口**

```java
public interface StorageStrategy {
    // 判断是否适用于该类型的货物
    boolean supports(GoodsType goodsType);
    // 为货物分配存储区域和货位
    StorageAllocation allocate(Goods goods, List<StorageZone> availableZones);
    // 获取存储环境要求（温度、湿度等）
    StorageRequirement getRequirement();
}
```

**第二步：实现具体策略类**

```java
public class NormalStorageStrategy implements StorageStrategy {
    public boolean supports(GoodsType type) { return type == GoodsType.NORMAL; }
    // 常温货物：就近存放、空间利用率优先
    public StorageAllocation allocate(...) { ... }
}

public class ColdChainStorageStrategy implements StorageStrategy {
    public boolean supports(GoodsType type) { return type == GoodsType.COLD_CHAIN; }
    // 冷链货物：冷藏区存放、温度监控
    public StorageAllocation allocate(...) { ... }
}

public class HazardousStorageStrategy implements StorageStrategy {
    public boolean supports(GoodsType type) { return type == GoodsType.HAZARDOUS; }
    // 危化品：隔离区存放、安全间距、合规存储
    public StorageAllocation allocate(...) { ... }
}
```

**第三步：策略注册与路由**

```java
public class StorageStrategyRegistry {
    private final List<StorageStrategy> strategies = new ArrayList<>();
    
    // 注册新策略（只需实现接口并注册，不修改任何现有代码）
    public void register(StorageStrategy strategy) {
        strategies.add(strategy);
    }
    
    // 根据货物类型自动匹配策略
    public StorageStrategy getStrategy(GoodsType goodsType) {
        return strategies.stream()
            .filter(s -> s.supports(goodsType))
            .findFirst()
            .orElseThrow(() -> new UnsupportedGoodsTypeException(goodsType));
    }
}
```

**如何实现"对扩展开放，对修改关闭"**：

- **对扩展开放**：当新增货物类型（如"鲜活品"）时，只需：①创建新的策略类`FreshStorageStrategy implements StorageStrategy`；②在策略注册表中注册新策略。不需要修改任何现有类的代码。

- **对修改关闭**：`StorageStrategyRegistry`、`InboundService`和现有的策略实现类都不需要修改。路由逻辑通过`supports()`方法自动匹配，无需if-else或switch-case分支。

- **进一步优化**：结合Spring的组件扫描（`@Component`）和依赖注入，新增的策略类可以被自动发现和注册，甚至不需要手动调用`register()`方法，实现真正的即插即用。

**核心设计原则的体现**：
- **开闭原则（OCP）**：新增货物类型不修改现有代码
- **里氏替换原则（LSP）**：所有策略类都可以替换使用
- **依赖倒置原则（DIP）**：上层模块依赖抽象接口，不依赖具体策略实现
- **单一职责原则（SRP）**：每个策略类只负责一种货物类型的存储逻辑
