## 试题一（必答题）：质量属性与架构评估（25分）

### 【背景描述】

某科技公司拟开发一套 **AI Agent 协作平台**（代号"蜂群"），用于支撑多个智能体（Agent）在统一平台上协同完成复杂任务。该平台的核心架构如下：

- **接入层**：提供 RESTful API 和 WebSocket 接口，供外部系统和用户客户端接入，由 API 网关统一管理流量和鉴权。
- **编排层**：包含 Agent 编排引擎和任务调度器。编排引擎负责解析用户意图并将复杂任务拆解为子任务，任务调度器将子任务分配给合适的 Agent 执行。
- **Agent 执行层**：运行多种类型的 Agent（如代码生成 Agent、数据分析 Agent、搜索 Agent 等），每个 Agent 封装了对大语言模型（LLM）的调用逻辑和工具调用能力。Agent 之间通过消息总线进行异步通信。
- **基础设施层**：包括 LLM 推理集群（支持多模型切换）、向量数据库（存储知识库嵌入）、关系型数据库（存储任务状态和用户数据）以及对象存储（存储中间产物和日志）。
- **监控与治理层**：负责 Agent 行为审计、Token 用量计费、调用链追踪和异常告警。

系统设计目标为日均处理 100 万次任务编排请求，峰值 QPS 约 5000，要求 Agent 执行链路的端到端延迟 P99 < 10 秒，系统年可用性 ≥ 99.95%。

| R2 | Agent 编排引擎应支持通过可视化界面拖拽配置工作流，定义 Agent 之间的协作拓扑 |

#### 问题2 参考答案（11分）

**四维度对比表（6分）**：

| 维度 | 策略A：主动冗余 | 策略B：被动冗余 |
|------|-----------------|-----------------|
| **故障恢复时间** | 极短（接近零切换时间），因为多个节点同时在处理，无需等待故障检测和切换 | 较长（需经历故障检测→状态同步→接管启动），通常为秒级到数十秒级 |
| **资源利用率** | **低**，多个节点处理相同请求，冗余计算造成大量资源浪费 | **较高**，备用节点平时处于热备/温备状态，主节点资源被充分利用 |
| **实现复杂度** | 中等，需要实现请求复制和结果仲裁机制（选取最先返回结果） | 中等偏高，需要实现健康检查、状态同步、故障转移和会话恢复机制 |
| **适用场景** | 适用于对延迟极其敏感、计算资源相对廉价的场景（如内存计算、网络路由） | 适用于计算资源昂贵、允许短暂切换延迟的场景 |

**推荐方案及理由（5分）**：

推荐采用**策略B（被动冗余）**，并结合以下优化措施：

1. **LLM 推理的特点决定了不宜采用主动冗余**：
   - GPU 资源昂贵且稀缺，主动冗余要求多个 GPU 节点同时执行相同推理任务，资源浪费极为严重（GPU 算力成本远高于普通 CPU 计算）
   - 单次 LLM 推理耗时较长（通常数百毫秒到数秒），主动冗余中"丢弃多余结果"意味着大量昂贵的 GPU 算力被白白浪费
   - 在流式输出（Streaming）场景下，主动冗余的结果仲裁机制实现难度更大

2. **被动冗余在该场景下可以满足 30 秒切换要求**：
   - 通过设置健康检查间隔（如 5 秒）和快速故障判定机制（如连续 3 次健康检查失败），可在 15～20 秒内完成故障检测
   - 备用节点预加载模型（热备状态），接管后可立即开始处理新请求
   - 对于故障发生时正在处理的请求，通过编排层的重试机制重新提交即可

3. **优化建议**：
   - 采用**N+M 冗余模式**（如 10 个主节点配 2 个备节点），而非 1:1 冗余，进一步提升资源利用率
   - 在 API 网关层实现**请求级故障转移**：当某节点响应超时时，立即将该请求重新路由到其他健康节点
   - 结合 Kubernetes 的 Pod 健康检查和自动重启机制，实现基础设施级的自愈能力

**评分要点**：
- 四维度对比完整且准确（每个维度 1.5 分，共 6 分）
- 明确给出推荐方案（1 分）
- 结合 LLM 推理场景的 GPU 资源昂贵特点进行分析（2 分）
- 给出合理的优化建议或补充措施（2 分）
- 若推荐策略A，只要理由充分合理（如从极致可用性角度论证），也可给满分

## 试题二：微服务架构设计（25分）

### 【背景描述】

某大型电商平台"优购网"创立于 2018 年，初期采用基于 Spring Boot 的单体架构，所有业务模块（用户管理、商品管理、订单处理、库存管理、支付结算、物流跟踪、营销活动、评价系统）都打包在同一个应用中。随着业务快速发展，日活用户突破 2000 万，平台面临以下问题：

- **部署效率低**：每次发版需要整体构建和部署，即使只修改了营销模块的一行代码，也需重新部署整个系统，发版周期从最初的每周一次延长到每两周一次。
- **扩展性不足**：大促期间订单模块的流量是日常的 50 倍，但无法单独对订单模块进行扩容，只能整体扩容，造成资源浪费。

- **故障蔓延**：一次营销活动模块的内存泄漏导致整个系统宕机 4 小时，影响所有业务线。

### 【参考答案】

#### 问题1 参考答案（8分）

**(a) 微服务拆分 4 个核心原则（4分，每个 1 分）**：

1. **单一职责原则**：每个微服务只负责一个明确的业务能力。例如，"优购网"将商品管理（商品信息的CRUD、上下架）和订单处理（下单、取消、退款）拆分为独立服务，各自职责清晰。

2. **高内聚低耦合原则**：服务内部功能高度相关，服务之间依赖最小化。例如，支付服务内聚了支付渠道对接、支付状态管理、对账等功能，对外只暴露"发起支付"和"查询支付状态"接口，不暴露内部实现细节。

3. **业务领域驱动原则**：以业务领域而非技术分层作为拆分依据。例如，不应将所有数据访问拆为一个"数据服务"，而应让订单服务拥有自己的订单数据库，库存服务拥有库存数据库（数据库独立原则）。

4. **演进式拆分原则（渐进式迁移）**：不追求一次性完美拆分，而是优先拆分变化最频繁或痛点最大的模块。例如，"优购网"应优先将营销服务拆出（因为故障蔓延的源头），再逐步拆分订单、库存等核心服务。

**(b) 限界上下文说明（4分）**：

**限界上下文的含义（1分）**：限界上下文是 DDD 中定义模型适用范围的边界。在同一个限界上下文内，所有术语、概念和模型具有明确且一致的含义；不同的限界上下文可以对同一现实事物有不同的建模方式。

**订单服务与库存服务的限界上下文划分（2分）**：
- **订单上下文**：关注"谁在什么时间买了什么、总金额多少、订单状态流转"。订单上下文中的"商品"是一个精简的值对象（包含商品ID、名称、单价、数量），不关心商品的完整属性（如重量、产地等）。
- **库存上下文**：关注"某商品在各仓库有多少可用库存、预留了多少、已出库多少"。库存上下文中的"商品"关联的是 SKU 和仓库信息。
- 两者对"商品"的理解不同，各自维护独立的模型，这正是限界上下文的边界所在。

**上下文间集成方式（1分）**：
- 订单上下文与库存上下文之间采用**防腐层（Anti-Corruption Layer, ACL）**或**发布-订阅事件**模式进行集成。例如，订单服务创建订单后发布"OrderCreated"领域事件到 Kafka，库存服务订阅该事件并执行库存扣减，两个上下文通过事件实现松耦合协作。

**评分要点**：
- 4 个原则各 1 分，需要有原则名称 + "优购网"具体示例
- 限界上下文定义准确 1 分，订单/库存上下文对比 2 分，集成方式 1 分

#### 问题2 参考答案（8分）

**三种方案工作原理（3分，每个 1 分）**：

1. **Saga 模式**：将分布式事务拆解为一系列本地事务，每个本地事务完成后触发下一个事务。如果某一步失败，按反向顺序执行补偿操作（Compensating Transaction）回滚已完成的步骤。分为编排式（Orchestration，由中心协调器驱动）和协同式（Choreography，由事件驱动各服务自行响应）两种实现方式。

2. **TCC 模式（Try-Confirm-Cancel）**：分为三个阶段——Try 阶段预留资源（如冻结库存、冻结余额），Confirm 阶段确认提交（将冻结资源转为实际扣减），Cancel 阶段取消释放（回退 Try 阶段的预留）。每个参与服务都需要实现这三个接口。

3. **本地消息表**：在本地事务中同时写入业务数据和消息记录（同一数据库事务保证原子性），再由后台任务定时扫描消息表，将未发送的消息投递到消息队列，下游服务消费消息完成后续操作。通过消息的可靠投递和幂等消费保证最终一致性。

**四维度对比表（4分）**：

| 维度 | Saga 模式 | TCC 模式 | 本地消息表 |
|------|-----------|----------|-----------|
| **一致性保证** | 最终一致性，依赖补偿机制回滚 | 较强一致性，Try 阶段锁定资源，可接近强一致 | 最终一致性，依赖消息可靠投递和幂等消费 |
| **性能影响** | 较好，无全局锁，各步骤异步执行 | 较差，Try 阶段需锁定资源，整体延迟较高 | 较好，异步处理，不阻塞主流程 |
| **实现复杂度** | 中等，需为每个正向操作实现补偿操作 | **高**，每个服务需实现 Try/Confirm/Cancel 三个接口，空回滚、悬挂等异常处理复杂 | 中等偏低，依赖本地事务和消息队列基础设施 |
| **适用场景** | 长流程、多步骤的业务编排（如订单流转） | 对一致性要求高、业务逻辑可预留资源的场景（如资金交易） | 对实时性要求不高、可接受短暂延迟的异步业务 |

**推荐方案（1分）**：

推荐采用 **Saga 模式（编排式）**，理由如下：

#### 问题3 参考答案（9分）

**(a) Sidecar 代理模式工作原理（3分）**：

Sidecar 代理模式是将一个**轻量级网络代理进程（如 Envoy）**以"边车"形式部署在每个微服务实例旁边（在 Kubernetes 中，与业务容器运行在同一个 Pod 内）。

工作原理：
- **透明流量拦截**：通过 iptables 规则或 eBPF 技术，将业务容器的所有入站（Inbound）和出站（Outbound）TCP 流量自动重定向到 Sidecar 代理，业务代码无需任何修改
- **请求处理**：Sidecar 代理在转发流量时，可以透明地执行服务发现、负载均衡、熔断限流、TLS 加密（mTLS 双向认证）、请求重试、超时控制、流量路由（如按权重灰度分流）等治理功能
- **遥测上报**：Sidecar 自动采集请求的延迟、成功率、流量等指标（Metrics），生成分布式追踪（Tracing）数据，上报给控制平面

**(b) Istio 控制平面与数据平面（3分）**：

**数据平面（Data Plane）（1.5分）**：
- 由部署在每个服务旁的 **Envoy Sidecar 代理**组成
- 职责：拦截和转发服务间的所有网络流量，执行控制平面下发的路由规则、安全策略、限流策略等
- 负责实际的请求转发、mTLS 加解密、遥测数据采集

**控制平面（Control Plane）（1.5分）**：
- Istio 的控制平面核心组件为 **Istiod**（整合了原来的 Pilot、Citadel、Galley 功能）
  - **Pilot 功能**：将高级路由规则（VirtualService、DestinationRule）转换为 Envoy 可理解的配置，通过 xDS 协议下发到各 Sidecar
  - **Citadel 功能**：负责证书签发和轮转，为服务间 mTLS 通信提供身份认证
  - **Galley 功能**：配置验证和分发，确保用户提交的配置正确
- 控制平面不处理实际业务流量，仅管理和配置数据平面的行为

**(c) 服务网格 vs Spring Cloud 对比（3分）**：

**服务网格的主要优势（1.5分）**：
1. **语言无关**：治理能力下沉到基础设施层，业务代码无需引入特定 SDK，"优购网"的 Python 推荐服务和 Java 服务可获得统一的治理能力
2. **治理与业务解耦**：流量管理、安全策略、可观测性等功能由运维团队在网格层配置，开发人员专注业务逻辑，无需在代码中集成熔断、限流等组件
3. **统一治理策略**：所有服务的治理规则在一处统一管理，策略一致性好，变更无需重新部署业务服务
4. **安全能力增强**：Istio 提供开箱即用的 mTLS、RBAC、请求级认证授权，安全实现更加透明

**服务网格的主要劣势（1.5分）**：
1. **性能开销**：每次请求需经过 Sidecar 代理的两次拦截（出站和入站），增加了网络延迟（通常 1～3ms），在极低延迟场景下需关注
2. **运维复杂度高**：需要运维团队掌握 Istio、Envoy、Kubernetes 等技术栈，学习曲线陡峭，故障排查难度增加
3. **资源消耗**：每个服务实例旁都需部署一个 Sidecar 代理，增加了内存和 CPU 开销（每个 Envoy 实例约需 50～100MB 内存）
4. **调试困难**：流量经过 Sidecar 转发后，网络链路变长，问题定位更加复杂

**评分要点**：
- Sidecar 原理需提到"透明拦截"和"iptables/eBPF"机制（3分）
- 控制平面需提到 Istiod 及其核心功能（Pilot/Citadel），数据平面需提到 Envoy（3分）
- 优劣势对比各至少答出 2 点，且与 Spring Cloud 形成对照（3分）

#### 问题1 参考答案（8分）

**(a) 五维度对比（5分，每维度 1 分）**：

| 维度 | MySQL（关系型数据库） | 时序数据库（InfluxDB/TDengine） |
|------|----------------------|-------------------------------|
| **数据模型** | 行列式存储，需要预定义严格的表结构（Schema）；时序数据需手动设计时间戳索引和分区表 | 原生支持时间序列数据模型，以时间戳为核心组织数据；支持 Tag（标签/维度）和 Field（指标值）的灵活组合，天然适配 IoT 数据特征 |

#### 问题3 参考答案（9分）

**(a) 三种分片策略（6分，每种 2 分）**：

**1. Range 分片（范围分片）**：
- **原理**：按照分片键的值范围划分数据。例如，按时间范围分片：2025年1月的数据存在分片A，2月的数据存在分片B，以此类推。
- **优点**：
  - 范围查询效率高（如查询某时间段的数据只需访问少数分片）
  - 支持按时间自动创建新分片，便于数据生命周期管理（旧分片可直接删除）
- **缺点**：
  - 容易出现**写入热点**：当前时间段的分片承载所有新写入，其他分片空闲
  - 需要手动或自动管理分片边界的调整

**2. Hash 分片（哈希分片）**：
- **原理**：对分片键（如设备ID）计算哈希值，根据哈希值对分片数取模，决定数据存储在哪个分片。例如，hash(device_id) % N = 分片编号。
- **优点**：
  - 数据分布均匀，避免写入热点
  - 实现简单，适用于高并发写入场景
- **缺点**：
  - 范围查询效率低（查询某时间段数据需扫描所有分片）
  - 扩容（增加 N 值）时需要**大量数据重新分布（rehash）**

**3. List 分片（列表分片）**：
- **原理**：按照分片键的离散值列表划分数据。例如，按设备类型分片：环境监测站的数据存在分片A，交通设备的数据存在分片B，电表水表的数据存在分片C。
- **优点**：
  - 业务语义清晰，便于针对不同类型的数据采用不同的存储和查询优化策略
  - 同类数据集中存储，相关查询效率高
- **缺点**：
  - 各分片数据量可能严重不均衡（如环境监测站数据量远大于交通设备）
  - 分片键的取值变化（如新增设备类型）需要人工干预分片配置

**(b) 一致性哈希原理（3分）**：

**基本原理（2分）**：

一致性哈希将哈希值空间组织为一个**虚拟环**（Hash Ring），值域为 0 到 2³²-1。

1. **节点映射**：将每个存储节点通过哈希函数映射到环上的一个位置。例如，hash(Node-A) = 位置 P1，hash(Node-B) = 位置 P2，hash(Node-C) = 位置 P3。

2. **数据映射**：将数据的分片键（如设备ID）通过相同的哈希函数映射到环上，然后**沿顺时针方向找到第一个节点**，该节点即为数据的存储位置。

3. **虚拟节点**：为解决节点少时数据分布不均匀的问题，为每个物理节点创建多个虚拟节点（如每个物理节点对应 100～200 个虚拟节点），使数据在环上分布更加均匀。

**如何解决扩容数据迁移问题（1分）**：

在传统 Hash 取模方案中，增加一个节点（N→N+1）会导致几乎所有数据需要重新分配。而在一致性哈希中：

- 新增一个节点 Node-D 时，只需将环上 Node-D 位置到其逆时针方向前一个节点之间的数据从原来负责的节点迁移到 Node-D，**其余数据不受影响**
- 数据迁移量约为 **1/N**（N 为节点总数），远小于传统方案的全量迁移
- 例如，原有 4 个节点时扩容为 5 个节点，仅需迁移约 20% 的数据，而传统取模方案需迁移约 80% 的数据

**评分要点**：
- 三种分片策略各描述原理 + 优缺点得 2 分（共 6 分）
- 一致性哈希需提到"哈希环"和"顺时针查找"（1 分），虚拟节点（0.5 分），扩容仅迁移 1/N 数据（1 分），如未提虚拟节点扣 0.5 分

## 试题四：系统安全架构（25分）

### 【背景描述】

#### 问题1 参考答案（8分）

**(a) 零信任三大核心原则（3分，每个 1 分）**：

1. **永不信任，始终验证（Never Trust, Always Verify）**：不再基于网络位置（内网/外网）授予信任，无论请求来自总行内网、分支机构还是远程 VPN，每次访问都必须经过身份认证和授权验证。在该行场景中，即使运维人员在总行办公室操作数据库，每次操作也需重新验证身份和权限。

2. **最小权限原则（Least Privilege）**：每个用户、设备和应用只授予完成当前任务所需的最小权限，且权限具有时效性。在该行场景中，数据库管理员的权限应细化到表级甚至行级，且仅在工作时间段内有效，非工作时间的批量数据导出请求应被拒绝或需要额外审批。

3. **假设已被攻破（Assume Breach）**：系统架构设计应假设攻击者已经存在于网络内部，因此需要实施微隔离（Micro-segmentation）、持续监控和异常检测。在该行场景中，即使 VPN 账号被盗，攻击者在内网的横向移动也应被微隔离策略阻止，异常行为应被实时检测和告警。

**(b) 零信任五大核心组件（5分，每个 1 分）**：

1. **身份与访问管理（IAM）**：作为零信任架构的基石，负责用户身份的全生命周期管理、多因素认证（MFA）和单点登录（SSO）。在该行中，统一管理 5000 万客户和内部员工的身份，实施"手机银行登录 = 密码 + 短信验证码 + 设备指纹"的多因素认证。

2. **策略引擎 / 策略决策点（PDP, Policy Decision Point）**：基于用户身份、设备状态、请求上下文（时间、位置、行为模式）等多维度信息，实时计算访问决策（允许/拒绝/需要增强认证）。在该行中，策略引擎可判断"运维人员在非工作时间从非常用 IP 发起批量数据导出请求"属于高风险行为，触发拒绝或增强认证。

3. **策略执行点（PEP, Policy Enforcement Point）**：部署在每个受保护资源前的网关或代理，负责拦截所有访问请求并向策略引擎请求决策，按决策结果放行或阻断。在该行中，核心账务系统前的 API 网关即为 PEP，所有请求必须通过该网关。

4. **微隔离（Micro-segmentation）**：将网络划分为细粒度的安全区域，每个应用或服务独立隔离，服务间通信必须经过授权。在该行中，信贷系统、核心账务系统、风控系统之间实施微隔离，即使攻击者突破了信贷系统，也无法直接横向访问核心账务系统。

5. **持续监控与分析（Continuous Monitoring & Analytics）**：实时采集所有访问日志和用户行为数据，通过 UEBA（用户与实体行为分析）等技术检测异常行为。在该行中，可发现运维人员的异常数据导出行为、账号被盗后的异常登录模式等。

**(b) 授权码模式 + OIDC 完整流程（4分）**：

以"用户打开手机银行查询账户余额"为例，完整流程如下：

```
步骤1: 用户打开手机银行 App，点击"查询余额"
       → App 检测到用户未登录，发起授权请求

步骤2: App（Client）将用户浏览器重定向到授权服务器的授权端点
       请求参数包含：
       - response_type=code（请求授权码）
       - client_id=mobile_banking_app
       - redirect_uri=mybank://callback
       - scope=openid profile account:read（openid 表示请求 OIDC 认证）
       - state=随机字符串（防 CSRF）
       - code_challenge=SHA256(code_verifier)（PKCE 参数）
       - code_challenge_method=S256

步骤3: 授权服务器展示登录页面，用户输入用户名、密码，并完成
       短信验证码二次认证（MFA）

步骤4: 授权服务器验证用户身份成功后，生成授权码（Authorization Code），
       通过重定向将授权码返回给 App
       → 重定向到 mybank://callback?code=AUTH_CODE_XYZ&state=随机字符串

步骤5: App（Client）使用授权码向授权服务器的令牌端点发起后端请求
       （此步骤为服务端到服务端调用，不经过浏览器）
       请求参数包含：
       - grant_type=authorization_code
       - code=AUTH_CODE_XYZ
       - redirect_uri=mybank://callback
       - client_id=mobile_banking_app
       - code_verifier=原始随机字符串（PKCE 验证）

步骤6: 授权服务器验证授权码有效性和 PKCE 参数，颁发：
       - Access Token（访问令牌，用于访问资源 API）
       - ID Token（OIDC 身份令牌，JWT 格式，包含用户身份信息：
         sub=用户唯一标识, name=用户姓名, iat=签发时间, exp=过期时间）
       - Refresh Token（刷新令牌，用于 Access Token 过期后重新获取）

步骤7: App 验证 ID Token 的签名和有效期，提取用户身份信息完成登录

步骤8: App 携带 Access Token 调用核心账务系统 API
       → GET /api/accounts/balance
       → Header: Authorization: Bearer ACCESS_TOKEN_XYZ

步骤9: 资源服务器（核心账务系统）验证 Access Token 的有效性
       （通过 JWT 本地验证或调用授权服务器的 Token Introspection 端点），
       确认令牌有效且具有 account:read 权限后，返回账户余额数据

步骤10: App 展示账户余额给用户
```

**(c) PKCE 防范授权码截获攻击（3分）**：

**攻击风险（1分）**：
在移动端 App 场景中，授权码通过自定义 URL Scheme（如 mybank://callback）返回给 App。如果恶意应用注册了相同的 URL Scheme，可能截获授权码，用窃取的授权码向授权服务器换取 Access Token，从而冒充合法 App 访问用户资源。

**PKCE 机制原理（2分）**：

PKCE（RFC 7636）通过引入一对动态验证值来防范此攻击：

1. **授权请求阶段**：
   - Client 生成一个高熵的随机字符串 `code_verifier`（43～128 个字符）
   - 计算其 SHA-256 哈希值 `code_challenge = SHA256(code_verifier)`
   - 将 `code_challenge` 和 `code_challenge_method=S256` 随授权请求发送给授权服务器
   - 授权服务器将 `code_challenge` 与生成的授权码关联存储

2. **令牌交换阶段**：
   - Client 用授权码换取令牌时，需提交原始的 `code_verifier`
   - 授权服务器对收到的 `code_verifier` 计算 SHA-256，与之前存储的 `code_challenge` 比对
   - 只有比对一致，才颁发令牌

3. **为何能防范攻击**：
   - 恶意应用即使截获了授权码，也不知道原始的 `code_verifier`（只在合法 Client 内存中存在，不经过网络传输）
   - 由于 `code_challenge` 是 `code_verifier` 的 SHA-256 哈希值，无法反推出 `code_verifier`
   - 因此恶意应用无法通过授权服务器的 PKCE 验证，攻击失败

**评分要点**：
- 四个角色映射正确（2分）
- 授权码流程需包含：重定向到授权端点→用户认证→返回授权码→用授权码换令牌→访问资源 这 5 个关键步骤（4分），提到 ID Token 和 scope=openid 为加分点
- PKCE 需说明 code_verifier/code_challenge 的生成和验证机制（2分），说明为何能防范截获攻击（1分）

#### 问题1 参考答案（8分）

**(a) Pod（2分）**：

Pod 是 Kubernetes 中**最小的可调度和可管理单元**，是一个或多个容器的逻辑组合。Pod 内的容器共享网络命名空间（同一 IP 地址和端口空间）、存储卷和进程间通信（IPC）命名空间。

**为什么最小调度单元是 Pod 而非容器**：因为某些应用场景需要多个紧密协作的进程共享网络和存储。例如，一个主应用容器需要配合一个日志采集容器（Sidecar）或一个配置初始化容器（Init Container），这些容器之间的通信和数据共享通过 Pod 的共享命名空间实现，比跨容器通信更高效。

**多容器 Pod 示例**：在"云途科技"的直播服务中，一个 Pod 可以包含：①主容器运行直播流媒体服务；②Sidecar 容器运行 Envoy 代理（服务网格数据平面）；③另一个 Sidecar 容器运行 Filebeat 日志采集器，将主容器的日志实时采集并发送到 Elasticsearch。三者共享网络和日志卷，紧密协作。

**(b) Service（2分）**：

Service 解决的核心问题是 **Pod IP 的动态变化**。Pod 是临时性的，随时可能被重建并获得新的 IP 地址。Service 为一组具有相同功能的 Pod 提供一个稳定的**虚拟 IP（Cluster IP）和 DNS 名称**，作为服务发现和负载均衡的入口。

三种类型：
- **ClusterIP**（默认）：仅在集群内部可访问。适用于微服务间的内部通信。例如，用户中心服务通过 ClusterIP 调用支付服务。
- **NodePort**：在每个节点上开放一个固定端口（30000-32767），外部流量通过 `NodeIP:NodePort` 访问。适用于测试环境或简单的外部访问。例如，测试集群中暴露预发布环境的服务供测试人员访问。
- **LoadBalancer**：在 NodePort 基础上，自动创建云厂商的外部负载均衡器。适用于生产环境对外暴露服务。例如，在线教育平台的 API 网关通过 LoadBalancer 类型 Service 接收外部用户请求。

**(c) Deployment（2分）**：

Deployment 是用于管理**无状态应用 Pod 副本集**的控制器，负责：声明期望的 Pod 副本数量、Pod 模板（镜像版本、资源限制等），并确保实际运行状态与期望状态一致（自愈能力：Pod 异常退出后自动重建）。

**滚动更新（Rolling Update）工作原理**：
1. Deployment 更新 Pod 模板（如将镜像版本从 v1 升级到 v2）
2. Deployment 控制器创建一个新的 ReplicaSet（RS-v2），逐步增加其 Pod 副本数
3. 同时逐步缩减旧 ReplicaSet（RS-v1）的 Pod 副本数
4. 更新过程受 `maxSurge`（最大超出副本数）和 `maxUnavailable`（最大不可用副本数）参数控制，确保更新期间始终有足够的 Pod 处理请求
5. 如果新版本 Pod 健康检查失败，可自动回滚到旧版本

例如，"云途科技"的录播课程服务有 10 个 Pod 副本，配置 maxSurge=2、maxUnavailable=1，滚动更新时最多有 12 个 Pod 运行（10+2），最少有 9 个 Pod 可用（10-1），实现零停机升级。

**(d) Ingress（2分）**：

**与 Service 的关系**：Ingress 是 Kubernetes 中管理**集群外部 HTTP/HTTPS 流量路由到集群内部 Service** 的资源对象。Service 提供了 L4（TCP/UDP）级别的负载均衡，Ingress 在此基础上提供 **L7（HTTP）级别的路由**能力。

**为什么需要 Ingress**：如果没有 Ingress，每个需要对外暴露的服务都需要创建独立的 LoadBalancer，造成成本浪费和管理困难。Ingress 通过统一的入口点实现基于域名和路径的路由规则，一个 Ingress Controller（如 Nginx Ingress Controller）即可管理所有外部流量。

**路由规则示例**：

```
规则1: www.yuntu.com/live/*    → 直播服务 Service (live-service:8080)
规则2: www.yuntu.com/course/*  → 录播课程 Service (course-service:8080)
规则3: www.yuntu.com/api/*     → API 网关 Service (api-gateway:8080)
规则4: admin.yuntu.com/*       → 管理后台 Service (admin-service:8080)
规则5: 默认后端               → 404 页面 Service
```

通过 TLS 终止功能，Ingress 还可统一管理 HTTPS 证书。

**评分要点**：
- 每个概念解释准确 + 场景举例恰当得 2 分
- Pod 需提到"共享网络和存储"以及多容器场景
- Service 需区分三种类型及适用场景
- Deployment 需说明滚动更新中新旧 ReplicaSet 的交替机制
- Ingress 需说明与 Service 的关系及 L7 路由能力

#### 问题2 参考答案（8分）

**(a) Metrics（指标监控）（2分）**：

**定义**：Metrics 是对系统运行状态的**数值化度量**，通常以时间序列形式存储，用于反映系统健康状况和性能趋势。

**关键指标类型**：
- **基础设施指标**：CPU/内存/磁盘/网络利用率、Pod 数量、节点状态
- **应用性能指标（RED）**：Rate（请求速率）、Errors（错误率）、Duration（延迟分布）
- **业务指标**：在线用户数、直播观看人数、支付成功率
- **指标类型**：Counter（计数器，如请求总数）、Gauge（仪表盘，如当前内存使用量）、Histogram（直方图，如延迟分布）、Summary（摘要，如延迟百分位数）

**工具选型**：**Prometheus**（采集和存储）+ **Grafana**（可视化和告警）。Prometheus 通过 Pull 模式定期抓取各服务暴露的 /metrics 端点，配合 Alertmanager 实现告警通知。在 Kubernetes 环境中，使用 kube-state-metrics 和 node-exporter 采集集群和节点指标。

**(b) Logging（日志管理）（2分）**：

**定义**：Logging 是对系统运行过程中**离散事件的文本记录**，包含时间戳、日志级别、上下文信息和详细消息，用于事后分析和故障排查。

**云原生环境的日志采集方案**：
- **DaemonSet 模式**：在每个 Kubernetes 节点上部署日志采集 Agent（如 Fluentd/Fluent Bit）作为 DaemonSet，采集节点上所有 Pod 的标准输出日志（stdout/stderr）以及容器日志文件
- **Sidecar 模式**：在需要特殊处理的 Pod 内部署日志采集 Sidecar 容器（适用于需要多日志流处理或特殊格式转换的场景）

**工具选型**：**EFK/ELK Stack** —— Fluent Bit（轻量级采集）→ Elasticsearch（存储和索引）→ Kibana（查询和可视化）。或者使用更现代的方案：**Grafana Loki**（轻量级日志聚合，不索引日志内容，仅索引标签，与 Grafana 深度集成，存储成本更低）。

**(c) Tracing（分布式追踪）（2分）**：

**定义**：Tracing 是对一个请求在分布式系统中**完整调用链路的跟踪记录**，记录请求经过的每个服务、每次调用的耗时和状态，用于分析性能瓶颈和故障根因。

**核心概念**：
- **Trace**：代表一个完整请求的生命周期，由唯一的 Trace ID 标识。例如，一次"用户进入直播课堂"的完整调用链就是一个 Trace。
- **Span**：Trace 中的一个操作单元，代表调用链中的一个步骤（如一次 RPC 调用、一次数据库查询）。每个 Span 记录操作名称、开始/结束时间、状态码等。Span 之间有父子关系，构成树状结构。
- **Context Propagation（上下文传播）**：在服务间调用时，自动传递 Trace ID 和 Span ID（通常通过 HTTP Header 如 `traceparent`，遵循 W3C Trace Context 标准），确保跨服务的调用链能被关联为同一个 Trace。

**工具选型**：**OpenTelemetry**（统一的遥测数据采集标准和 SDK，支持 Metrics/Logging/Tracing 三合一）+ **Jaeger** 或 **Grafana Tempo**（Trace 存储和可视化）。OpenTelemetry 已成为 CNCF 云原生可观测性的事实标准，推荐作为统一的遥测数据采集层。

**(d) 三者协同排障流程（2分）**：

以直播课堂卡顿问题为例，排障流程如下：

**Step 1 — Metrics 发现异常**：运维人员在 Grafana 监控大屏上发现直播服务的 P99 延迟从正常的 200ms 飙升到 5 秒，同时发现转码服务的某个 Pod 内存使用率持续增长，接近 OOM 阈值。Metrics 告警触发，初步定位异常范围。

**Step 2 — Tracing 定位瓶颈**：运维人员打开 Jaeger/Tempo 查询延迟超过 5 秒的 Trace，在调用链视图中发现：API 网关 → 直播服务 → 转码服务 这条链路中，转码服务的 Span 耗时占总延迟的 90%。进一步查看发现，是转码服务的特定 Pod（Pod-xyz）响应异常，精确定位到问题 Pod。

**Step 3 — Logging 确认根因**：在 Kibana/Loki 中查询 Pod-xyz 的日志，发现大量 "java.lang.OutOfMemoryError: Java heap space" 错误日志，以及 Kubernetes 事件日志中记录了该 Pod 多次被 OOM Killed 后自动重启。最终确认根因：转码服务存在内存泄漏，特定场景下（高码率视频转码）导致 JVM 堆内存耗尽。

**三者协同价值**：Metrics 回答"系统哪里出了问题"（What），Tracing 回答"请求的瓶颈在哪个服务和调用"（Where），Logging 回答"具体发生了什么"（Why）。三者结合形成完整的故障定位闭环。

**评分要点**：
- 三大支柱各 2 分，需包含定义 + 工具选型
- 协同排障需展示 Metrics→Tracing→Logging 的递进关系（2分）
- 提到 OpenTelemetry 统一采集标准为加分项

#### 问题3 参考答案（9分）

**(a) Serverless 核心特征（3分，答出 4 点得满分）**：

1. **无需管理服务器**：开发者只需编写和上传业务函数代码，无需关心底层服务器的部署、配置、补丁更新和扩缩容，由平台（如 AWS Lambda、阿里云函数计算、Knative）自动管理。

2. **事件驱动执行**：函数由特定事件触发执行（如 HTTP 请求、消息队列消息、定时触发器、对象存储事件等），非事件触发时函数不运行。

3. **自动弹性伸缩**：平台根据请求量自动调整函数实例数量，从 0 扩展到数千个并发实例，并在无请求时自动缩减到 0。支持"缩容到零"（Scale to Zero），真正实现按需使用。

4. **按使用量计费**：只对函数实际执行的时间和消耗的资源（如内存、CPU）付费，精确到毫秒级。无请求时不产生费用，相较于常驻容器/虚拟机大幅降低成本。

5. **无状态执行**：函数实例是临时性的、无状态的，每次调用之间不保留内存或本地存储中的数据，状态需外存到数据库、缓存或对象存储中。

**(b) 适合与不适合的场景（3分）**：

**适合 Serverless 的 3 个场景（1.5分）**：

1. **短期活动和秒杀页面**：产品团队的限时秒杀活动持续 2 天，活动期间流量突增但活动结束后流量归零。使用 Serverless 函数处理秒杀请求，自动伸缩应对流量峰值，活动结束后缩容到零，不产生费用。避免了为短期活动走完整容器化部署流程。

2. **异步事件处理（文件/视频处理触发器）**：用户上传课程视频后触发 Serverless 函数进行视频转码、生成缩略图、提取字幕等。这类任务是事件驱动、处理时间有限且各任务相互独立，非常适合 Serverless。

3. **定时任务和后台批处理**：每天凌晨生成学习报告、定期清理过期数据、发送课程提醒通知等。使用定时触发器调用 Serverless 函数，按执行次数计费，比维护 CronJob Pod 更经济。

**不适合 Serverless 的 3 个场景（1.5分）**：

1. **直播流媒体服务**：直播课堂需要长连接（WebSocket）、持续运行、低延迟响应，单次连接可能持续数小时。Serverless 函数有执行时长限制（通常最长 15 分钟），且冷启动延迟不可接受，不适合长连接场景。

2. **核心交易服务（支付系统）**：支付处理对延迟极其敏感（要求 P99 < 100ms），且需要维持数据库连接池等有状态资源。Serverless 的冷启动延迟和连接池管理困难使其不适合此类核心交易场景。

3. **高频持续调用的服务（用户中心 API）**：用户中心服务被所有其他服务高频调用，QPS 持续稳定在数千，函数实例基本不会缩容到零。此时 Serverless 的按调用计费反而比常驻容器更贵，且没有"按需伸缩"的成本优势。

**(c) 冷启动问题及缓解手段（3分）**：

**冷启动原因（1分）**：

当 Serverless 函数在一段时间内没有被调用后，平台会回收其运行环境（容器实例被销毁）。当新请求到达时，平台需要重新执行以下步骤：
1. 分配计算资源（CPU、内存）
2. 下载并解压函数代码包
3. 启动容器/MicroVM 运行时
4. 初始化语言运行时（如 JVM 启动、Python 解释器加载）
5. 执行函数的初始化代码（如加载配置、建立数据库连接）

以上步骤的耗时即为**冷启动延迟**，可能从数百毫秒到数秒不等（Java/C# 等 JIT 编译语言尤为严重）。

**缓解冷启动的技术手段（2分，答出 3 种得满分）**：

1. **预留实例 / 预热（Provisioned Concurrency）**：预先配置一定数量的"常驻"函数实例，保持运行时始终就绪。例如，为"云途科技"的课程推荐函数预留 10 个实例，确保前 10 个并发请求零冷启动。缺点是会产生常驻费用。

2. **定时预热（Scheduled Warm-up）**：通过定时触发器（如每 5 分钟）发送一个"空请求"调用函数，保持实例处于"温热"状态，避免被平台回收。这是一种低成本的"保活"策略，适用于访问模式可预测的场景。

3. **减小代码包体积和依赖**：精简函数代码包，减少依赖库数量和体积，加快代码下载和加载速度。例如，使用 Python 的轻量级框架代替 Django 等全功能框架；使用 Tree Shaking 去除未使用的 JavaScript 模块。

4. **选择轻量级语言运行时**：Node.js、Python、Go 的冷启动时间通常远低于 Java、C#。如果冷启动敏感，优先选择轻量级语言。Go 语言由于是编译型语言且无需运行时虚拟机，冷启动通常在 100ms 以内。

5. **SnapStart / 快照恢复技术**：部分平台（如 AWS Lambda SnapStart）在函数初始化完成后对内存状态进行快照（Snapshot），后续冷启动时直接从快照恢复而不是重新初始化，可将 Java 函数的冷启动从 5 秒降低到 200ms 以内。

**评分要点**：
- Serverless 特征答出 4 点得 3 分（每点 0.75 分）
- 适合/不适合场景各 3 个，需结合"云途科技"场景（各 1.5 分）
- 冷启动原因需说明"容器实例被销毁后重新初始化"的过程（1 分）
- 缓解手段答出 3 种得 2 分（每种约 0.67 分），需简要说明原理
