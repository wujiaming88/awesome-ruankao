## 一、质量属性场景分析（⭐必答题高频，连续4次考）

### 答题框架

#### 题型1：需求→质量属性映射（填表题，12-14分）

**套路**：逐条分析需求描述中的关键词，映射到质量属性

| 关键词特征 | 对应质量属性 | 真题示例 |
|-----------|------------|---------|
| "X秒/分钟内完成""响应时间""吞吐量" | **性能** | "1分钟内分配资源""10秒内给出错误提示""3秒内返回" |
| "故障时切换""不间断服务""自动恢复""X分钟内检测/恢复" | **可用性** | "自动切换到备份服务器""1分钟内检测故障""1小时内恢复" |
| "防止未经授权访问""加密""安全" | **安全性** | "确保用户数据安全""防止未经授权的访问" |
| "新功能X天内完成""扩容""添加模块" | **可修改性** | "一周内完成新功能开发""扩容5天内完成" |
| "多语言""快捷键""不同分辨率""用户体验" | **易用性** | "适配不同分辨率""支持多语言界面""提供快捷键" |
| "自动化测试框架""远程测试" | **可测试性** | "支持远程自动化测试框架" |
| "多种终端访问""第三方导入""系统对接" | **互操作性** | "支持多种终端""从第三方数据源导入" |
| "根据类型调用""支持搜索""数据导入导出" | **功能性** | "根据任务类型调用模型""支持搜索历史任务" |

> ⚠️ **易错点**：
> - "数据导入导出在1分钟内结束" → **性能**（有时间约束就是性能）
> - "支持多语言接口+搜索关联" → **功能性**（功能需求不是质量属性场景）
> - "扩容5天内完成" → **可修改性**（不是性能）

#### 题型2：质量属性场景六要素填空（11-13分）

**六要素万能模板（必背！）**：

| 要素 | 英文 | 一句话定义 | 答题关键 |
|------|------|-----------|---------|
| **刺激源** | Source | 谁/什么发出了刺激 | 人/系统/外部事件 |
| **刺激** | Stimulus | 发生了什么事 | 具体的事件描述 |
| **环境** | Environment | 什么条件下发生的 | 正常运行/过载/故障模式 |
| **制品** | Artifact | 系统哪部分被影响 | 具体模块/子系统/整体系统 |
| **响应** | Response | 系统怎么反应的 | 具体的处理动作 |
| **响应度量** | Response Measure | 怎么衡量响应好不好 | 可量化的指标 |

**可用性场景答题模板**：
```
刺激源：[内部组件/外部网络/硬件设备]
刺激：[服务器硬件故障/网络中断/进程崩溃]
环境：[正常运行时/高峰期/降级模式]
制品：[主服务节点/数据库服务器/网络通信模块]
响应：[检测到故障→自动切换到备用节点→记录日志→通知运维]
响应度量：[在XX秒内完成故障转移/可用性≥99.9%]
```

**性能场景答题模板**：
```
刺激源：[并发用户/批量任务/外部系统请求]
刺激：[大量并发请求到达/批量数据处理请求]
环境：[正常负载/高并发峰值]
制品：[Web服务器/数据处理模块/API网关]
响应：[系统在规定时间内处理完请求并返回结果]
响应度量：[95%请求响应时间≤Xms/吞吐量≥X TPS]
```

#### 题型3：可用性战术对比（Ping/Echo vs 心跳，2024下考过）

| 维度 | Ping/Echo | 心跳（Heartbeat） |
|------|----------|-----------------|
| 发起方 | **监控组件**主动发送ping | **被监控组件**主动发送 |
| 通信模式 | 请求-响应（短连接） | 周期性单向发送（长连接） |
| 资源消耗 | 较高（每次发送+等待响应） | **较低**（只发简短心跳消息） |
| 网络负载 | 较高（频繁请求+响应） | **较低**（心跳包很短且频率可控） |
| 故障检测 | 受网络延迟影响 | **更快**（无心跳立即察觉） |

**结论句（直接抄）**：心跳策略更优，因为被监控组件主动发送简短消息表明运行状态，无需监控组件频繁发送请求并等待回应，减少了资源消耗和网络负载，故障检测效率更高。

### 关键得分点
1. 需求映射题：**不要遗漏非功能需求**，每条都要判断
2. 六要素题：**六个要素缺一不可**，响应度量必须有量化指标
3. 战术对比题：**必须说明原因**，不能只写结论

### 常见陷阱
- "功能性需求"不是质量属性（如"支持多语言接口并提供搜索关联"是功能性）
- "可扩展性"属于可修改性的子属性，不要单独列
- "X天/周内完成"是可修改性，不是性能（性能指运行时响应）

### 真题示例
- 2024上必答：微服务质量属性效用树+六要素
- 2024下必答：LLM平台10条需求→质量属性映射+Ping/Echo vs 心跳
- 2025上必答：大模型训练平台12条需求→质量属性+解释器风格
- 2025下必答：AES-256加密平台六要素+安全性与性能权衡

## 二、架构风格辨析

### 答题框架

**五大类架构风格速查**：

| 类别 | 代表风格 | 核心特征 | 适用场景 |
|------|---------|---------|---------|
| **数据流** | 管道-过滤器 | 数据在管道中传输，过滤器独立处理 | 数据转换流水线、编译器 |
| **调用/返回** | 主程序-子程序、层次式 | 显式调用，等待返回 | 传统应用、分层系统 |
| **独立构件** | 事件驱动、隐式调用 | 松耦合，事件触发 | GUI、消息系统 |
| **仓库** | 数据库、黑板 | 中央数据存储，共享状态 | 决策支持、AI求解 |
| **虚拟机** | 解释器、规则系统 | 自定义执行引擎 | 脚本执行、动态规则 |

**解释器风格四组件（2025上考过）**：
1. 解释器引擎
2. 被解释的程序
3. 解释器引擎的内部状态
4. 程序执行的当前状态

> **答题套路**：先说"该系统适合用XX风格"，然后给出2-3条理由（从需求特征匹配到风格特征）

### 常见对比框架

| 对比维度 | 管道-过滤器 | 事件驱动 | 解释器 | 黑板 |
|---------|-----------|---------|--------|------|
| 数据流向 | 单向流动 | 事件触发 | 逐步解释 | 共享存储 |
| 耦合度 | 低 | 最低 | 中 | 中 |
| 可并行 | ✅ 可以 | ✅ 可以 | ❌ 顺序 | ✅ 可以 |
| 实时性 | 中 | 高 | 低 | 中 |
| 适用 | 数据处理 | 异步通信 | 动态执行 | 不确定求解 |

- 2025上必答：在线大模型训练平台 → 解释器风格（动态解析Python代码）

## 三、Redis/缓存架构（⭐案例选做高频，4/5次考）

### 3.1 RDB vs AOF 持久化对比（填表模板）

| 维度 | RDB | AOF |
|------|-----|-----|
| 原理 | 定时生成内存数据**二进制快照** | 以**日志追加**方式记录每个写命令 |
| 文件格式 | 二进制（dump.rdb） | 文本命令（appendonly.aof） |
| 恢复速度 | **快**（直接加载二进制） | 慢（重放所有命令） |
| 文件大小 | **小**（压缩二进制） | 大（文本命令） |
| 数据安全性 | 低（可能丢失分钟级数据） | **高**（最多丢1秒，everysec策略） |
| 性能影响 | BGSAVE fork子进程，大内存时短暂阻塞 | 持续追加写入 |
| 适用场景 | 定期备份、灾难恢复 | 数据安全要求高的场景 |

**得分句**：生产环境通常**同时开启**RDB和AOF，利用RDB快速恢复，AOF保证数据完整性。

### 3.2 Redis主从复制流程（填空模板）

**全量复制流程**：
```
1. 从节点发送 PSYNC 命令
2. 主节点执行 BGSAVE，生成 RDB 快照文件
3. 主节点将生成期间的新写命令缓存到 复制缓冲区
4. BGSAVE完成 → 发送RDB文件给从节点
5. 从节点加载RDB文件恢复数据
6. 主节点发送复制缓冲区中的增量命令
7. 从节点执行增量命令，达到一致
```

**增量复制流程**：
```
1. 从节点断线重连 → 发送 PSYNC <runid> <offset>
2. 主节点检查 offset 是否在 repl_backlog 范围内
3. 在范围内 → 发送增量数据
4. 超出范围 → 退化为全量复制
```

### 3.3 缓存击穿解决方案对比（2025下考过）

| 维度 | 互斥锁方案（物理过期） | 逻辑过期+异步更新方案 |
|------|---------------------|---------------------|
| 一致性 | ✅ **强一致**（锁释放后都是最新数据） | ⚠️ 最终一致（短暂返回旧数据） |
| 可用性 | ⚠️ 较低（锁等待时部分请求阻塞） | ✅ **高可用**（任何时候都返回数据） |
| 性能 | ⚠️ 不稳定（锁竞争时延迟大） | ✅ **稳定快速**（无锁等待） |
| 复杂度 | ★★★ 较低 | ★★★★ 较高 |
| 适用场景 | 强一致性要求、并发量中等 | **高并发、允许短暂数据不一致** |

**互斥锁方案流程**：查Redis → 未命中 → SETNX获取锁 → 成功则查DB+写缓存+释放锁 → 失败则休眠重试

**逻辑过期方案流程**：查Redis → 检查逻辑过期时间 → 未过期直接返回 → 已过期则返回旧数据 + 异步线程更新缓存

### 3.4 Cache-Aside策略（2024下考过）

**读流程**：
```
① 先查缓存
② 命中 → 直接返回
③ 未命中 → 查数据库 → 写入缓存 → 返回
```

**写流程**：
```
① 更新数据库
② 删除缓存（不是更新缓存！）
```

**并发不一致原因**：更新DB和删除Cache不是原子操作，并发读写交错执行

**解决方案（至少写3种）**：
1. 延时双删（更新DB→删除Cache→延时再删一次）
2. 消息队列异步同步（MQ通知删除缓存）
3. 订阅Binlog失效（Canal监听DB变更→删除缓存）
4. 加锁保证原子性
5. 设置缓存过期时间兜底

### 3.5 分布式锁（2024上考过）

**三种实现**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| Redis SETNX | SET key val NX PX timeout | 性能高 | 主从切换可能丢锁 |
| ZooKeeper临时顺序节点 | 创建临时节点，编号最小者获锁 | 可靠性高，锁自动释放 | 性能不如Redis |
| etcd Lease+Watch | 基于租约+监听机制 | 强一致性（Raft协议） | 部署运维复杂 |

**Redis分布式锁死锁场景**：
- 业务时间超过锁超时 → 锁自动释放 → 其他线程获锁 → 原线程DEL误删他人锁
- 解决：锁value设唯一ID，释放时Lua脚本验证身份

## 五、大数据架构（Lambda/Kappa）

### Lambda架构三层填空模板（2023下必答题考过）

| 层次 | 英文 | 功能 | 技术栈 |
|------|------|------|--------|
| **批处理层** | Batch Layer | 全量历史数据离线处理，保证**准确性** | MapReduce/Spark/HDFS/HBase |
| **速度层(加速层)** | Speed Layer | 增量实时数据处理，补偿批处理**延迟** | Spark Streaming/Flink/Kafka |
| **服务层** | Serving Layer | **合并**批处理视图和实时视图，对外查询 | MemSQL/HBase/Elasticsearch |

### Lambda vs Kappa 对比

| 维度 | Lambda架构 | Kappa架构 |
|------|-----------|----------|
| 数据处理方式 | 同时支持**批处理和流处理**两条链路 | 仅通过**流处理**一条链路 |
| 实时性 | 速度层低延迟+批处理层高精度 | 统一流处理，实时性好 |
| 复杂度 | **高**（两套代码） | **低**（一套代码） |
| 数据一致性 | 批处理保证最终一致性 | 通过**重放(replay)**保证一致性 |
| 适用场景 | 同时需要历史+实时处理 | 实时性要求高、历史重计算需求少 |

## 七、数据库设计

### 范式分析口诀

| 范式 | 消除什么 | 判断方法 |
|------|---------|---------|
| 1NF → 2NF | 消除**部分函数依赖** | 非主属性是否完全依赖于整个主键 |
| 2NF → 3NF | 消除**传递函数依赖** | 非主属性之间是否有依赖 |
| 3NF → BCNF | 消除**主属性对码的部分/传递依赖** | 每个决定因子都是候选码 |
| BCNF → 4NF | 消除**多值依赖** | 是否存在非平凡多值依赖 |

### SQL常考语法

```sql
-- GROUP BY + HAVING（必考）
SELECT dept, COUNT(*) as cnt
FROM employee
GROUP BY dept
HAVING COUNT(*) > 5;  -- HAVING用于分组后过滤

-- 自然连接 vs 笛卡尔积
-- 自然连接：去重复列，保留匹配行
-- 笛卡尔积：m×n行
```

> ⚠️ **关系运算不含删除操作**；操作对象和结果都是**集合**

## 八、DDD领域驱动设计（2025下新考点）

### 限界上下文答题模板

```
1. 识别限界上下文（按业务域划分）
2. 定义每个上下文的核心职责和领域模型
3. 确定上下文间的映射关系：
   - 防腐层(ACL)：下游通过适配层访问上游，防止模型污染
   - 发布-订阅：通过领域事件异步通信
```
