# 质量属性分析专题

> 系统架构设计师案例分析高频考点——质量属性分析详解。涵盖六大质量属性的场景六要素、常见战术及历年真题考法。

---

## 一、质量属性概述

软件架构的质量属性是指系统在功能需求之外，为满足利益相关者需求而必须具备的特性。在系统架构设计师考试中，质量属性分析是案例分析的核心考点之一。

### 六大质量属性

| 质量属性 | 英文 | 核心关注点 |
|----------|------|------------|
| **性能** | Performance | 系统响应时间、吞吐量、资源利用率 |
| **可用性** | Availability | 系统正常运行时间比例，故障恢复能力 |
| **安全性** | Security | 抵抗未授权访问的能力，数据保密完整 |
| **可修改性** | Modifiability | 系统变更的难易程度和成本 |
| **可测试性** | Testability | 软件发现故障并隔离的难易程度 |
| **易用性** | Usability | 用户使用系统的难易程度 |

### 质量属性场景六要素

每个质量属性场景包含以下六个要素：

```
┌─────────────────────────────────────────────────┐
│  刺激源（Source）    →  产生刺激的实体              │
│  刺激（Stimulus）    →  到达系统时需要考虑的条件      │
│  环境（Environment） →  刺激发生时系统的当前状态      │
│  制品（Artifact）    →  被刺激的系统部分或组件        │
│  响应（Response）    →  刺激到达后系统的处理活动      │
│  响应度量（Measure） →  对响应的度量标准             │
└─────────────────────────────────────────────────┘
```

## 二、六大质量属性详解

### 1. 性能（Performance）

#### 定义
性能是指系统在特定条件下，对事件的响应速度、处理能力和资源使用效率。

#### 场景六要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **刺激源** | 用户、外部系统、定时任务 | 1000个并发用户 |
| **刺激** | 事件到达（周期性/随机/偶发） | 同时发起查询请求 |
| **环境** | 正常模式/过载模式 | 正常运行模式下 |
| **制品** | 被刺激的系统或服务 | 订单查询服务 |
| **响应** | 处理事件、改变服务级别 | 系统返回查询结果 |
| **响应度量** | 延迟、吞吐量、丢失率 | 响应时间不超过2秒 |

#### 常见战术

**（1）资源需求——减少资源消耗**
- **提高计算效率**：优化算法复杂度
- **减少计算开销**：引入缓存（减少重复计算）
- **管理事件到达率**：采样、限流
- **控制事件频率**：合并请求、批处理

**（2）资源管理——合理分配资源**
- **引入并发**：多线程/多进程并行处理
- **维护数据或计算的多个副本**：数据缓存、CDN
- **增加可用资源**：垂直扩展（增加CPU/内存）、水平扩展（增加服务器）

**（3）资源仲裁——调度策略**
- **FIFO（先进先出）**
- **固定优先级调度**
- **动态优先级调度**（如最早截止时间优先 EDF）
- **轮转调度**

#### 典型场景示例

```
场景：电商平台双十一大促时，系统需在1秒内处理用户的商品搜索请求。

刺激源：1万个并发用户
刺激：同时发起商品搜索请求
环境：峰值流量（过载模式）
制品：商品搜索服务
响应：返回搜索结果
响应度量：99%的请求在1秒内响应，吞吐量≥10000 TPS
```

### 2. 可用性（Availability）

#### 定义
可用性是指系统在需要时处于正常运行状态的能力，通常用系统正常运行时间占总时间的比例来衡量。

#### 可用性等级

| 等级 | 可用性 | 年停机时间 |
|------|--------|------------|
| 基本可用 | 99% | 3.65天 |
| 较高可用 | 99.9% | 8.76小时 |
| 高可用 | 99.99% | 52.6分钟 |
| 极高可用 | 99.999% | 5.26分钟 |

#### 场景六要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **刺激源** | 系统内部/外部故障源 | 服务器硬件故障 |
| **刺激** | 故障、遗漏、时序错误、响应错误 | 主数据库宕机 |
| **环境** | 正常运行/降级运行 | 正常运行模式下 |
| **制品** | 被要求可用的资源 | 订单处理系统 |
| **响应** | 记录故障、通知、恢复 | 自动切换到备库 |
| **响应度量** | 不可用时间、检测修复时间 | 10秒内完成故障切换 |

#### 常见战术

**（1）故障检测**
- **心跳监测（Heartbeat）**：周期性发送心跳包检测组件存活
- **Ping/Echo**：主动探测组件是否响应
- **异常检测**：监控异常日志和指标

**（2）故障恢复——准备与修复**
- **主动冗余（热备）**：多节点同时运行，故障时无缝切换
- **被动冗余（温备）**：备用节点保持数据同步，需短暂切换
- **冷备（Cold Spare）**：备用节点定期更新，恢复时间较长
- **回滚（Rollback）**：恢复到故障前的正确状态
- **重试（Retry）**：发生故障后自动重试

**（3）故障预防**
- **从服务中移除**：维护前将节点从集群中摘除
- **事务（Transaction）**：保证操作的原子性
- **进程监控**：监控进程状态，异常时自动重启
- **异常预防**：输入校验、参数检查

#### 典型场景示例

```
场景：金融交易系统要求全年运行，发生故障后10秒内自动恢复。

刺激源：服务器硬件故障
刺激：应用服务器进程崩溃
环境：正常运行模式
制品：交易处理系统
响应：故障检测、自动切换到备用服务器、恢复交易处理
响应度量：故障切换时间<10秒，年可用性≥99.99%
```

### 3. 安全性（Security）

#### 定义
安全性是指系统在合法使用的同时，抵抗未授权使用的能力。安全性包括机密性、完整性、不可否认性、可控性和可审计性。

#### 安全性要素

| 要素 | 说明 |
|------|------|
| **机密性** | 数据只对授权用户可见 |
| **完整性** | 数据未被未授权篡改 |
| **不可否认性** | 操作不可抵赖 |
| **可控性** | 对信息传播和内容的控制 |
| **可审计性** | 操作可追溯 |

#### 场景六要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **刺激源** | 已识别/未识别的攻击者、内部人员 | 外部黑客 |
| **刺激** | 未授权访问、篡改、拒绝服务 | 尝试SQL注入攻击 |
| **环境** | 联网/断网、正常/被攻击 | 系统正常运行中 |
| **制品** | 系统服务或数据 | 用户个人信息数据库 |
| **响应** | 认证、授权、记录、阻止 | 拦截攻击并记录日志 |
| **响应度量** | 攻击被成功阻止的比例 | 拦截率≥99.9%，0数据泄露 |

#### 常见战术

**（1）抵抗攻击**
- **身份认证（Authenticate）**：用户名密码、多因子认证（MFA）、生物识别
- **授权（Authorize）**：RBAC（基于角色）、ABAC（基于属性）
- **维护数据机密性**：数据加密（AES、RSA）、传输加密（SSL/TLS）
- **维护数据完整性**：数字签名、消息摘要（MD5、SHA）、校验和
- **限制暴露**：最小权限原则、网络分区
- **限制访问**：防火墙、IP白名单、WAF

**（2）检测攻击**
- **入侵检测系统（IDS）**
- **异常行为分析**
- **审计日志分析**

**（3）从攻击中恢复**
- **恢复状态**：备份恢复、事务回滚
- **攻击者识别**：溯源分析、IP封禁

### 4. 可修改性（Modifiability）

#### 定义
可修改性是指系统能够快速、低成本地进行变更的能力。变更包括功能增加、缺陷修复、适应新环境等。

#### 场景六要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **刺激源** | 开发人员、用户、管理员 | 产品经理 |
| **刺激** | 增加/删除/修改功能、适应新平台 | 要求新增微信支付功能 |
| **环境** | 设计时/编译时/运行时 | 系统运行时 |
| **制品** | 需要修改的系统元素 | 支付模块 |
| **响应** | 修改、测试、部署 | 不影响其他功能的情况下完成修改 |
| **响应度量** | 修改所需时间和成本 | 2人日内完成开发和测试 |

#### 常见战术

**（1）局部化变更——减小修改范围**
- **维持语义一致性**：模块内高内聚
- **抽象化**：面向接口编程
- **信息隐藏**：封装实现细节
- **分离关注点**：使用MVC等分离不同关注点
- **预期变更**：将可能变化的部分独立封装

**（2）防止连锁反应——减少依赖**
- **使用中介者**：引入中间层（如适配器、消息队列）
- **限制通信路径**：减少模块间直接调用
- **使用接口**：模块间通过接口通信

**（3）延迟绑定**
- **配置文件**：运行时修改配置
- **多态**：运行时绑定实现
- **插件机制**：运行时加载新组件
- **依赖注入**：运行时注入依赖

---

### 5. 可测试性（Testability）

#### 定义
可测试性是指通过测试揭示软件缺陷的难易程度，它影响发现并定位软件故障所需的成本。

#### 场景六要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **刺激源** | 测试人员、开发人员 | 集成测试工程师 |
| **刺激** | 完成一段代码/集成阶段 | 新功能开发完成 |
| **环境** | 设计时/开发时/部署时 | 系统集成测试阶段 |
| **制品** | 被测试的系统或组件 | 支付模块 |
| **响应** | 可观察、可控制 | 提供测试结果和日志 |
| **响应度量** | 测试覆盖率、发现缺陷的效率 | 语句覆盖率≥85% |

#### 常见战术

**（1）提高可观察性**
- **记录/回放**：记录系统运行状态以供分析
- **内置监视器**：健康检查接口、性能指标暴露
- **日志**：详细的分级日志输出

**（2）提高可控制性**
- **特化接口**：提供专用测试接口
- **抽象数据源**：mock外部依赖
- **沙箱模式**：隔离测试环境

---

### 6. 易用性（Usability）

#### 定义
易用性是指用户使用软件系统完成特定任务的难易程度，包括学习成本、操作效率、错误容忍等。

#### 场景六要素

| 要素 | 说明 | 示例 |
|------|------|------|
| **刺激源** | 终端用户 | 首次使用系统的用户 |
| **刺激** | 学习使用、高效操作、最小化影响、适应需求 | 首次使用系统 |
| **环境** | 运行时 | 系统正常运行 |
| **制品** | 系统界面 | 用户操作界面 |
| **响应** | 提供帮助、自适应、反馈 | 提供操作引导 |
| **响应度量** | 任务完成时间、错误率 | 5分钟内完成核心操作 |

#### 常见战术

- **运行时战术**：
  - 任务模型预测用户行为
  - 系统模型反馈当前状态
  - 用户模型记忆用户偏好
- **设计时战术**：
  - 将用户界面与业务逻辑分离（MVC/MVP/MVVM）
  - 支持撤销/重做操作
  - 提供一致的界面风格

## 三、质量属性之间的权衡

### 常见权衡关系

| 权衡对 | 说明 |
|--------|------|
| **性能 ↔ 安全性** | 加密和认证增加处理开销，影响性能 |
| **性能 ↔ 可修改性** | 高性能优化（如缓存、内联）增加修改难度 |
| **可用性 ↔ 安全性** | 安全策略（如频繁认证）可能影响可用性体验 |
| **可用性 ↔ 性能** | 冗余部署提高可用性，但增加资源消耗 |
| **可修改性 ↔ 性能** | 抽象层增加灵活性，但引入间接调用的开销 |

### 架构评估中的关键概念

| 概念 | 定义 |
|------|------|
| **敏感点** | 为了实现某种质量属性而做出的架构决策 |
| **权衡点** | 影响多个质量属性的架构决策（同时是多个属性的敏感点） |
| **风险点** | 可能引起不良后果的架构决策 |
| **非风险点** | 经过分析后确认为安全的架构决策 |

## 四、历年真题中的质量属性考法

### 题型一：识别质量属性（高频）

**题目形式：** 给出若干场景描述，要求判断每个场景对应的质量属性。

**答题方法：**
1. 找关键词：响应时间/吞吐量 → 性能；故障恢复 → 可用性；攻击/加密 → 安全性
2. 注意区分相近属性（如可靠性属于可用性范畴）

**常见关键词对照：**

| 关键词 | 对应属性 |
|--------|----------|
| 响应时间、吞吐量、并发、延迟 | 性能 |
| 故障恢复、宕机、冗余、备份、心跳 | 可用性 |
| 加密、认证、授权、攻击、注入 | 安全性 |
| 修改、扩展、新增功能、变更 | 可修改性 |
| 测试覆盖、可观察、可控制 | 可测试性 |
| 学习成本、操作效率、用户体验 | 易用性 |

### 题型二：描述质量属性场景（高频）

**题目形式：** 给出一个系统场景，要求用质量属性场景的六要素进行描述。

**答题模板：**
```
该场景对应的质量属性是【XXX】。
场景六要素分析：
- 刺激源：【XXX】
- 刺激：【XXX】
- 环境：【XXX】
- 制品：【XXX】
- 响应：【XXX】
- 响应度量：【XXX】
```

### 题型三：选择合适的架构战术

**题目形式：** 针对某个质量属性需求，要求选择或推荐合适的战术。

**答题方法：**
1. 先明确质量属性类型
2. 列出该属性下的战术分类
3. 根据场景特征选择最合适的战术
4. 说明选择理由

### 题型四：分析架构风险/敏感点/权衡点

**题目形式：** 给出架构描述，要求分析其中的敏感点和权衡点。

**答题方法：**
1. 敏感点：找到"为了达到XX质量属性而做出的设计决策"
2. 权衡点：找到"同时影响两个或以上质量属性的设计决策"
3. 使用"该决策影响了XX属性的同时，也影响了YY属性"的句式

## 五、实战练习

### 练习题1

> 某在线教育平台要求：在1000个并发用户同时观看直播课程时，视频延迟不超过3秒。

**参考答案：**
- 质量属性：性能
- 刺激源：1000个并发用户
- 刺激：同时发起视频直播观看请求
- 环境：正常运行模式（峰值时段）
- 制品：视频直播服务
- 响应：播放视频流
- 响应度量：视频延迟≤3秒

### 练习题2

> 某银行系统要求：当主数据中心发生故障时，系统能在30秒内切换到灾备中心，继续提供服务。

**参考答案：**
- 质量属性：可用性
- 刺激源：主数据中心（内部故障源）
- 刺激：数据中心整体故障（如断电、网络中断）
- 环境：正常运行模式
- 制品：银行核心业务系统
- 响应：检测故障 → 触发灾备切换 → 恢复服务
- 响应度量：切换时间≤30秒
- 推荐战术：主动冗余（热备）+ 心跳检测

### 练习题3

> 某电商系统要求：能在不停止服务的情况下，在2个工作日内完成新增支付方式（如数字人民币）的功能上线。

**参考答案：**
- 质量属性：可修改性
- 刺激源：产品经理（需求变更方）
- 刺激：新增数字人民币支付方式
- 环境：运行时（不停机）
- 制品：支付模块
- 响应：开发、测试、灰度发布
- 响应度量：2个工作日内完成上线
- 推荐战术：抽象化（支付接口抽象）+ 插件机制（支付方式插件化）+ 热部署
