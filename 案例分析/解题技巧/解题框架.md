# 系统架构设计师案例分析——通用解题框架

### 考试基本信息

| 项目 | 内容 |

| 考试时间 | 90分钟 |

## 二、质量属性分析答题模板

### 解题步骤

```
步骤1：识别题目涉及的质量属性（性能/可用性/安全性等）
步骤2：写出该质量属性的定义
步骤3：根据题目场景，分析质量属性场景的六要素
步骤4：列出对应的架构战术
步骤5：结合题目给出具体解决方案
```

可采用的架构战术包括：

## 三、架构风格对比答题模板

### 解题步骤

```
步骤1：识别题目涉及的架构风格
步骤2：分别阐述各架构风格的定义和特点
步骤3：从多个维度进行对比（耦合度、扩展性、性能等）
步骤4：结合场景分析适用性
步骤5：给出推荐方案并说明理由
```

### 常见架构风格对比框架

#### （1）单体架构 vs 微服务架构

| 对比维度 | 单体架构 | 微服务架构 |
|----------|----------|------------|
| **部署方式** | 整体部署 | 独立部署 |
| **技术栈** | 统一技术栈 | 可异构 |
| **数据管理** | 共享数据库 | 独立数据库 |
| **耦合度** | 高 | 低 |
| **扩展性** | 垂直扩展 | 水平扩展 |
| **开发效率** | 初期高，后期低 | 初期低，后期高 |
| **运维复杂度** | 低 | 高 |
| **适用场景** | 小型项目、快速原型 | 大型项目、团队规模大 |

#### （2）SOA vs 微服务

| 对比维度 | SOA | 微服务 |
|----------|-----|--------|
| **服务粒度** | 粗粒度 | 细粒度 |
| **通信方式** | ESB（重量级） | HTTP/REST、gRPC（轻量级） |
| **数据管理** | 可共享数据库 | 各自独立数据库 |
| **治理方式** | 集中式治理 | 去中心化治理 |
| **部署** | 通常集中部署 | 独立容器化部署 |
| **适用场景** | 企业级集成 | 互联网应用 |

#### （3）B/S vs C/S

| 对比维度 | B/S | C/S |
|----------|-----|-----|
| **客户端** | 浏览器（零安装） | 专用客户端（需安装） |
| **升级维护** | 服务器端统一更新 | 客户端需逐一更新 |
| **用户体验** | 受浏览器限制 | 体验丰富 |
| **安全性** | 依赖Web安全 | 可增加安全层 |
| **网络依赖** | 依赖网络 | 可离线运行 |

## 四、数据库设计答题模板

### 解题步骤

```
步骤1：理解题目业务场景和数据需求
步骤2：识别实体和关系
步骤3：绘制/补全ER图
步骤4：进行关系模式设计和范式分析
步骤5：编写SQL语句（如需要）
```

### 常见考点

#### （1）ER图设计

```
答题要点：
- 识别实体（矩形框）
- 确定属性（椭圆框）
- 确定关系（菱形框）
- 标注联系类型：1:1、1:N、M:N
- 注意弱实体和部分参与/全参与
```

#### （2）范式分析

| 范式 | 要求 | 消除的问题 |
|------|------|------------|
| **1NF** | 属性不可再分 | 消除非原子值 |
| **2NF** | 1NF + 消除非主属性对码的部分函数依赖 | 消除部分依赖 |
| **3NF** | 2NF + 消除非主属性对码的传递函数依赖 | 消除传递依赖 |
| **BCNF** | 每个决定因素都是候选码 | 消除主属性的传递依赖 |

**答题模板：**
```
该关系模式R的函数依赖集为：
F = {【列出函数依赖】}

候选码为：【候选码】

该关系模式属于【X】NF，因为【原因分析】。

存在的问题是：【具体问题，如数据冗余、更新异常等】

分解方案为：
R1(属性1, 属性2, ...)，候选码为...
R2(属性1, 属性2, ...)，候选码为...
分解后各关系模式均属于【更高范式】。
```

#### （3）SQL编写

```
常见考点：
- SELECT 多表连接查询（JOIN）
- 子查询（IN/EXISTS）
- 聚合函数（COUNT/SUM/AVG/MAX/MIN + GROUP BY + HAVING）
- 视图创建（CREATE VIEW）
- 索引创建和优化
- 存储过程基础
```

#### （4）分布式数据库

```
答题要点：
- 数据分片策略：水平分片、垂直分片、混合分片
- 分片原则：完备性、重构性、不相交性
- CAP定理：一致性、可用性、分区容错性（三选二）
- BASE理论：基本可用、软状态、最终一致性
- 分布式事务：2PC、3PC、TCC、Saga、消息最终一致性
```

## 五、设计模式答题模板

### 解题步骤

```
步骤1：阅读题目类图或代码，识别设计模式
步骤2：回忆该模式的结构和角色
步骤3：补全类图中缺失的类、方法或关系
步骤4：填写代码空缺部分
```

### 常考设计模式

| 模式 | 意图 | 关键角色 | 识别特征 |
|------|------|----------|----------|
| **工厂方法** | 定义创建对象的接口，子类决定实例化哪个类 | Creator, ConcreteCreator, Product | `createXxx()` 方法 |
| **抽象工厂** | 提供创建一系列相关对象的接口 | AbstractFactory, ConcreteFactory | 多个 `createXxx()` |
| **单例** | 保证一个类只有一个实例 | Singleton | `getInstance()` |
| **策略** | 定义算法族，封装后可互相替换 | Context, Strategy, ConcreteStrategy | 行为参数化 |
| **观察者** | 一对多依赖，状态变更时通知所有依赖者 | Subject, Observer | `notify()`, `update()` |
| **适配器** | 将一个接口转换为客户期望的接口 | Target, Adapter, Adaptee | 接口不兼容 |
| **装饰器** | 动态地给对象添加职责 | Component, Decorator | 层层包装 |
| **模板方法** | 定义算法骨架，子类重写特定步骤 | AbstractClass, ConcreteClass | 抽象方法 + final方法 |

### 答题模板

```
该设计使用了【XXX】模式。

【XXX】模式的意图是：【一句话说明意图】。

在本题中：
- 【角色1】对应：【题目中的具体类/接口】
- 【角色2】对应：【题目中的具体类/接口】
- 【角色3】对应：【题目中的具体类/接口】

空白处应填写：
（1）【答案】
（2）【答案】
...
```

## 七、答题通用技巧

### 时间分配

```
总计90分钟，共5题（选做4题）

建议分配：
- 浏览全卷：5分钟
- 每道题：20分钟
- 检查校对：5分钟

单题时间分配：
- 审题理解：3分钟
- 组织答案：2分钟
- 书写答题：12分钟
- 检查补充：3分钟
```

### 答题格式规范

1. **分点作答**：使用（1）（2）（3）或①②③编号
2. **先给结论再给理由**：开头先写答案，再展开解释
3. **使用专业术语**：体现专业水平
4. **结合题目场景**：不要脱离题目泛泛而谈
5. **注意字数**：关键点一定要写全，不要惜字如金

### 常见"套路"汇总

| 题目问法 | 答题思路 |
|----------|----------|
| "分析...的优缺点" | 分别列出优点3-4条 + 缺点2-3条 |
| "对比A和B" | 列表格，从5-6个维度对比 |
| "补全...图" | 仔细分析上下文，注意数据流/调用关系 |
| "写出质量属性场景" | 严格按六要素格式回答 |
| "给出改进方案" | 先指出问题 → 分析原因 → 给出方案 |
| "填写...代码" | 注意语法、返回类型、参数列表 |
